This document lists the configuration options for the <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file where you define your CI/CD configuration on <NT>Bitrise</NT>.

- Check out our overview for the basic principles of the configuration YAML file: [Configuration YAML overview](/en/bitrise-ci/configure-builds/configuration-yaml/configuration-yaml-overview).
- <NT>Read more</NT> about where to store the configuration YAML file: [Managing a project's configuration YAML file](/en/bitrise-ci/configure-builds/configuration-yaml/storing-an-apps-configuration-yaml).
- Create advanced modular configurations: [Modular YAML configuration](/en/bitrise-ci/configure-builds/configuration-yaml/modular-yaml-configuration).


## Project level properties

### `format_version`

**<NT>Required</NT>**: A configuration YAML file is invalid without it.

The version of the <NT>Bitrise</NT> configuration format.

The <NT>Bitrise CLI</NT> checks it to ensure compatibility between the configuration file and the CLI version. This is relevant for locally run builds: on the <NT>Bitrise website</NT>, the CLI is always up to date, so the `format_version` property doesn't have an effect on builds triggered.

**<NT>Example</NT> of `format_version`**

```yaml
format_version: '25'
```

### `default_step_lib_source`

The default <NT><GlossTerm baseform="step">Step</GlossTerm></NT> library for the <NT>Steps</NT> used in your configuration. <NT>Bitrise</NT> uses this source if no specific source is provided for a given <NT>Step</NT>.

The <NT>default value</NT> is https://github.com/bitrise-io/bitrise-steplib.git. You can use your own fork of this repository as a <NT>Step</NT> source, or you can refer to <NT>Steps</NT> by including their exact source repository: [Step reference/ID format](/en/bitrise-ci/references/steps-reference/step-reference-id-format).

**<NT>Example</NT> of `default_step_lib_source`**

```yaml
default_step_lib_source: https://github.com/bitrise-io/bitrise-steplib.git
```

### `project_type`

The type of your <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>, such as `ios`, `android`, `flutter`, and so on.

The project type can be modified at any point but after the initial setup, it has no relevance.

**<NT>Example</NT> of `project_type`**

```yaml
project_type: flutter
```

### `title`

The title of the <NT>Bitrise</NT> configuration.

**<NT>Example</NT> of `title`**

```yaml
title: My Bitrise project
```

### `summary`

A short summary of the <NT>Bitrise</NT> configuration.

**<NT>Example</NT> of `summary`**

```yaml
summary: This project runs tests for my app.
```

### `description`

A detailed description of the <NT>Bitrise</NT> configuration.

**<NT>Example</NT> of `description`**

```yaml
description: This project runs unit and UI tests when a pull request is opened to the dev branch.
```

### `services`

<NT>Docker</NT> container definitions used by <NT>Steps</NT> as service containers on <NT>Linux</NT> hosts.

:::note[<NT>Linux</NT> only]

This is only supported on <NT>Linux</NT> <NT><GlossTerm baseform="stack">stack</GlossTerm></NT>s: the property doesn't work on macOS stacks!

:::

One or more service can be configured for a group of <NT>Steps</NT>, allowing running <NT>Docker</NT> containers as services for advanced integration testing. These are tied to the lifecycle of a `with` group and will run alongside it in the background. When all <NT>Steps</NT> within the groups finish, they are cleaned up.

You need to define:

- The ID of the service which is used to refer to the service in a with group.
- The image name of the service.
- The image version of the service.

Read more: [Service containers](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/service-containers).

**<NT>Example</NT> of `services`**

```yaml
services:
 postgres:
   image: postgres:16
   envs: 
   - POSTGRES_USER: postgres
   - POSTGRES_DB: bitrise 
   ports:
   - 5432:5432 
   options: >-
     --health-cmd pg_isready
     --health-interval 10s
     --health-timeout 5s
     --health-retries 5
 redis: 
   image: redis:7
     ports: - 6379:6379
     options: >-
     --health-cmd "redis-cli ping"
     --health-interval 10s
     --health-timeout 5s
     --health-retries 5
```

### `containers`

<NT>Docker</NT> container definitions used by <NT>Steps</NT> as execution containers.

<NT>Docker</NT> containers can be defined for any <NT>Bitrise</NT> <NT>project</NT> in the configuration YAML file. You define the container in the top level of the configuration file and then refer to it in a `with` group in the <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT> configuration. You need to define:

- The ID of the containers. It will be used to reference this container.
- The name of the <NT>Docker</NT> image you want to use.
- The version of the image.

`containers reference`: [Docker container properties](/en/bitrise-ci/references/configuration-yaml-reference#docker-container-properties).

Read more: [Execution containers](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/execution-containers).

**<NT>Example</NT> of `containers`**

```yaml
containers:
  node-21:
    image: node:21.6
```

### `app`

This property contains project-level configuration information, such as [Environment Variables](/en/bitrise-ci/configure-builds/environment-variables).

Meta information such as the build stack doesn't belong here but to the `meta` property.

**<NT>Example</NT> of `app`**

```yaml
app:
  envs:
  - TEST: '' 
    opts:
      is_expand: false
```

### `meta`

Stores project metadata key-value pairs related to the <NT>Bitrise</NT> configuration. Most importantly, it defines the default stack and the <NT>build machine</NT> type for the project.

Read more: [Setting the stack in the bitrise.yml file](/en/bitrise-ci/configure-builds/configuring-build-settings/setting-the-stack-for-your-builds#setting-the-stack-in-the-configuration-yaml).

**<NT>Example</NT> of `meta`**

```yaml
meta:
  bitrise.io:
    stack: linux-docker-android-22.04
    machine_type_id: g2.linux.2medium
```

### `trigger_map`

Defines the build <NT><GlossTerm baseform="trigger">trigger</GlossTerm></NT>s on a project level. This is the legacy method of configuring triggers: we recommend using target-based triggers defined within a <NT><GlossTerm baseform="pipeline">Pipeline</GlossTerm></NT> or a <NT>Workflow</NT> instead.

Read more: [Legacy project-based triggers](/en/bitrise-ci/run-and-analyze-builds/build-triggers/legacy-project-based-triggers).

**<NT>Example</NT> of `trigger_map`**

```yaml
trigger_map:
- pull_request_source_branch: "*"
  type: pull_request
  workflow: primary
```

### `pipelines`

Define <NT>Pipelines</NT> in your project's configuration. <NT>Pipelines</NT> can be used to organize the entire CI/CD process and to set up advanced configurations with multiple different tasks running parallel and/or sequentially.

`pipelines` reference: [Pipeline level properties](/en/bitrise-ci/references/configuration-yaml-reference#pipeline-level-properties).

Read more: [About Pipelines](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/about-pipelines).

**<NT>Example</NT> of `pipelines`**

```yaml
pipelines:
  test:
  title: Test Pipeline
  summary: This Pipeline runs unit tests.
  workflows:
    primary: {}
```

### `workflows`

The <NT>Workflows</NT> included in your configuration. A <NT>Workflow</NT> is a collection of <NT>Steps</NT>: reusable, configurable units of work. <NT>Steps</NT> are executed sequentially within a <NT>Workflow</NT>.

`workflows` reference: [Workflow level properties](/en/bitrise-ci/references/configuration-yaml-reference#workflow-level-properties).

**<NT>Example</NT> of `workflows`**

```yaml
workflows: 
  primary:
    steps: 
    - script: {}
```

### `step_bundles`

<NT>Step bundles</NT> allow you to group multiple <NT>Steps</NT> into a single unit. With <NT>Step bundles</NT>, you can reuse <NT>Steps</NT> and sequences of <NT>Steps</NT>.

You can insert <NT>Step bundles</NT> into any <NT>Workflow</NT>. Unlike utility <NT>Workflows</NT> and <NT>Workflow</NT> chaining, <NT>Step Bundles</NT> can be placed anywhere in a <NT>Workflow</NT>.

`step_bundles` reference: [Step bundle properties](/en/bitrise-ci/references/configuration-yaml-reference#step-bundle-properties).

Read more: [Step bundles](/en/bitrise-ci/workflows-and-pipelines/steps/step-bundles).

**<NT>Example</NT> of `step_bundles`**

```yaml
step_bundles:
  primary:
    steps:
    - git-clone: {}
    - restore-cache: {}
```

### `include`

The `include` property allows you to use other configuration YAML files in your <NT>Bitrise</NT> configuration, to break down large, complex YAML files into smaller, modular components.

A modular YAML configuration includes:

- A `bitrise.yml` file in the root of your repository.
- Other YAML files in the same or a different repository. To include a file from a different repository, the repository must belong to the same Git account or organization as the primary repository.
- One or more `include` keywords in the <NT><GlossTerm baseform="bitrise.yml">bitrise.yml</GlossTerm></NT> file. These point to other YAML files and bring their configuration into the main project configuration.

Read more: [Modular YAML configuration](/en/bitrise-ci/configure-builds/configuration-yaml/modular-yaml-configuration).

**<NT>Example</NT> of `include`**

```yaml
include:
  - path: path/to/config_module.yml
```

### `tools`

Tool version to set up. The version syntax supports:

- Exact versions: for example, `'1.2.3'`.
- Partial matches to the latest release: for example, `'22:latest'`.
- Partial matches to installed versions: for example, `'1.2:installed'`.
- Special aliases `'latest'` and `'installed'` to select the latest or the highest installed version.

Read more: [Configuring tool versions](/en/bitrise-ci/configure-builds/configuring-build-settings/configuring-tool-versions).

**<NT>Example</NT> of `tools`**

```yaml
tools:
  nodejs: 22:installed
  ruby: 3.3:latest
```


## <NT>Pipeline</NT> level properties

### `pipelines:<pipeline-ID>:title`

The title of the <NT>Pipeline</NT>.

This is not the same as the ID of the <NT>Pipeline</NT>: you set the ID when creating the <NT>Pipeline</NT>. Unlike the ID, the title does not have to be unique. It appears on the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> UI in the **<NT>Pipelines</NT>** section: if you open the <NT>Pipeline</NT> selector dropdown menu.

**<NT>Example</NT> of `pipelines:<pipeline-ID>:title`**

```yaml
pipelines:
  test:
    title: Test pipeline
```

### `pipelines:<pipeline-ID>:summary`

A short summary of the <NT>Pipeline</NT>. Optional.

**<NT>Example</NT> of `pipelines:<pipeline-ID>:summary`**

```yaml
pipelines:
  test:
    summary: This Pipeline runs unit tests.
```

### `pipelines:<pipeline-ID>:description`

A detailed description of the <NT>Pipeline</NT>. Optional.

**<NT>Example</NT> of `pipelines:<pipeline-ID>:description`**

```yaml
pipelines:
  test:
    description: 'This Pipeline runs unit tests in parallel, using test sharding.'
```

### `pipelines:<pipeline-ID>:triggers`

Target-based triggers defined for the <NT>Pipeline</NT>: if a code event matches the condition defined in a trigger, <NT>Bitrise</NT> will trigger a build with the <NT>Pipeline</NT>.

For the detailed syntax of the `triggers` property and its available options, check out:

Read more: [YAML syntax for build triggers](/en/bitrise-ci/run-and-analyze-builds/build-triggers/yaml-syntax-for-build-triggers).

**<NT>Example</NT> of `pipelines:<pipeline-ID>:triggers`**

In this example, any commit pushed to the `main` branch triggers a build of the <NT>Pipeline</NT>.

```yaml
pipelines:
  test:
    triggers:
      push:
      - branch: main
```

### `pipelines:<pipeline-ID>:status_report_name`

The name that will appear on the status report sent to connected services (like <NT>GitHub</NT>, <NT>GitLab</NT>, <NT>Bitbucket</NT>, and so on) after the build is finished.

It can have both static and dynamic values, as well as combine the two types.

**<NT>Supported</NT> characters and variables**:

- `A-Za-z,.():/-_0-9 []|<>`
- `<project_slug>`: The unique identifier of your project.
- `<project_title>`: Optional title of your project.
- `<target_id>`: The ID of the triggered <NT>Workflow</NT> or <NT>Pipeline</NT>.
- `<event_type>`: The code event that triggered the build: `PR/push/tag`.

**<NT>Example</NT> of `pipelines:<pipeline-ID>:status_report_name`**

```yaml
pipelines:
  test:
    status_report_name: ci/bitrise/510526/push
```

### `pipelines:<pipeline-ID>:workflows`

The <NT>Workflows</NT> that are part of the <NT>Pipeline</NT> configuration.

You can create dependencies between <NT>Workflows</NT> with the `depends_on` property.

**<NT>Example</NT> of `pipelines:<pipeline-ID>:workflows`**

```yaml
pipelines:
  test:
    workflows:
      primary:
```

### `pipelines:<pipeline-ID>:priority`

The priority setting determines the position of the <NT>Pipeline</NT> build in the <NT>build queue</NT>: the higher the priority, the sooner the <NT>Pipeline</NT> build will run.

**<NT>Supported values</NT>:**

- An integer between -100 and 100.

**<NT>Read more</NT>:**

- [Pipeline priority](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline#pipeline-priority)
- [Build priority](/en/bitrise-ci/configure-builds/configuring-build-settings/build-priority)

**<NT>Example</NT> of `pipelines:<pipeline-ID>:priority`**

```yaml
pipelines:
  test:
    priority: 10
```


## Properties of <NT>Workflows</NT> in <NT>Pipelines</NT>

### `workflows:depends_on`

This property defines the <NT>Workflows</NT> that this <NT>Workflow</NT> depends on. If any of the <NT>Workflows</NT> in the list fails, this <NT>Workflow</NT> won't run.

You can use both types of YAML array syntax to list dependant <NT>Workflows</NT>: block sequence and flow sequence. Any <NT>Workflow</NT> appearing in the dependency list must be part of the same <NT>Pipeline</NT>.

:::caution[No circle or loop configuration]

The dependency graph resulting from your configuration can't contain a circle or loop. You can't start builds with an invalid <NT>Pipeline</NT> configuration.

:::

**<NT>Supported values</NT>**: <NT>Workflow</NT> names with YAML array syntax.

Read more: [Configuring a Bitrise Pipeline](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline).

**<NT>Example</NT> of `workflows:depends_on`**

Block sequence array syntax:

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        depends_on:
        - A
      C:
        depends_on:
        - A
      D:
        depends_on:
        - B
        - C
```

Flow sequence array syntax:

```yaml
D:
  depends_on: [B, C]
```

### `workflows:abort_on_fail`

When this property is `true`, the <NT>Pipeline</NT> and any other running <NT>Workflows</NT> are aborted if this <NT>Workflow</NT> fails.

**<NT>Supported values</NT>**: boolean

Read more: [Configuring a Bitrise Pipeline](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline).

**<NT>Example</NT> of `workflows:abort_on_fail`**

In this example, both A and B are running in parallel. If B fails, the entire <NT>Pipeline</NT>, including <NT>Workflow</NT> A, is immediately aborted.

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        abort_on_fail: true
```

### `workflows:should_always_run`

This property defines whether a <NT>Workflow</NT> should run if a previous <NT>Workflow</NT> failed.

Be aware of transitive dependency: if a <NT>Workflow</NT> that is set to always run has a parent <NT>Workflow</NT> that fails, it will still run. However, <NT>Workflows</NT> that depend on it will not. For example, let's say we have <NT>Workflow</NT> C that depends on <NT>Workflow</NT> B. <NT>Workflow</NT> B depends on <NT>Workflow</NT> A. Of the three <NT>Workflows</NT>, only B is set to always run:

- If A fails, B will run but regardless of its result, C won't run because by depending on B, it also depends on A.
- C only runs if both A and B are successful.

**<NT>Supported values</NT>:**

- `none`: If a parent <NT>Workflow</NT> fails, the <NT>Workflow</NT> will not run. This is the <NT>default value</NT>.
- `workflow`: If a parent <NT>Workflow</NT> fails, the <NT>Workflow</NT> will run anyway.

Read more: [Configuring a Bitrise Pipeline](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline)

**<NT>Example</NT> of `workflows:should_always_run`**

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        depends_on: [ A ]
        should_always_run: workflow
      C:
        depends_on: [ B ]
      D:
        depends_on: [ C ]
        should_always_run: workflow
```

### `workflows:run_if`

A Go template expression that defines the conditions for running the <NT>Workflow</NT>.

The property needs an `expression` field that contains a Go template, with three helper functions:

- `getenv`: Accesses an Environment Variable's value.
- `enveq`: Compares an Environment Variable to a given value.
- `envcontain`: Checks whether an Environment Variable contains a given string.

The <NT>Bitrise CLI</NT> evaluates the expression during runtime. If a <NT>Workflow</NT> is skipped because of a `run_if` expression, it will be counted as a successful <NT>Workflow</NT> so its dependent <NT>Workflows</NT> will run.

Read more: [Examples of run_if expressions](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally#examples-of-run_if-expressions).

**<NT>Example</NT> of `workflows:run_if`**

In this example, B will only run if the EXAMPLE_KEY Environment Variable has the value example value.

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        run_if:
          expression: {{ enveq "EXAMPLE_KEY" "example value" }}
        depends_on: [A]
```

### `workflows:parallel`

The `parallel` property allows you to run copies of the same <NT>Workflow</NT> in parallel, in a single instruction. This is particularly useful for test sharding.

The property determines the number of copies running in parallel. Each copy receives two new <NT>environment variables</NT>:

- $BITRISE_IO_PARALLEL_INDEX: a zero based index for each copy of the <NT>Workflow</NT>.
- $BITRISE_IO_PARALLEL_TOTAL: the total number of copies.

**<NT>Supported values</NT>**: An integer.

Read more: [Parallelism in Pipelines](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline#running-variations-of-the-same-workflow).


## <NT>Workflow</NT> level properties

### `workflows:<workflow-id>:title`

The title of the <NT>Workflow</NT>.

It appears on the UI of the <NT>Workflow Editor</NT>.

**<NT>Example</NT> of `workflows:<workflow-id>:title`**

```yaml
workflows:
  primary:
    title: Primary Workflow
```

### `workflows:<workflow-id>:summary`

A short summary of the <NT>Workflow</NT>.

**<NT>Example</NT> of**

```yaml
workflows:
  primary:
    summary: This Workflow runs unit tests.
```

### `workflows:<workflow-id>:description`

A detailed description of the <NT>Workflow</NT>.

**<NT>Example</NT> of `workflows:<workflow-id>:description`**

```yaml
workflows:
  primary:
    description: This Workflow runs unit tests with test sharding, and exports the test results to the test reports page.
```

### `workflows:<workflow-id>:triggers`

Target-based triggers defined for the <NT>Workflow</NT>: if a code event matches the condition defined in a trigger, <NT>Bitrise</NT> will trigger a build with the <NT>Workflow</NT>.

Each trigger defines a code event and at least one condition. If you define multiple conditions in a single trigger, all of them must match to trigger a build.

For the detailed syntax of the `triggers` property and its available options, check out:

Read more: [YAML syntax for build triggers](/en/bitrise-ci/run-and-analyze-builds/build-triggers/yaml-syntax-for-build-triggers).

**<NT>Example</NT> of `workflows:<workflow-id>:triggers`**

In this example, a build is triggered if:

- A commit is pushed to the `main` branch.
- A pull request is opened from any branch.

```yaml
workflows:
  primary:
    triggers:
      push:
      - branch: main
      pull_request:
      - source_branch: "*"
```

### `workflows:<workflow-id>:status_report_name`

The name that will appear on the status report sent to connected services (like <NT>GitHub</NT>, <NT>GitLab</NT>, <NT>Bitbucket</NT>, and so on) after the build is finished.

It can have both static and dynamic values, as well as combine the two types.

**<NT>Supported</NT> characters and variables**:

- `A-Za-z,.():/-_0-9 []|<>`
- `<project_slug>`: The unique identifier of your project.
- `<project_title>`: Optional title of your project.
- `<target_id>`: The ID of the triggered <NT>Workflow</NT> or <NT>Pipeline</NT>.
- `<event_type>`: The code event that triggered the build: `PR/push/tag`.

**<NT>Example</NT> of `workflows:<workflow-id>:status_report_name`**

```yaml
workflows:
  test:
    status_report_name: ci/bitrise/510526/push
```

### `workflows:<workflow-id>:before_run`

:::tip[<NT>Step bundles</NT>]

This is a legacy feature. We recommend using <NT>Step bundles</NT> instead: [Step bundles](/en/bitrise-ci/workflows-and-pipelines/steps/step-bundles).

:::

The <NT>Workflows</NT> that will run before this <NT>Workflow</NT> starts.

Use this property and `after_run` to create chains of <NT>Workflows</NT>.

Read more: [Chaining Workflows together](/en/bitrise-ci/workflows-and-pipelines/workflows/managing-workflows#chaining-workflows-together).

**<NT>Example</NT> of `workflows:<workflow-id>:before_run`**

In this example, we're creating a configuration in which the `test` <NT>Workflow</NT> runs before the `deploy` <NT>Workflow</NT>:

```yaml
workflows:
  test:
    steps:
    # test Steps

  deploy:
    before_run:
    - test
    steps:
    # deploy Steps
```

### `workflows:<workflow-id>:after_run`

:::tip[<NT>Step bundles</NT>]

This is a legacy feature. We recommend using <NT>Step bundles</NT> instead: [Step bundles](/en/bitrise-ci/workflows-and-pipelines/steps/step-bundles).

:::

The <NT>Workflows</NT> that will run after this <NT>Workflow</NT> is successfully finished.

Use this property and `before_run` to create chains of <NT>Workflows</NT>.

Read more: [Chaining Workflows together](/en/bitrise-ci/workflows-and-pipelines/workflows/managing-workflows#chaining-workflows-together).

**<NT>Example</NT> of `workflows:<workflow-id>:before_run`**

In this example, we're creating a configuration in which the `deploy` <NT>Workflow</NT> runs after the `ci` <NT>Workflow</NT>:

```yaml
workflows:
  deploy:
    steps:
    # deploy Steps

  ci:
    after_run:
    - deploy
    steps:
    # CI Steps
```

### `workflows:<workflow-id>:envs`

<NT>Environment Variables</NT> defined for the <NT>Workflow</NT>.

Each Environment Variable is a key-value pair. <NT>Environment Variables</NT> can use other <NT>Environment Variables</NT> as values: [Using Env Vars in the value of an Env Var](/en/bitrise-ci/configure-builds/environment-variables#using-env-vars-in-the-value-of-an-env-var).

These variables are only available for the selected <NT>Workflow</NT>. Other <NT>Workflows</NT> defined in the same configuration YAML file can't access them. Project-level <NT>Environment variables</NT> have a higher priority: [Availability order of Environment Variables](/en/bitrise-ci/configure-builds/environment-variables#availability-order-of-environment-variables).

Read more: [Environment Variables](/en/bitrise-ci/configure-builds/environment-variables).

**<NT>Example</NT> of `workflows:<workflow-id>:envs`**

```yaml
workflows:
  primary:
    envs:
    - TEST: test
    - ENV_LABEL: dev
      opts:
        is_expand: false
```

### `workflows:<workflow-id>:steps`

The list of <NT>Steps</NT> in the <NT>Workflow</NT>.

The basic syntax of a <NT>Step</NT> reference is: `<step_lib_source>::<step-id>@<version>:`. The <NT>Step ID</NT> is always required; the source and the version are optional.

- If you don't set a source, `default_step_lib_source` will be used: [Project level properties](/en/bitrise-ci/references/configuration-yaml-reference#project-level-properties).
- If you don't set a version, the <NT>latest version</NT> will be used.

Read more:

- [Step reference/ID format](/en/bitrise-ci/references/steps-reference/step-reference-id-format)
- [Step versions](/en/bitrise-ci/workflows-and-pipelines/steps/step-versions)

**<NT>Example</NT>s of `workflows:<workflow-id>:steps`**

In this example, we don't specify an exact source and we use the <NT>latest version</NT> of major version 1:

```yaml
workflows:
    steps:
    - script: {}
```

In this example we're providing a source for the <NT>Step</NT>:

```yaml
workflows:
    steps:
    - https://github.com/bitrise-io/bitrise-steplib.git::script@1: {}
```

### `workflows:<workflow-id>:priority`

The priority setting determines the position of the <NT>Workflow</NT> build in the <NT>build queue</NT>: the higher the priority, the sooner the <NT>Workflow</NT> build will run.

**<NT>Supported values</NT>**: An integer between -100 and 100.

Read more: [Build priority](/en/bitrise-ci/configure-builds/configuring-build-settings/build-priority).

**<NT>Example</NT> of `workflows:<workflow-id>:priority`**

```yaml
workflows:
  primary:
    priority: 10
```

### `workflows:<workflow-id>:tools`

Declarative configuration for tool versions used in the build.

You can define tool versions for specific <NT>Workflows</NT>. Instead of defining the tools property at the top level of the configuration, you nest it under one or more <NT>Workflows</NT>.

The version syntax supports:

- Exact versions: for example, `'1.2.3'`.
- Partial matches to the latest release: for example, `'22:latest'`.
- Partial matches to installed versions: for example, `'1.2:installed'`.
- Special aliases `'latest'` and `'installed'` to select the latest or the highest installed version.

Read more: [Configuring tool versions](/en/bitrise-ci/configure-builds/configuring-build-settings/configuring-tool-versions).

**<NT>Example</NT> of `workflows:<workflow-id>:tools`**

```yaml
workflows:
  primary:
    tools:
      nodejs: 22:installed
```

### `workflows:<workflow-id>:meta`

Stores project metadata key-value pairs. Most importantly, it defines the stack and the machine type for the <NT>Workflow</NT>.

Read more: [Setting the stack in the bitrise.yml file](/en/bitrise-ci/configure-builds/configuring-build-settings/setting-the-stack-for-your-builds#setting-the-stack-in-the-configuration-yaml).

**<NT>Example</NT> of `workflows:<workflow-id>:meta`**

```yaml
workflows:
  deploy:  
    meta:      
      bitrise.io:       
        stack: linux-docker-android-22.04
        machine_type_id: g2.linux.2medium
```

## <NT>Step</NT> level properties

### `steps:<step-id>:title`

The title of the <NT>Step</NT>.

This appears on the <NT>Workflow Editor</NT>, replacing the <NT>Step ID</NT>.

**<NT>Example</NT> of `steps:<step-id>:title`**

```yaml
workflows:
    steps:
    - script@1:
        title: Test sharding script
```

### `steps:<step-id>:summary`

A short summary of the <NT>Step</NT>.

**<NT>Example</NT> of `steps:<step-id>:summary`**

```yaml
workflows:
    steps:
    - script@1:
        summary: Custom script for sharding.
```

### `steps:<step-id>:description`

A detailed description of the <NT>Step</NT>.

**<NT>Example</NT> of `steps:<step-id>:description`**

```yaml
workflows:
    steps:
    - script@1:
        summary: A custom script for setting up test sharding for Flutter unit tests.
```

### `steps:<step-id>:is_always_run`

If this property is true, the <NT>Step</NT> will always run, even if a previous <NT>Step</NT> in the <NT>Workflow</NT> failed.

By default, if a previous <NT>Step</NT> fails in the <NT>Workflow</NT>, subsequent <NT>Steps</NT> will not run. A <NT>Step</NT> will only run if this property is set to `true`.

**<NT>Supported values</NT>**: boolean

Read more: [Skipping Steps](/en/bitrise-ci/workflows-and-pipelines/steps/skipping-steps).

**<NT>Example</NT> of `steps:<step-id>:is_always_run`**

```yaml
workflows:
  primary:
    steps:
    - script:
        is_always_run: true
```

### `steps:<step-id>:is_skippable`

If this property is true, the build won't fail even if this <NT>Step</NT> fails.

By default if a <NT>Step</NT> fails, subsequent <NT>Steps</NT> are not executed. If the first failing step has this property set to `true`, it won't make the build fail. Subsequent <NT>Steps</NT> will still run and the build can finish successfully.

**<NT>Supported values</NT>**: boolean

**<NT>Example</NT> of `steps:<step-id>:is_skippable`**

```yaml
workflows:
  primary:
    steps:
    - script:
        is_skippable: true
```

### `steps:<step-id>:run_if`

This property sets conditions for running a <NT>Step</NT>. It requires a valid Go template expression.

A `run_if` can be any valid Go template, as long as it evaluates to true or false (or any of the String representation, for example `True`, `t`, `yes` or `y` are all considered to be true). If the template evaluates to `true`, the <NT>Step</NT> will run, otherwise it won’t.

Read more:

- [Enabling or disabling a Step conditionally](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally).
- [Examples of run_if expressions](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally#examples-of-run_if-expressions).

**<NT>Example</NT> of `steps:<step-id>:run_if`**

In this example, the build skips the <NT>Step</NT> if the value of CUSTOM_ENV_VAR_KEY is not `test value to test against`.

```yaml
run_if: |-
        {{enveq "CUSTOM_ENV_VAR_KEY" "test value to test against"}}
```

### `steps:<step-id>:timeout`

This property defines a time limit for a <NT>Step</NT>: if the <NT>Step</NT> runs longer than the defined time, the <NT>Step</NT> fails. Define the limit in seconds.

This is useful if, for example, your builds hang for not immediately obvious reasons - you can set timeouts for the <NT>Step</NT> or <NT>Steps</NT> which are suspected to have caused the problem.

Read more: [Setting a time limit for Steps](/en/bitrise-ci/workflows-and-pipelines/steps/setting-a-time-limit-for-steps).

**<NT>Example</NT> of `steps:<step-id>:timeout`**

```yaml
- xcode-test:
     timeout: 120
```

### `steps:<step-id>:no_output_timeout`

This property defines a time limit for steps: if a <NT>Steps</NT> runs longer than the defined time without producing an output, the <NT>Step</NT> fails. Define the limit in seconds.

Read more: [Detecting and aborting hanging Steps](/en/bitrise-ci/workflows-and-pipelines/steps/detecting-and-aborting-hanging-steps).

**<NT>Example</NT> of `steps:<step-id>:no_output_timeout`**

```yaml
output_slows_down:
    steps:
    - script:
        no_output_timeout: 12
```

### `steps:<step-id>:inputs`

Inputs defined for the <NT>Step</NT>: these are values that can be set when the <NT>Step</NT> is added to a <NT>Workflow</NT>.

<NT>Step</NT> input syntax consists of a `KEY: value` pair.

Read more: [Step inputs reference](/en/bitrise-ci/references/steps-reference/step-inputs-reference).

**<NT>Example</NT> of `steps:<step-id>:inputs`**

```yaml
inputs:
- my_key_for_the_env: "default value"
```

### `steps:<step-id>:outputs`

Outputs defined for the <NT>Step</NT>: these are values that the <NT>Step</NT> can set during its execution, which can be used by other <NT>Steps</NT> in the <NT>Workflow</NT>.

You can check out the default outputs of a <NT>Step</NT> in the <NT>Workflow Editor</NT> on bitrise.io or in the `step.yml` file of the <NT>Step</NT>. <NT>Step</NT> outputs can be defined in the `step.yml` file of the project by setting the `outputs` attribute. <NT>Step</NT> out syntax consists of two main parts: a `KEY: value` pair and an `opts` field. The key and the value are required, the `opts` field is optional.

Read more: [Step outputs reference](/en/bitrise-ci/references/steps-reference/step-outputs-reference).

**<NT>Example</NT> of `steps:<step-id>:outputs`**

```yaml
workflows:
  primary:
  steps:
  - gradle-runner:
      outputs:
      - BITRISE_APK_PATH: ALIAS_APK_PATH
```


## Trigger properties

### `triggers:enabled`

The property determines whether a defined trigger is active. The <NT>default value</NT> is `true`. If you want to disable the trigger, set it to `false`.

**<NT>Supported values</NT>**: boolean

**<NT>Example</NT> of `triggers:enabled`**

In this example, the trigger is disabled.

```yaml
workflows:
  primary:
    triggers:
      enabled: false
```

### `triggers:push`

Code push triggers define the conditions for triggering a <NT>Bitrise</NT> build when code is pushed to the project's repository.

For example, a commit to the specified branch of the project's repository triggers a build.

**<NT>Example</NT> of `triggers:push`**

In this example, a code push to the `main` branch triggers a build.

```yaml
workflows:
  primary:
    triggers:
      push:
      - branch: main
```

#### `triggers:push:branch`

Triggers a build when code is pushed to the specified branch.

**<NT>Supported values</NT>**:

- A string.
- The `pattern` property which allows simple text matching within all types of triggers.
- The `regex` property which allows using regular expressions as a trigger condition.

:::important[One type only]

Use exactly one out of a string, a pattern, or a regex. You can't use more than one in the same trigger!

:::

**<NT>Example</NT> of `triggers:push:branch`**

```yaml
workflows:
  primary:
    triggers:
      push:
      - branch: main
```

##### `triggers:push:commit_message`

Triggers a build when code is pushed with the specified commit message.

**<NT>Supported values</NT>**:

- A string.
- The `pattern` property which allows simple text matching within all types of triggers.
- The `regex` property which allows using regular expressions as a trigger condition.
- The `last_commit` property that defines whether <NT>Bitrise</NT> should evaluate every commit message or changed file in a code push or only those belonging to the most recent commit. Its <NT>default value</NT> is `false`.

:::important[One type only]

Use exactly one out of a string, a pattern, or a regex. You can't use more than one in the same trigger!

:::

**<NT>Example</NT> of `triggers:push:commit_message`**

In this example, a commit message that contains the string `[workflow: deploy]` triggers a build:

```yaml
primary
  triggers:
    push:
    - commit_message:
        regex: '.*\[workflow: deploy\].*'
```

##### `triggers:push:changed_files`

Triggers a build when a code push results in changes to the specified file or files.

You can specify files or a folders.

**<NT>Supported values</NT>**:

- A string.
- The `pattern` property which allows simple text matching within all types of triggers.
- The `regex` property which allows using regular expressions as a trigger condition.
- The `last_commit` property that defines whether <NT>Bitrise</NT> should evaluate every commit message or changed file in a code push or only those belonging to the most recent commit. Its <NT>default value</NT> is `false`.

:::important[One type only]

Use exactly one out of a string, a pattern, or a regex. You can't use more than one in the same trigger!

:::

**<NT>Example</NT> of `triggers:push:changed_files`**

```yaml
primary
  triggers:
    push:
    - changed_files: 
        pattern: "myfile.txt"
```

## <NT>Docker</NT> container properties

### `containers:<container-id>:type`

The type of the container.

**<NT>Supported values</NT>**:

- `execution`: Containers that run a <NT>Step</NT> or a <NT>Step bundle</NT>.
- `service`: Containers that run alongside a <NT>Step</NT> or <NT>Step bundle</NT>, running background services, such as `postgres`.

Read more: [About Docker containers on Bitrise](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/about-docker-containers-on-bitrise).

**<NT>Example</NT> of `containers:<container-id>:type`**

```yaml
containers:
  node-18:
    type: execution
```

### `containers:<container-id>:image`

This property defines the <NT>Docker</NT> image used for the container, in a `name:tag` format. You can use any public image from <NT>Docker</NT> Hub.

**<NT>Example</NT> of `containers:<container-id>:image`**

```yaml
containers:
  node-18:
    image: node:18
```

### `containers:<container-id>:credentials`

The credentials allow you to control the parameters supplied for the `docker login` command. Check your registries documentation on how to use docker login.

The following properties are supported:

- `username`
- `password`
- `server`

**<NT>Example</NT> of `containers:<container-id>:credentials`**

```yaml
containers:
  node-18:
    credentials:
      username: user
      password: pass
```

#### `containers:<container-id>:credentials:username`

The username that will be used with the `docker login` command.

**Example of `containers:<container-id>:credentials:username`**

```yaml
containers:
  node-18:
    credentials:
      username: _json_key_base64
```

#### `containers:<container-id>:credentials:password`

The password that will be used with the `docker login` command.

**Example of `containers:<container-id>:credentials:password`**

```yaml
containers:
  node-18:
    credentials:
      password: $GCP_SERVICE_ACCOUNT
```

#### `containers:<container-id>:credentials:server`

Fully qualified registry server URL for the `docker login` command. This is optional if the server is already part of the image reference.

**Example of `containers:<container-id>:credentials:server`**

```yaml
containers:
  node-18:
    credentials:
      server: ghcr.io
```

### `containers:<container-id>:ports`

This property defines an array of ports which are used to access the service. Define ports in the format of `[HostPort]:[ContainerPort]`.

**<NT>Example</NT> of `containers:<container-id>:ports`**

```yaml
containers:
  ports:
  - 3000:3000
  - 2000:2000
```

### `containers:<container-id>:envs`

Define an array of <NT>Environment Variables</NT> for the <NT>Docker</NT> container. For each Env Var, define a key and a value.

Read more: [Environment Variables](/en/bitrise-ci/configure-builds/environment-variables).

**<NT>Example</NT> of `containers:<container-id>:envs`**

```yaml
containers:
  node-18:
    envs:
    - VARIABLE_KEY: variable_value
```

### `containers:<container-id>:options`

Define additional <NT>Docker</NT> container resource options. These are parameters that will be passed to the `docker create` command.

Read more about the possible options: [docker service create](https://docs.docker.com/reference/cli/docker/service/create/).

:::important[Not supported options]

`--network`, `--volume (-v)`, and `--entrypoint` are not supported.

:::

**<NT>Example</NT> of `containers:<container-id>:options`**

```yaml
containers:
  node-18:
    options: "--privileged --health-interval 1s"
```
