A <NT><GlossTerm baseform="Step">Step</GlossTerm></NT> reference can contain three components:

1. The StepLib source: a library of <NT>Step</NT> repositories, such as the official [Bitrise Step Library](https://github.com/bitrise-steplib).
1. The <NT>Step ID</NT>: every <NT>Step</NT> must have an ID as part of its `<GlossTerm baseform="step.yml">step.yml</GlossTerm>` definition.
1. The <NT>Step</NT> version: the numerical version of the <NT>Step</NT> you want to include.

Use the following syntax to reference a <NT>Step</NT>:

```yaml
<step_lib_source>::<step-id>@<version>:
```

From these three components only the <NT>Step ID</NT> is required. For example:

```yaml
- script:
```

This could be written as:

```text
https://github.com/bitrise-io/bitrise-steplib.git::script@1:
```

- If the version is not defined, the <NT>latest version</NT> of the step will be used.
- If the StepLib source is not defined, the `default_step_lib_source` will be used.

We recommend pinning the major version of a <NT>Step</NT> when referencing it. In the example above, `script@1` means major version 1 of the [**Script**](https://github.com/bitrise-io/steps-script) <NT>Step</NT> is pinned: the <NT>Step</NT> will run with the latest available minor and patch versions of the <NT>Step</NT>. For example, if the latest minor version is 1.2, and the latest patch version is 1.2.2 then pinning major version 1 means the <NT>Step</NT> will use version 1.2.2.


## Special <NT>Step</NT> sources

There are two special <NT>Step</NT> sources:

- `git::`
- and `path::`

When you use one of these sources, the <NT>Step</NT> won’t be identified through a <NT>Step</NT> Library but through the ID data you specify.

### The git:: source

The `git::` source is the repository of the <NT>Step</NT> on your git hosting provider. The `script` <NT>Step</NT>’s <NT>GitHub</NT> source is: `https://github.com/bitrise-io/steps-script`. To reference the `script` <NT>Step</NT> directly through a git reference, you can use the `git::` source, the <NT>Step</NT>’s <NT>git clone</NT> URL, and the branch or tag in the repository.

To reference the `1.1.3` version tag of the script <NT>Step</NT>’s repository:

```yaml
- git::https://github.com/bitrise-io/steps-script.git@1.1.3:
```

In general, whenever you can use a <NT>Step</NT> version through a <NT>Step</NT> Library, you should do that, instead of using the `git::` source type. Features like local step caching, network caching, or alternative download URLs are only supported for <NT>Steps</NT> shared in a StepLib.

But this type of referencing allows certain things you can’t get through a StepLib. For example, the `git::` source type can be used for not-yet-published or work-in-progress states of a <NT>Step</NT>. If you develop your own <NT>Step</NT>, you can use this `git::` source type to test your step before you publish it in a StepLib.

To reference `soon-to-be-released` branch of your repository where you're developing a <NT>Step</NT>:

```yaml
- git::https://github.com/bitrise-io/steps-script.git@soon-to-be-released:
```

### The path:: source

The `path::` source specifies a local path to a <NT>Step</NT>'s repository, and it requires no version information. Both absolute and relative local paths are supported. A relative path is relative to the folder containing the `bitrise.yml` file.

```yaml
- path::/path/to/my/step:

- path::./relative/path:
```

This is useful for <NT>Step</NT> development. It can also be used if you want to include your build <NT>Steps</NT> in your project’s source code.
