## Project {#project}

A <NT>Bitrise</NT> <NT>project</NT> is the container for the entire Mobile DevOps process of your development work. Each <NT>workspace</NT> can own multiple projects. A project allows you to create a CI configuration and set up <NT>Release Management</NT> to distribute your mobile app to testers and to online stores.

## Configuration YAML {#configuration-yaml}

The configuration YAML file stores your entire build configuration for a project. It specifies your stack and the build triggers, and defines the <NT>Workflows</NT> of the app. When you make changes on the graphical UI of our <NT>Workflow Editor</NT>, you actually modify your configuration YAML.

## bitrise.yml {#bitriseyml}

The configuration YAML file stores your entire build configuration for a project. It specifies your stack and the build triggers, and defines the <NT>Workflows</NT> of the app. When you make changes on the graphical UI of our <NT>Workflow Editor</NT>, you actually modify your configuration YAML.

## <NT>Pipeline</NT> {#pipeline}

A <NT>Bitrise</NT> <NT>Pipeline</NT> is the top level of the <NT>Bitrise CI</NT>/CD configuration. <NT>Pipelines</NT> can be used to organize the entire CI/CD process and to set up advanced configurations with multiple different tasks running parallel and/or sequentially.

## <NT>Project scanner</NT> {#project-scanner}

The project scanner is a tool that identifies the given project's type and generates a basic <NT>Bitrise</NT> configuration. Each supported project type has its own scanner: these scanners are stored as separate packages.

## Secret {#secret}

A Secret is a specific type of [Environment Variable](/en/bitrise-ci/configure-builds/environment-variables): they hide their information in an encrypted format and their value is not exposed in the build logs nor in the `bitrise.yml` configuration. You can store confidential information, such as passwords or API keys as <NT>Secrets</NT>.

## <NT>Stack</NT> {#stack}

A build stack indicates the full configuration of the virtual machine that <NT>Bitrise</NT> uses to run your build. Each stack includes an operating system, and a large number of pre-installed software and tools. For example, our <NT>Xcode</NT> stacks run on macOS operating systems and contain, among many other tools, the <NT>Xcode</NT> version that is indicated in the name of the stack.

## Stage {#stage}

A Stage is a collection of <NT>Workflows</NT>. Stages are the top-level building blocks of <NT>Pipelines</NT>. A Stage can contain multiple <NT>Workflows</NT>, which all run in parallel in the same Stage. If all <NT>Workflows</NT> are successful in a Stage, the <NT>Pipeline</NT> moves on to the next Stage. If any of the <NT>Workflows</NT> fail, the <NT>Pipeline</NT> ends without running the other Stages unless you configure a given Stage to always run.

## <NT>Step</NT> {#step}

A <NT>Step</NT> is a block of script execution that encapsulates a build task on <NT>Bitrise</NT>: the code to perform that task, the inputs and parameters you can define for the task, and the outputs the task generates.

## step.yml {#stepyml}

The interface definition of a <NT>Bitrise Step</NT>. It defines the <NT>Step inputs</NT> and the generated outputs, as well as any other <NT>Step</NT> property such as the category and the description of the <NT>Step</NT>. It also points to the <NT>Step</NT>'s source code.

## Trigger {#trigger}

A trigger or build trigger is a configuration for automatically launching a <NT>Bitrise</NT> build when a specified event happens. Triggers require a webhook set up at your Git hosting provider.

## <NT>Workflow</NT> {#workflow}

A <NT>Workflow</NT> is a collection of <NT>Steps</NT>, <NT>Environment Variables</NT>, and other configurations. When <NT>Bitrise</NT> starts a build, it runs one or more <NT>Workflows</NT> according to the configuration defined in the `bitrise.yml` file.

## <NT>Workflow Editor</NT> {#workflow-editor}

The <NT>Bitrise</NT> <NT>Workflow Editor</NT> allows you to edit your <NT>Workflows</NT>, configure <NT>Steps</NT>, upload files (including <NT>code signing</NT> files) and manage your app's triggers and stacks on a graphical user interface. It is available both online and [offline](/en/bitrise-ci/bitrise-cli/installing-and-upgrading-the-offline-workflow-editor).

## <NT>Workspace</NT> {#<NT>workspace</NT>}

A <NT>Workspace</NT> is an environment that allows you to manage your <NT>Bitrise</NT> apps and the team members working on the apps. You can create multiple <NT>Workspaces</NT>, and you can be invited to <NT>Workspaces</NT> by other <NT>Bitrise</NT> users. To be able to add apps and run builds, you either need to be part of a <NT>Workspace</NT>, or you have to be an outside contributor on an app's team.

## Remote Dev Environment {#remote-dev-environment}

A Remote Dev Environment (RDE) is an on-demand cloud development machine — macOS or <NT>Linux</NT> — that runs on <NT>Bitrise</NT> infrastructure, using the same stacks and caches as <NT>Bitrise CI</NT>. You create a session, connect to it from a terminal, IDE, or AI agent, and archive or restore it later.

## Session {#session}

A session is a single Remote Dev Environment instance: a running (or restorable) cloud machine created from a template or directly from a <NT>stack and machine type</NT>. Terminating a session preserves its persistent disk so you can restore it later.

## Template {#template}

A template is a reusable Remote Dev Environment configuration — stack, machine type, warmup and startup scripts, inputs, and IDE folder links — that you create sessions from.

## Saved input {#saved-input}

A saved input is a reusable, user-scoped value or credential (such as an SSH public key or an API token) that you can map into Remote Dev Environment sessions. Secret values are encrypted at rest.
