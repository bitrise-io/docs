With <NT>Bitrise</NT>’s iOS device testing solution, you can run UI tests for iOS apps on physical devices without having to set up and register your own devices: you just need to use our dedicated <NT><GlossTerm baseform="Step">Step</GlossTerm></NT>s and set the device type(s) on which you want to test your app. There are no limits to using the <NT>Step</NT>, other than your overall build time limit. It also works for iOS apps developed using other frameworks, such as Flutter or <NT>React Native</NT>.

Our device testing solution is based on [Firebase Test Lab](https://firebase.google.com/docs/test-lab/): it uses real, production devices running in a Google data center to test your app. The devices are flashed with updated APIs and have customizable locale settings. You can find the resulting logs, videos and screenshots on <NT>Bitrise</NT>.

For iOS apps, <NT>Firebase</NT> Test Lab runs [XCTest](https://developer.apple.com/documentation/xctest) tests. Find the list of the available devices [here](https://firebase.google.com/docs/test-lab/ios/available-testing-devices).

:::note[Limitations]

You might be limited by your overall build time. Also note that a single build can contain only one [**iOS Device Testing**](https://github.com/bitrise-steplib/steps-virtual-device-testing-for-ios) <NT>Step</NT>, performing one type of test. This is because <NT>Bitrise</NT> sends the build slug to <NT>Firebase</NT> Test Lab. Sending the same build slug results in a `Build already exists`error.

:::


## Running device tests with <NT>Firebase</NT> for iOS apps

To run device tests for iOS apps with the <NT>Firebase</NT> Test Lab solution, you will need to add two <NT>Steps</NT> to your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>:

- **<NT>Xcode Build for testing for iOS</NT>**.
- **<NT>iOS Device Testing</NT>**.

:::note[Multiplatform apps]

The <NT>Step</NT> can run device tests for iOS apps built with other frameworks, such as Flutter or <NT>React Native</NT>. You just need to make sure the **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT> can access your app's `.xcodeproj` or `.xcworkspace` file.

:::

The **[Xcode Build for testing for iOS](https://www.bitrise.io/integrations/steps/xcode-build-for-test)** <NT>Step</NT> performs the `xcodebuild` command’s `build-for-testing` action: it builds the tests defined in your iOS app’s [Xcode scheme](https://developer.apple.com/library/archive/featuredarticles/XcodeConcepts/Concept-Schemes.html). The <NT>Step</NT> exports a .zip file that contains your test directory (by default, it’s `Debug-iphoneos`) and the `xctestrun` file.

To use this <NT>Step</NT>, you will need [code signing](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) files for the test app. You can choose between using automatic provisioning and manual provisioning.

The **[iOS Device Testing](https://github.com/bitrise-steplib/steps-virtual-device-testing-for-ios)** <NT>Step</NT> takes the path to this .zip file - exported as an Environment Variable - as input to run your tests and export the test results to <NT>Bitrise</NT>.

:::important[Automatic vs manual <NT>code signing</NT> asset management]

The **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT> can manage your <NT>code signing</NT> assets without you having to manually manage your provisioning profiles. This only requires you to select the relevant inputs in the <NT>Step</NT> (see below) and upload the right certificates to the **<NT>Code Signing</NT>** tab. This is the [automatic code signing asset management](/en/bitrise-ci/code-signing/ios-code-signing/managing-ios-code-signing-files---automatic-provisioning) method.

If you decide to choose the [manual code signing asset management](/en/bitrise-ci/code-signing/ios-code-signing/managing-ios-code-signing-files---manual-provisioning), then add the **Certificate and Profile Installer** <NT>Step</NT> before the **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT>. The former will take care of the <NT>code signing</NT> asset management while the latter builds your project. Make sure the **<NT>Automatic code signing method</NT>** input of the **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT> is set to `off`.

The instructions below detail the steps of setting up <NT>automatic code signing</NT> asset management on <NT>Bitrise</NT> with the **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT>.

:::

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Add the **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.

   :::tip[Build for testing]

   This guide only mentions the most important inputs for setting up an app for device testing. For more information about the <NT>Step</NT>, see [Xcode Build for testing for iOS](https://bitrise.io/integrations/steps/xcode-build-for-test).

   :::
1. In the **<NT>Scheme name</NT>** input, set the <NT>Xcode</NT> Scheme you want to use. By default, the value of the input is the `$BITRISE_SCHEME` [Environment Variable](/en/bitrise-ci/configure-builds/environment-variables) which is stored when your app is created.
1. In the **<NT>Build</NT> <NT>Configuration</NT>** input, add the name of the [Xcode build configuration](https://developer.apple.com/documentation/xcode/configuring-the-build-settings-of-a-target) you want to use.

   If no configuration is specified, the <NT>Xcode</NT> project's default build configuration will be used.
1. Set a destination option for `xcodebuild` in the **<NT>Device destination specifier</NT>** input of the <NT>Step</NT>.

   The <NT>default value</NT> is `generic/platform=ios`: this means the tests can be run on any iOS device.
1. Set the **<NT>Automatic code signing method</NT>** input to the <NT>Apple service connection</NT> you want to use for <NT>code signing</NT>. The available options are:

   - `off` if you don’t do <NT>automatic code signing</NT>.
   - `api-key` if you use [API key authorization](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).
   - `apple-id`if you use [Apple ID authorization](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).
1. Add the **<NT>iOS Device Testing</NT>** <NT>Step</NT> to the <NT>Workflow</NT>.The <NT>Step</NT> has to come after the **<NT>Xcode Build for testing for iOS</NT>** <NT>Step</NT>.

   :::important[API setting inputs]

   The inputs related to the Test API (**Test API's base URL** and **API token** cannot be changed on the graphical UI. You should not attempt to modify their values.

   :::
1. In the **<NT>Test devices</NT>** input field, specify the devices on which you want to test the app.

   Find the list of the available devices [here](https://firebase.google.com/docs/test-lab/ios/available-testing-devices). You can add multiple devices to the input, in the following format: `deviceID,version,language,orientation`
1. Make sure you have the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>, with version 1.4.1 or newer. With the older versions of the <NT>Step</NT>, you won’t be able to check your results in the **<NT>Tests</NT>** tab!
1. [Start a build](/en/bitrise-ci/run-and-analyze-builds/starting-builds/approving-pull-request-builds).

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Add the `xcode-build-for-test` <NT>Step</NT> to your <NT>Workflow</NT>.

   :::tip[Build for testing]

   This guide only mentions the most important inputs for setting up an app for device testing. For more information about the <NT>Step</NT>, see [Xcode Build for testing for iOS](https://bitrise.io/integrations/steps/xcode-build-for-test).

   :::

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
   ```
1. In the `scheme` input, set the <NT>Xcode</NT> Scheme you want to use. By default, the value of the input is the `$BITRISE_SCHEME` Environment Variable which is stored when your app is created.

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
   ```
1. In the `configuration` input, add the name of the [Xcode build configuration](https://developer.apple.com/documentation/xcode/configuring-the-build-settings-of-a-target) you want to use.

   If no configuration is specified, the <NT>Xcode</NT> project's default build configuration will be used.

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
           - configuration: Debug
   ```
1. Set a destination option for `xcodebuild` in the `destination` input of the <NT>Step</NT>.

   The <NT>default value</NT> is `generic/platform=ios`: this means the tests can be run on any iOS device.

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
           - configuration: Debug
           - destination: generic/platform=iOS
   ```
1. Set the `automatic_code_signing` input to the <NT>Apple service connection</NT> you want to use for <NT>code signing</NT>. The available options are:

   - `off` if you don’t do <NT>automatic code signing</NT>.
   - `api-key` if you use [API key authorization](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).
   - `apple-id` if you use [Apple ID authorization](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
           - configuration: Debug
           - destination: generic/platform=iOS
           - automatic_code_signing: api-key
   ```
1. Add the `virtual-device-testing-for-ios` <NT>Step</NT> to the <NT>Workflow</NT>. The <NT>Step</NT> has to come after the `xcode-build-for-test` <NT>Step</NT>.

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
           - configuration: Debug
           - destination: generic/platform=iOS
           - automatic_code_signing: api-key
       - virtual-device-testing-for-ios:
           inputs:
   ```
1. In the `test_devices` input field, specify the devices on which you want to test the app.

   Find the list of the available devices [here](https://firebase.google.com/docs/test-lab/ios/available-testing-devices). You can add multiple devices to the input, in the following format: `deviceID,version,language,orientation`

   ```yaml
   my-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
           - configuration: Debug
           - destination: generic/platform=iOS
           - automatic_code_signing: api-key
       - virtual-device-testing-for-ios:
           inputs:
           - test_devices: 'iphone8,14.7,en,portrait
   ```
1. Make sure you have the `deploy-to-bitrise-io` <NT>Step</NT> in your <NT>Workflow</NT>, with version 1.4.1 or newer. With the older versions of the <NT>Step</NT>, you won’t be able to check your results in the **<NT>Tests</NT>** tab!

   ```yaml
   fmy-workflow:
     steps:
       - git-clone: {}
       - xcode-build-for-test:
           inputs:
           - scheme: $BITRISE_SCHEME
           - configuration: Debug
           - destination: generic/platform=iOS
           - test_plan: test
           - automatic_code_signing: api-key
           - project_path: $BITRISE_PROJECT_PATH
       - virtual-device-testing-for-ios:
           inputs:
           - test_devices: 'iphone8,14.7,en,portrait'
       - deploy-to-bitrise-io: {}
   ```
1. [Start a build](/en/bitrise-ci/run-and-analyze-builds/starting-builds/approving-pull-request-builds).

</TabItem>
</Tabs>

If all goes well, you should be able to view your results among the [build artifacts](/en/bitrise-ci/run-and-analyze-builds/managing-build-files/build-artifacts-online) or in the <NT>Tests</NT> tab.
