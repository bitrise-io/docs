View your test results in one place after a <NT>Bitrise CI</NT> build. All tests will be available on the build page, even if you have multiple testing <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s in your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s. Our goal is to make sure you have accessible and `actionable` test results on <NT>Bitrise</NT>, leading to faster time to recover from test failures and a better overall developer experience.


## Deploying test results

Deploy your test results on the build page with minimal configuration. The requirements depend on your setup:

- Using the official <NT>Bitrise</NT> testing <NT>Steps</NT> to run tests
- Using other <NT>Steps</NT> to run tests

### Deploying results from the official <NT>Bitrise</NT> testing <NT>Steps</NT>

The supported testing <NT>Steps</NT> are the following:

- **<NT>Android Unit Test</NT>**: [Android unit tests](/en/bitrise-ci/testing/testing-android-apps/android-unit-tests).
- **<NT>Android Instrumented Test</NT>**: [Running instrumented tests for Android apps](/en/bitrise-ci/testing/testing-android-apps/running-instrumented-tests-for-android-apps).
- **<NT>Xcode Test for iOS</NT>**: [Running unit and UI tests for iOS apps](/en/bitrise-ci/testing/testing-ios-apps/running-unit-and-ui-tests-for-ios-apps).
- **<NT>iOS Device Testing</NT>**: [Device testing for iOS](/en/bitrise-ci/testing/device-testing-with-firebase/device-testing-for-ios).
- **<NT>Virtual Device Testing for Android</NT>**: [Device testing for Android](/en/bitrise-ci/testing/device-testing-with-firebase/device-testing-for-android).
- **<NT>Flutter Test</NT>**: [Testing Flutter apps](/en/bitrise-ci/testing/testing-flutter-apps/running-the-dart-analyzer-on-bitrise).

If you use any of these <NT>Steps</NT>, make sure you have the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>. <NT>Bitrise</NT> will automatically deploy your test results to the build page.

:::important[Attachments and flaky tests]

If your tests generate attachments or you want <NT>Bitrise</NT> to show flaky tests, the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> must be of version 2.19.1 or newer.

:::

### Deploying results from other <NT>Steps</NT>

If you run your tests using other <NT>Steps</NT> (for example, you can use [Script](https://bitrise.io/integrations/steps/script) <NT>Steps</NT> for a fully custom testing solution), you need some additional configuration.

1. Add the <NT>Step</NT> running your tests to your <NT>Workflow</NT>.
1. Add the **Export <NT>test results</NT> to the Test Reports** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Configure the <NT>Step</NT>: [Using the Export test results to the Test Reports Step](/en/bitrise-ci/testing/deploying-and-viewing-test-results#using-the-export-test-results-to-test-reports-step).
1. Make sure you have the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.

   :::important[Attachments and flaky tests]

   If your tests generate attachments or you want <NT>Bitrise</NT> to show flaky tests, the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> must be of version 2.19.1 or newer.

   :::


## Viewing test results

:::note[Rich HTML reporting]

If you use [rich HTML test reports](/en/bitrise-ci/testing/testing-ios-apps/viewing-xcode-test-results-in-rich-html-format), you can still find those on the **<NT>Artifacts</NT>** tab.

:::

To view your results, open the build page and select the **<NT>Tests</NT>** tab. Your tests are sorted into different tabs based on their status:

- Failed
- Passed
- Skipped
- Error
- Flaky

By default, <NT>Bitrise</NT> shows the list of failed tests. You can view test run details within each of these categories.

Click any test to check the details: the duration of the test, the output, and any attachments that the test generated. For easier debugging of failures, test reports display attached image files in line with their associated test cases.

:::note[Supported file formats]

Currently, we support the following file formats for attachments:

`.jpg, .jpeg, .png, .txt, .log, .mp4, .webm, .ogg`

:::

You can view flaky tests on the **<NT>Flaky</NT>** tab. Flaky tests are tests that sometimes fail and sometimes succeed without any changes in the code. If a test fails and then succeeds on [an automatic retry](/en/bitrise-ci/testing/testing-ios-apps/running-unit-and-ui-tests-for-ios-apps#test-repetitions), <NT>Bitrise</NT> marks the test as flaky, and you can view it on the **<NT>Flaky</NT>** tab. It also displays the details and attachments for each retry of the test. You can find more information on how to detect flaky tests and how to quarantine them, if needed, in [Detecting and quarantining flaky tests](/en/bitrise-ci/testing/detecting-and-quarantining-flaky-tests).

![test-run.png](/img/_paligo/uuid-53eba539-6b4d-b4cf-c33d-6168fca79640.png)

Test reports also support the use of the `system-out` tag in JUnit. If a test case includes this tag, we display its contents on the **<NT>Test output</NT>** tab. If a test case has both a `system-out` tag and produces an error message, we display both.


## Collating test attachments with test results

Many tests generate image files (screenshot tests, XCUITests, etc). <NT>Bitrise</NT> test reports automatically collate image files with associated test cases for <NT>Xcode</NT> tests (using the `xcresult` file).

:::note[Supported file formats]

Currently, we support the following file formats for attachments:

`.jpg, .jpeg, .png, .txt, .log, .mp4, .webm, .ogg`

:::

However, if your test doesn't generate an `xcresult` file, you can still achieve the same thing by generating a JUnit XML file and using <NT>Bitrise Steps</NT>:

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Generate a `JUnit.xml` file from your tests.

   The general file structure should look something like this:

   ```xml
   <?xml version="1.0" encoding="UTF-8"?>
   <testsuites time="15.682687">
       <testsuite name="Tests.Registration" time="6.605871">
           <testcase name="testCase1" classname="Tests.Registration" time="2.113871" />
       </testsuite>
   </testsuites>
   ```
1. Add a `<properties>` element to associate an attachment with a `testcase`.

   The `name` attribute of the element must be set with a value `attachment_#` where `#` is the ordered index of the attachment file, and the value is the filename. <NT>Bitrise</NT> will show attachments for any test with the attachment properties, but it's most common to only attach screenshots to failed tests. Mark a test as failed with a `<failure>` element.

   ```xml
   <testcase name="testCase1" classname="Tests.Registration" time="2.113871" />
     <failure message="Assertion error message" type="AssertionError">
       Call stack printed here
     </failure>
     <properties>
       <property name="attachment_1" value="pp1.jpg" />
       <property name="attachment_2" value="pp2.jpg" />
     </properties>            
   </testcase>
   ```
1. Add the **<NT>Export test results to Test Reports</NT>** <NT>Step</NT> to your <NT>Bitrise</NT> <NT>Workflow</NT>. It should be version 1.1.0 or higher.
1. Set a test name in the **<NT>The name of the test</NT>** input.
1. Set the path to your JUnit XML file in the **<NT>Test result search pattern</NT>** input.
1. Add the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> to the end of your <NT>Workflow</NT>.

   You don't have to change the default input values.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

   The general file structure should look something like this:

   ```xml
   <?xml version="1.0" encoding="UTF-8"?>
   ```

1. Add the `custom-test-results-export` <NT>Step</NT> to your <NT>Bitrise</NT> <NT>Workflow</NT> after generating your JUnit XML file. It should be version 1.1.0 or higher.

   ```yaml
   workflows:
     inline_attachment:
       steps:
       - generate-text-file:
           inputs:
           - file_name: junit.xml
           - file_content: "xml content with references to image files here"
       - custom-test-results-export:
   ```
1. Set a test name in the `test_name` input.

   ```yaml
   - custom-test-results-export@1:
       inputs:
       - test_name: example_tests
   ```
1. Set the path to your JUnit XML file in the `search_pattern` input.

   ```yaml
   - custom-test-results-export@1:
       inputs:
       - test_name: example_tests
       - search_pattern: junit.xml
   ```
1. Add the `deploy-to-bitrise-io` <NT>Step</NT> to the end of your <NT>Workflow</NT>.

   You don't have to change the default input values. The full <NT>Workflow</NT> in your <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file might look something like this:

   ```yaml
   workflows:
     inline_attachment:
       steps:
       - generate-text-file@0:
           inputs:
           - file_name: junit.xml
           - file_content: "xml content with references to image files here"
       - custom-test-results-export@1:
           inputs:
           - test_name: example_tests
           - base_path: "."
           - search_pattern: junit.xml
       - script@1:
           inputs:
           - content: |
               #!/usr/bin/env bash
               set -ex
               set -o pipefail

               TEST_RESULTS_DIR=$(find "$BITRISE_TEST_DEPLOY_DIR" -type d -name "example_tests" -print -quit)
               cp ~/*.jpg $TEST_RESULTS_DIR
           title: Copy image files to test directory
       - deploy-to-bitrise-io@2: {}
   ```

## Using the <NT>Export test results to Test Reports</NT> <NT>Step</NT>

You can use the **<NT>Export test results to Test Reports</NT>** <NT>Step</NT> to make sure your test results appear on the **<NT>Tests</NT>** tab, even if you use <NT>Steps</NT> that don’t automatically export their results.

With the correct configuration, the <NT>Step</NT> finds the test results in your project’s repository and puts them in the export directory.

1. <Partial_OpeningTheWorkflowEditor />
1. Add the **<NT>Export test results to Test Reports</NT>** <NT>Step</NT> to your <NT>Workflow</NT> after the <NT>Step</NT> that runs your tests.
1. In the **<NT>The name of the test</NT>** input, set <NT>the name of the test</NT> run.
1. In the **Test result base path** input, set the path where your test results can be found.

   ![Managing_an_app_s_bitrise.png](/img/_paligo/uuid-fd13332e-571f-e706-88ed-a20a29db7aea.png)

   We recommend setting a folder here, though you can also set a specific filepath. The <NT>default value</NT> is the source directory of your project. Example patterns:

   - If your project’s root directory is `app`: `app/build/test-results/testDemoDebugUnitTest/`
   - If your test results are within a project folder but `app` is not the root directory: ./app/build/test-results/testDemoDebugUnitTest/
1. In the **<NT>Test result search pattern</NT>** input, set a pattern that matches your test result file.

   This search pattern is used to search every file and folder of the path that was set in the **<NT>Path</NT> where custom test results reside** input.

   If there is more than one match, the <NT>Step</NT> will export the first match with a warning in the logs. If you set a specific filepath in the previous input, just set * here.

   Example patterns:

   - Matching all files within the base path: `*`
   - Matching all files within a given directory of the base path: `*/build/test-results/testDemoDebugUnitTest/*`
1. In the **<NT>Step</NT>’s test result directory** input, make sure the path is correct.

   Do NOT modify this input’s value: this is the folder where the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> will look for the test results to export them. It should be set to the `$BITRISE_TEST_RESULT_DIR` Env Var.
1. Make sure you have a **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.

   If your tests generate attachments, make sure the <NT>Step</NT> is of version 2.19.x or newer.
