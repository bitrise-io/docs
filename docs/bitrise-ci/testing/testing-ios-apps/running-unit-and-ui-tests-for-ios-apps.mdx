You can run both unit and UI tests for iOS apps on <NT>Bitrise</NT> with our dedicated <NT><GlossTerm baseform="Step">Step</GlossTerm></NT>. The <NT>Step</NT> runs the [test targets you defined](https://developer.apple.com/documentation/xctest/defining_test_cases_and_test_methods/) in your <NT>Xcode</NT> project and [exports the results for you to view it in detail](/en/bitrise-ci/testing/deploying-and-viewing-test-results).

To run <NT>Xcode</NT> tests on <NT>Bitrise</NT>, you need two <NT>Steps</NT> to run <NT>Xcode</NT> tests and view their results:

[**Xcode Test for iOS**](https://github.com/bitrise-steplib/steps-xcode-test).

**Deploy to**[**Bitrise.io**](http://Bitrise.io).

:::note[<NT>Code signing</NT> files]

Running <NT>Xcode</NT> tests and deploying their results to <NT>Bitrise</NT> do not require any [code signing](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) files. So don’t worry about them just yet!

:::

The [**Xcode Test for iOS**](https://github.com/bitrise-steplib/steps-xcode-test) <NT>Step</NT> runs your tests, exports your test results, the `.xcresult` file, all test attachments, and the log of the `xcodebuild test` command.

:::tip[<NT>Xcode</NT> test results in HTML]

You can also view your <NT>Xcode</NT> test results in a rich HTML format, using the [**Generate Xcode test report html**](https://github.com/bitrise-steplib/bitrise-step-generate-xcode-html-report) <NT>Step</NT>: [Viewing Xcode test results in rich HTML format](/en/bitrise-ci/testing/testing-ios-apps/viewing-xcode-test-results-in-rich-html-format).

:::

To run tests using the <NT>Step</NT>:


<Tabs>
  <TabItem value="workflow-editor" label="Workflow Editor">

  1. Make sure [you install all of the app's dependencies](/en/bitrise-ci/dependencies-and-caching/ios-dependencies/managing-dependencies-with-carthage) in your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>.
  1. Add the [**Xcode Test for iOS**](https://github.com/bitrise-steplib/steps-xcode-test) <NT>Step</NT> to the <NT>Workflow</NT>.
  1. Make sure the **<NT>Project path</NT>** input points to the correct location.

     The input asks for the path to your `.xcodeproj,``.xcworkspace` , or `Package.swift` file. In most cases, you don't need to change this input: when adding a new app, the <NT><GlossTerm baseform="project scanner">project scanner</GlossTerm></NT> automatically finds the relevant file and stores its location in the [Environment Variable](/en/bitrise-ci/configure-builds/environment-variables) that is the <NT>default value</NT> of the input.

     ![xcode-test-for-ios.png](/img/_paligo/uuid-bba07440-4913-8b44-0391-1dd275534500.png)
  1. Make sure the **Scheme** input points to the scheme you want to use to build the app.

     The <NT>default value</NT> is the [Environment Variable](/en/bitrise-ci/configure-builds/environment-variables) that stores the scheme you set during the initial configuration of the app. If you wish to use a different scheme, type its name to the input field.

     :::important[Shared scheme only]

     The scheme must be a shared <NT>Xcode</NT> scheme!

     :::
  1. Configure the device destination in the **<NT>Device destination specifier</NT>** input: the input takes comma-separated key-value pairs. For example, if you wish to build an app to test on an iPhone 14 with the latest available OS:

     ```bash
     platform=iOS,name=iPhone 14 Plus,OS=latest
     ```

     The input sets the `-destination` option of `xcodebuild`. Read more about the possible options: [How do I run unit tests from the command line?](https://developer.apple.com/library/archive/technotes/tn2339/_index.html#//apple_ref/doc/uid/DTS40014588-CH1-UNIT)
  1. Optionally, set a specific [test plan](https://developer.apple.com/documentation/xcode/organizing-tests-to-improve-feedback?changes=_8) in the **Test plan** input.

     The input sets the `-testPlan` option of the `test` action of `xcodebuild`. If you leave this empty, the test plan specified in the <NT>Xcode</NT> scheme will be used.
  1. Add the [**Deploy to Bitrise.io - Build Artifacts, Test Reports, and Pipeline intermediate files**](https://github.com/bitrise-steplib/steps-deploy-to-bitrise-io) <NT>Step</NT> to the end of your <NT>Workflow</NT> to be able to access the test results and other outputs: [Deploying and viewing test results](/en/bitrise-ci/testing/deploying-and-viewing-test-results).

:::note[Headless mode]

From <NT>Xcode</NT> 9 onwards, tests are run in headless mode by default: this means that the simulator will run in the background only. To change it, go to the <NT>Step</NT>’s Debug input group and set the **<NT>Run</NT> the simulator in headless mode** input’s value to `no`. However, with this option, tests will take more time.

```yaml
- xcode-test:
    inputs:
      - headless_mode: 'no'
```

:::

:::important[Using xcpretty]

The `xcpretty` output tool does not support parallel tests. If parallel tests are enabled in your project, go to the <NT>Step</NT>’s **xcodebuild log formatting** input group and set the **<NT>Log</NT> formatter** input’s value to `xcodebuild` or `xcbeautify`.

```yaml
- xcode-test:
    inputs:
      - log_formatter: xcbeautify
```

:::


## Rerunning failed tests (Not available in <NT>Xcode</NT> 13)

The **Should retry test on failure? (Not available in <NT>Xcode</NT> 13+)** input in the **Test repetition** section of the **<NT>Xcode Test for iOS</NT>** <NT>Step</NT> allows you to automatically rerun ALL your tests, not just the failed ones. If you set this input to `yes`, the <NT>Step</NT> will run `xcodebuild` one more time in the case of test failure.

From <NT>Xcode</NT> 13 and above, the feature is not available anymore. Use the **Test <NT>Repetition</NT>s Mode (Available in <NT>Xcode</NT> 13+)** input with the `retry_on_failure` option selected. This allows you to rerun only the failed test/s instead of running all your tests. You can find this test repetition feature with our **<NT>Xcode Test for iOS</NT>** <NT>Step</NT> from version 3.0.0 and above.


## Test Repetitions

[Xcode’s test repetition modes](https://developer.apple.com/videos/play/wwdc2021/10296/) are available with the [Xcode Test for iOS Step](https://www.bitrise.io/integrations/steps/xcode-test) on <NT><GlossTerm baseform="stack">stack</GlossTerm></NT>s running <NT>Xcode</NT> 13 and above on <NT>Bitrise</NT>. With test repetitions, you can run any type of tests multiple times in various ways such as retry on failure, until failure and until max repetitions.

The **<NT>Xcode Test for iOS</NT>** <NT>Step</NT>’s **Test <NT>Repetition</NT>s Mode (Available in <NT>Xcode</NT> 13+)** input offers the following options:

- `none`: The tests won’t repeat.
- `until_failure`: Repeats a test until the test fails or until the maximum repetition. The default number of test runs is three.
- `retry_on_failure`: Failed tests run until they succeed or until the repetition number you specify. The default number of test repetitions is three.
- `up_until_maximum_repetitions`: Reruns all tests until maximum test repetition is reached regardless of the test outcomes.

![test-repetition.png](/img/_paligo/uuid-d2ee7fd1-7743-7720-d114-5f38632cd2e1.png)

Please note that these options are only available from **<NT>Xcode Test for iOS</NT>** version 3.0.0 and above.

With the **Maximum Test <NT>Repetition</NT>s (Available in <NT>Xcode</NT> 13+)** input you can specify the maximum number of test repetitions. Please note that you have to add a greater number than one if the **Test <NT>Repetition</NT> Mode (Available in <NT>Xcode</NT> 13+)** input is set to other than `none`.

Enable the **Relaunch <NT>Tests</NT> for Each Repetition (Available in <NT>Xcode</NT> 13+)** input to launch tests in a completely new process for each repetition.
