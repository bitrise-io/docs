By default, builds run in simple chronological order: the build that was triggered first runs first. However, you can configure <NT>build priority</NT>, allowing you to fast-track critical builds to the top of the queue.

You can assign numerical priorities: a value between -100 and 100. The higher value, the higher the priority of the build. A build with a higher priority is executed sooner than a build with a lower priority. By default, all builds have a priority of 0.

You can set priorities for:

- <NT><GlossTerm baseform="pipeline">Pipeline</GlossTerm></NT>s and <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s. You can assign a priority to individual Worklows or even <NT>Pipelines</NT>. Assigning a priority to a <NT>Pipeline</NT> will assign all its constituent <NT>Workflows</NT> the same priority
- <NT><GlossTerm baseform="trigger">Trigger</GlossTerm></NT>s: Each trigger condition can be given a priority.
- Manual builds: When starting a build on the UI or via the <NT>Bitrise</NT> API, you can set a priority that determines its position in the queue.

:::note[Availability]

The <NT>build priority</NT> feature is only available for <NT><GlossTerm baseform="workspace">Workspace</GlossTerm></NT>s on Teams, Pro, and Enterprise plans. If you are not on these plans, but interested in the feature, [contact us](https://bitrise.io/contact)!

:::


## Configuring priority when starting a build manually

You can set a priority when [starting a build manually](/en/bitrise-ci/run-and-analyze-builds/starting-builds/starting-builds-manually). The priority only takes effect if other builds are queued when you start a build.

The priority set when manually starting the build overrides all other priorities.

1. Open <NT>Bitrise CI</NT> and select your project.
1. On the main project page, click **<NT>Start build</NT>**.
1. Select the **<NT>Advanced</NT>** option.

   ![manual-build-priority.png](/img/_paligo/uuid-9fdfc68c-9744-f7ae-fee6-b4cc984bdf3e.png)
1. Find the **<NT>Priority</NT>** field and set an integer value between -100 and 100. The <NT>default value</NT> is 0.


## Configuring priority for build triggers

You can set priority for build triggers on the GUI of the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> and in your <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file. Trigger priority overrides <NT>Pipeline</NT> and <NT>Workflow</NT> priority.

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Open the <NT>Workflow Editor</NT>.
1. Select the <NT>Workflow or Pipeline</NT> you need.
1. Go to the **<NT>Triggers</NT>** tab.
1. Create a new trigger or edit an existing one and find the **<NT>Priority</NT>** input.

   ![trigger-priority.png](/img/_paligo/uuid-02de5b3a-d9bd-5890-cfc3-52462a3c8ff2.png)
1. Set an integer value between -100 and 100: higher number means higher priority. The <NT>default value</NT> is 0.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Open your configuration YAML file.
1. Find a trigger in a <NT>Pipeline</NT> or <NT>Workflow</NT>.
1. Add the **priority** input with an integer value between -100 and 100: higher number means higher priority. The <NT>default value</NT> is 0.

   ```yaml
   workflows:
     primary:
       steps:
       - activate-ssh-key: {}
       - git-clone: {}
       - deploy-to-bitrise-io: {}
       triggers:
         push:
         - branch: main
           priority: 10
   ```

</TabItem>
</Tabs>


## Configuring priority for <NT>Pipelines</NT> and <NT>Workflows</NT>

You can configure priority by setting a value between -100 and 100 for the **<NT>Priority</NT>** property of any given <NT>Workflow</NT> or <NT>Pipeline</NT>. The <NT>default value</NT> is 0.

<NT>Pipeline</NT> priorities override <NT>Workflow</NT> priorities. If a pipeline is triggered and assigned a priority, all constituent workflows will be triggered with the same priority.

### Configuring <NT>Pipeline</NT> priority

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Open the <NT>Workflow Editor</NT>.
1. On the left, select **<NT>Pipelines</NT>**.
1. Select the <NT>Pipeline</NT> you need and click **<NT>Properties</NT>**.
1. Select the **<NT>Properties</NT>** tab.

   ![pipeline-priority.png](/img/_paligo/uuid-c92f6dbc-0dee-4ec9-3680-53b83a30f2a2.png)
1. In the **<NT>Priority</NT>** input field, provide an integer between -100 and 100. The <NT>default value</NT> is 0.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Open your Configuration YAML file.
1. Find the <NT>Pipeline</NT> you need and add a `priority` input with an integer value between -100 and 100.

   If you don't have a `priority` input set, the <NT>Pipeline</NT> will have a default priority of 0.

   ```yaml
   pipelines:
     best-pipeline:
       priority: 10
   ```

</TabItem>
</Tabs>

<Tabs>

1. Select the <NT>Workflow</NT> you need from the dropdown menu and select the **<NT>Properties</NT>** tab on the right.

1. Find the <NT>Workflow</NT> you need and add a `priority` input with an integer value between -100 and 100.

   If you don't have a `priority` input set, the <NT>Workflow</NT> will have a default priority of 0.


## Order of precedence for priority settings

You can set priority levels for:

- <NT>Pipelines</NT> and <NT>Workflows</NT>.
- Triggers.
- Manually triggered builds.

These priorities can conflict. For example, you have a <NT>Workflow</NT> called test. It has a priority of 10. You also have a trigger configured for this <NT>Workflow</NT>: it is triggered if code is pushed to the development branch. This trigger has a priority of 15.

In such a scenario, <NT>Bitrise</NT> determines the <NT>build priority</NT> by the order of precedence:

1. Manual overrides: Manually starting a build overrides all other priority settings.
1. Trigger-specific priority settings: A trigger's priority setting overrides the priority settings of the <NT>Pipelines</NT> or <NT>Workflows</NT> it triggers.
1. <NT>Pipeline</NT>-level priority settings: The <NT>Pipeline</NT>'s priority overrides the priority settings of its constituents <NT>Workflows</NT> which are all given the same priority as the <NT>Pipeline</NT>.
1. <NT>Workflow</NT>-level settings: If no other configuration element is given a priority setting, the <NT>Workflow</NT> priority determines overall <NT>build priority</NT>.

Here’s an example snippet that shows how to set priorities at various levels:

```yaml
workflows:
  A:
    steps: []
  B:
    steps: []
    priority: 4
    triggers:
      push:
      - branch: release
        priority: 2
      - branch: development

pipelines:
  P:
    priority: 3
    workflows:
      A: {}
      B: {}
```

In this example:

- If <NT>Workflow</NT> A is run as a standalone build, it has a priority of 0 (the default priority value).
- If <NT>Workflow</NT> B is run as a standalone build (for example, if code is pushed to the `development` branch), it has a priority of 4.
- If code is pushed to the `release` branch, <NT>Workflow</NT> B is triggered with a priority of 2.
- If <NT>Pipeline</NT> P is triggered, both A and B <NT>Workflows</NT> will run with priority 3.
