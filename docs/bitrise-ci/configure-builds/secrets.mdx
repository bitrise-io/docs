<NT><GlossTerm baseform="secret">Secret</GlossTerm></NT>s are a specific type of Environment Variable: they hide their information in an encrypted format and their value is not exposed in the build logs. They aren't shown in the `bitrise.yml` configuration either. You can store confidential information, such as passwords or API keys as <NT>Secrets</NT>.

Just like <NT>Environment Variables</NT>, <NT>Secrets</NT> can be used as the value of a <NT><GlossTerm baseform="step">Step</GlossTerm></NT> input: [Using an Env Var as a Step input](/en/bitrise-ci/configure-builds/environment-variables#using-an-env-var-in-a-step-input)


## Setting a Secret

:::note[Every Env Var value is a string]

The value of an Environment Variable or Secret can only be a string. Even if you set a number (for example, 7), it will be passed on as a string.

:::

You can create a Secret either on the **<NT>Secrets</NT>** page in the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> or when modifying <NT>Step inputs</NT>.

<Tabs>
<TabItem value="secrets-page" label="Secrets page" default>

1. <Partial_OpeningTheWorkflowEditor />
1. On the left, select **<NT>Secrets</NT>**.
1. Under **<NT>Project level Secrets</NT>**, click the **<NT>Add new</NT>** button.

   ![secrets-screenshot.png](/img/_paligo/uuid-b269a597-65ef-34a5-1828-ad1ab954437d.png)
1. Add a key and a value.

   :::important[Using the $ character]

   You can use the `$` character in the value of an Env Var or Secret (for example, in a password) but in that case always leave the **<NT>Replace variables in inputs</NT>** option unchecked. If you replace the variable's key with its value in inputs, the <NT>Bitrise CLI</NT> will treat the value as another Env Var because of the `$` character.

   :::
1. Optionally, you can [replace the key of your Secret with its value in inputs](/en/bitrise-ci/configure-builds/environment-variables#replacing-variables-in-inputs): check the **<NT>Replace variables in inputs</NT>** option.
1. Optionally, you can expose your Secret to pull requests by selecting the option.

   In most cases, we strongly recommend [not to expose Secrets](/en/bitrise-ci/configure-builds/secrets#exposing-a-secret-to-pull-requests).
1. Click **<NT>Save</NT>**.

</TabItem>
<TabItem value="step-inputs" label="Step inputs">

1. <Partial_OpeningTheWorkflowEditor />
1. <NT>Select a Workflow</NT> and find the <NT>Step</NT> you want to configure.
1. Find the sensitive input you wish to modify.
1. Click the **$** sign and then click **<NT>Create</NT>** in the dialog.
1. In the **<NT>Create secret</NT>** dialog, specify a key and a value for your new Secret.

   ![create-variable.png](/img/_paligo/uuid-2a0854ff-b62a-6215-5c2e-09cf6b244786.png)
1. Optionally, you can [replace the key of your Secret with its value in inputs](/en/bitrise-ci/configure-builds/environment-variables#replacing-variables-in-inputs) by checking the option.
1. Click **<NT>Create</NT>**.

</TabItem>

## Editing an existing Secret

Once you’ve added a new secret Env Var in the **<NT>Secrets</NT>** tab, you come back to it any time, modify its content or make it protected from curious eyes!

1. <Partial_OpeningTheWorkflowEditor />
1. On the left, select **<NT>Secrets</NT>**.
1. Click **<NT>Edit</NT>** next to the value of your Secret.
1. Modify its content as needed.
1. Click **<NT>Save</NT>**.


## Protecting the value of a Secret

Normally, you can **show** and **hide** the value of a Secret by clicking the **eye** icon. This feature is useful if you have a long list of secret env vars in **<NT>Secrets</NT>** and you wish to check the value of only one Secret while leaving the other values hidden. If a value is hidden, it’s represented with the **crossed out eye** icon.

However, you can hide the value of a Secret permanently by making it protected. If you do so, no one, including yourself, will be able to change or even view the value again.

:::warning[A Secret's protection can't be undone]

Making a Secret protected is irreversible. If you ever need to change the value, you will have to delete the Secret and create a new one.

:::

1. <Partial_OpeningTheWorkflowEditor />
1. On the left, select **<NT>Secrets</NT>**.
1. Click **<NT>Edit</NT>** next to the value of your Secret.

   ![edit-button-secrets.png](/img/_paligo/uuid-2de8b6b8-3fc0-5da9-9f45-1b22546e1bb2.png)
1. Check **<NT>Protected</NT>**.
1. The dialog will warn that the process is irreversible. Click **<NT>Save and protect</NT>**.

That's it. You can no longer view the value of the Secret.


## Exposing a Secret to pull requests

By default, pull requests do not have access to the values of <NT>Secrets</NT>. This means that if a pull request opened from a fork of a repository <NT><GlossTerm baseform="trigger">trigger</GlossTerm></NT>s a build, that build can't use the app's <NT>Secrets</NT>.

If you need to grant pull request builds access to a particular Secret, you can expose that Secret to pull requests. However, even when exposed, its value won't be visible in the logs: it will be displayed as [REDACTED].

:::important[Protected <NT>Secrets</NT> can't be exposed]

Once a Secret has been [made protected](/en/bitrise-ci/configure-builds/secrets#protecting-the-value-of-a-secret), you can't expose it to pull request builds.

:::

:::important[Public apps]

If your app is a public app, you can't expose the app's <NT>Secrets</NT> to pull requests builds.

:::

1. <Partial_OpeningTheWorkflowEditor />
1. On the left, select **<NT>Secrets</NT>**.
1. Click **<NT>Edit</NT>** next to the value of your Secret.
1. Toggle the **<NT>Expose for Pull Requests</NT>?** toggle.

Once the Secret is exposed, pull request builds can access it.


## Managing <NT>Secrets</NT> across multiple apps

By default, all <NT>Secrets</NT> are handled on the app level. You can reuse Secret keys across multiple <NT>Bitrise</NT> apps, even if their corresponding values are different for each app.

However, it is possible to set up a Secret that holds the same value for all your apps, and manage that Secret from one location. For example, if all your apps need access to the same API, it makes sense to store the Secret containing the API key in a central location. If the API key ever changes, you only need to change it in that single location and the change applies to all your <NT>Bitrise</NT> apps.

There are two ways to achieve this:

- [Managing your Secrets from a Workspace](#managing-secrets-on-a-workspace-level). This is only available on Enterprise plans.
- [Managing secrets from a central location](#managing-secrets-from-a-central-vault-or-database) such as a vault or database and pulling it with a <NT>Step</NT>.

### Managing <NT>Secrets</NT> on a <NT>Workspace</NT> level

On Enterprise plans, you can store <NT>Secrets</NT> on a <NT>Workspace</NT> level. This means that the same Secret is accessible to all apps owned by that <NT>Workspace</NT>. This means increased security: only <NT>Workspace</NT> Owners and Managers can access these <NT>Secrets</NT>.

:::note[Downgrading]

If you downgrade from an Enterprise plan, <NT>Workspace</NT> <NT>Secrets</NT> will be converted to app level <NT>Secrets</NT>.

:::

When passing <NT>Secrets</NT> to your build, app level <NT>Secrets</NT> take precedence over <NT>Workspace</NT> level <NT>Secrets</NT> in the [availability order](/en/bitrise-ci/configure-builds/environment-variables#availability-order-of-environment-variables). This also means that if you have two <NT>Secrets</NT> with the same key, the app level Secret will be used during the build.

To add a new secret:

1. On the <NT>Bitrise</NT> main page, select **<NT>Settings</NT>** on the navigation menu on the left. It takes you to the **<NT>Workspace settings</NT>** page.
1. On the **<NT>Workspace settings</NT>** page, select **<NT>Shared resources</NT>**.

   ![shared-res.png](/img/_paligo/uuid-8b262205-594a-d6ae-9985-3c47a3df5175.png)
1. Click the **<NT>Add new</NT>** button.
1. Add a key and a value.

   ![ws-secrets.png](/img/_paligo/uuid-0d1616e8-82d6-9de7-f192-1aab7058880b.png)
1. Configure the usage details of the Secret:

   - **<NT>Replace variables in inputs</NT>**: This passes the value of the Secret as a string to the build. Use this for embedded <NT>Secrets</NT> or <NT>Environment Variables</NT>: when the value of your Secret is the key of another Secret or Environment Variable. For most use cases, you don't need this checked.
   - **<NT>Expose for pull requests</NT>**: Exposing a Secret to a PR means that the Secret is available to use in pull request builds. Depending on your repository's privacy settings, this is a potential security risk!
   - **<NT>Protected</NT>**: The value cannot be changed or viewed again. This setting is irreversible once saved.
1. When you're ready, click **<NT>Done</NT>**.

### Managing <NT>Secrets</NT> from a central vault or database

Setting up <NT>Secrets</NT> in a central location requires two things:

- A central vault or database - such as HashiCorp or Doppler - to store the <NT>Secrets</NT>. It must be accessible via a CLI.
- A **Script** <NT>Step</NT> to access the central vault/database, pull the Secret and set it to sensitive on <NT>Bitrise</NT>.

To create a new Secret and store it in a central location during a build:

1. Add the <NT>Secrets</NT> as a key-value pair to your vault or database where you want to store them.
1. Add a **Script** <NT>Step</NT> to ALL <NT>Workflows</NT> where you want to use the <NT>Secrets</NT>.
1. Add the necessary commands to access your vault and pull the <NT>Secrets</NT>. The exact commands depend on the service you’re using.
1. Use the `envman` tool to mark the <NT>Secrets</NT> as sensitive. The envman tool has the following syntax:

   `envman add --key KEY --value value --sensitive` .
1. Make sure the <NT>Step</NT> doesn’t display the value of the Secret in the build log. To do so, remove `set -x` from the <NT>Step</NT>’s `content`.

:::warning[Secret redaction]

Please note that if you have Secret redaction turned off, your <NT>Secrets</NT> will not be redacted and thus their value can still be visible in logs.

:::

**Storing your <NT>Secrets</NT> in a HashiCorp Vault**

Let’s say you have a [HashiCorp Vault](https://www.vaultproject.io/) instance called `secret/hello`. You have two <NT>Secrets</NT> in this vault instance: `foo` with the value `world` and `foo2` with the value `world2`. To use these <NT>Secrets</NT> in a <NT>Bitrise</NT> build, you need to:

- Export them from the Vault instance.
- Iterate over them and mark both of them as sensitive.

You can use this Script to achieve both:

```bash
# Exporting the Secrets
vault kv get --format=json secret/hello | jq -r '.data.data | to_entries[] | [.key, .value] | @tsv' | 
# Iterating over the Secrets and marking them as sensitive
while IFS=$'\t' read -r key value; do
    envman add --key "$key" --value "$value" --sensitive
done
```


## Redacting <NT>Secrets</NT>

The <NT>Bitrise CLI</NT> automatically redacts your Secret Env Vars and prints `[REDACTED]` with newlines after the key so that the Secret Env Vars are NOT VISIBLE in the build log. This applies to both private and public apps.

You can turn this off at any time though if you need to have your <NT>Secrets</NT> visible in logs.

:::warning

This is a potential security risk. We strongly recommend not to turn off Secret redaction.

:::

1. <Partial_OpeningTheWorkflowEditor />
1. On the left, select **<NT>Secrets</NT>**.
1. Click **<NT>Add new</NT>** to register the `BITRISE_SECRET_FILTERING` secret Environment Variable with false value.
1. Click **<NT>Save changes</NT>** in the top right corner.


## <NT>Secrets</NT> in self-hosted environments

You can run <NT>Bitrise</NT> builds on <NT>self-hosted infrastructure</NT>: hardware or virtualized environments that you control. If your security policy doesn't allow you to use <NT>Bitrise</NT> <NT>Secrets</NT> hosted on bitrise.io, this guide offers a different way of accessing <NT>Secrets</NT> and using them in your builds.

In a self-hosted build, your host machine is behind your virtual private cloud. As such, you can access the secret service of your choice without exposing the <NT>Secrets</NT> to the <NT>Bitrise</NT> control plane and the website. Get the Secret from your service and then add it to the <NT>Bitrise</NT> build using `envman`:

1. Optionally, create a placeholder Secret on <NT>Bitrise</NT> with any value: [Setting a Secret](/en/bitrise-ci/configure-builds/secrets#setting-a-secret).

   This is necessary only if you wish to use the Secret in a sensitive <NT>Step</NT> input on the GUI of the <NT>Workflow Editor</NT>. Sensitive inputs only accept <NT>Secrets</NT> and in the <NT>Workflow Editor</NT> you can only add existing <NT>Secrets</NT> as the value of a sensitive input.

   If you edit your <NT>Bitrise</NT> configuration in YAML, you don't need a placeholder as you can set any value to your input.
1. Use a **Script** <NT>Step</NT> to get your Secret from the service you use.
1. In the script, turn off debugging by setting `set +x`:

   This is very important: in debug mode, the value of the Secret might be visible in the build log!

   ```bash
   #!/usr/bin/env bash
   # fail if any commands fails
   set -e
   # make pipelines' return status equal the last command to exit with a non-zero status, or zero if all commands exit successfully
   set -o pipefail
   # debug log
   set +x
   ```
1. Fetch the Secret value from the service you use and then add it to `envman`.

   ```bash
   set +x
   GIT_PASSWORD_SECRET=TODO Your call to fetch the secret value
   envman add --key GIT_HTTP_PASSWORD --value $GIT_PASSWORD_SECRET
   ```

   The actual commands depend on the service you use. For example, if you use <NT>Bitrise</NT> on <NT>AWS</NT> and your <NT>EC2</NT> machine can access the <NT>AWS Secrets Manager</NT>, you could get the secret with a command similar to this:

   ```bash
   set +x
   USERNAME=aws secretsmanager get-secret-value --secret-id BitriseDemoGitUsername --region eu-central-1 | jq  -r '.SecretString'
   envman add --key GIT_HTTP_USERNAME --value $USERNAME
   PASS=aws secretsmanager get-secret-value --secret-id BitriseDemoGitPassword --region eu-central-1 | jq  -r '.SecretString'
   envman add --key GIT_HTTP_PASSWORD --value $PASS
   ```
