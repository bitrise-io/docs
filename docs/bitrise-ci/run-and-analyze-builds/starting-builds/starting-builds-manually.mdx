Starting a build manually on <NT>Bitrise</NT> takes literally two clicks - if you leave everything on the default settings. If you do not wish to do that, or at least want to be sure what the default settings are, read on!

To start a build using the basic configuration options:

1. On the **<NT>Bitrise CI</NT>** page of <NT>your app</NT>, click the **<NT>Start build</NT>** button.

## Running <NT>Workflows</NT> from the <NT>Workflow Editor</NT>

You can manually start a build by running a single <NT><GlossTerm baseform="Workflow">Workflow</GlossTerm></NT> from the <NT><GlossTerm baseform="Workflow Editor">Workflow Editor</GlossTerm></NT>.

To do so:

1. Open your project on <NT>Bitrise</NT> and click the **<NT>Workflows</NT>** button to open the <NT>Workflow Editor</NT>.
1. On the left, select **<NT>Workflows</NT>**, and use the dropdown menu to select the <NT>Workflow</NT> you would like to run.

   ![Workflow dropdown menu in the Workflow Editor](/img/run-and-analyze-builds/2026-07-14-workflow-editor-workflow-dropdown.png)
1. Click the **<NT>Run</NT> <NT>Workflow</NT>** button (the one with the **play** icon).

   ![Run Workflow button next to the Workflow dropdown](/img/run-and-analyze-builds/2026-07-14-workflow-editor-run-button.png)
1. Specify the branch that you want to build in the **<NT>Branch</NT>** field.

   ![Run dialog with a Branch field and Start build button](/img/run-and-analyze-builds/2026-07-14-run-workflow-branch-dialog.png)
1. Click **<NT>Start build</NT>**.

And that's it! As soon as you click the **<NT>Start build</NT>** button, your build will be kicked off, and the build's page will open in a new tab.


## Advanced configuration options for starting/scheduling builds

If you choose **<NT>Advanced</NT>** in the **<NT>Build</NT> configuration** window, <NT>you have</NT> all the options available with **<NT>Basic</NT>**, and a few more:

You can select a source type in the **<NT>Source</NT>** dropdown menu. In addition to the default **<NT>Branch</NT>** option, you can also select:

- **<NT>Git Tag</NT>**: Builds a particular branch or commit that is tagged with the Tag you enter. If you set this, you can't set a commit hash.

  :::important[<NT>Git Tag</NT>s and commit hashes are conflicting options]

  You cannot specify both a <NT>Git Tag</NT> and a commit hash. When you set one, the other one will disappear. This ensures you cannot enter conflicting input values.

  :::
- **<NT>Commit Hash</NT>**: Copy a specific commit’s hash here to build that particular commit. This option can even send a build status update to your git hosting service. If you set this, you can't set a <NT>Git Tag</NT>.

  :::note[Commit hash takes precedence over the <NT>Branch</NT> option]

  If the **<NT>Branch</NT>** option specifies a branch where the commit hash you chose doesn't exist, <NT>Bitrise</NT> will find the branch that does have the given commit and run a build with that branch.

  :::

![start-build-advanced.png](/img/_paligo/uuid-67f3a4f0-9315-bec3-0ea7-cca80d77b103.png)

In addition to different sources, you can also add:

- **<NT>Priority</NT>**: Determines the priority of your build in the <NT>build queue</NT>. The higher the number, the sooner your build will run. The priority must be an integer between -100 and 100. The <NT>default value</NT> is 0. For more information, check out [Build priority](/en/bitrise-ci/configure-builds/configuring-build-settings/build-priority).
- **<NT>Custom Environment Variables</NT>**: Create a custom Environment Variable that is used in the build. See [Setting a custom Env Var when starting a build](/en/bitrise-ci/configure-builds/environment-variables#setting-a-custom-env-var-when-starting-a-build).
- **<NT>Generated cURL command</NT>**: Based on the options you set in the Build configuration window, we provide an automatically <NT>generated cURL command</NT>. This can be copied and pasted, and you can run it on any platform that has cURL installed.

## Rebuilding failed builds

Another way of starting a build manually is to rebuild a failed build. You can:

- Rebuild a <NT>Workflow</NT> build: [Rebuilding a failed Workflow build](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/rebuilding-a-failed-build#rerunning-a-failed-workflow-build).
- Rebuild a <NT><GlossTerm baseform="pipeline">Pipeline</GlossTerm></NT>, either the entire <NT>Pipeline</NT> or only unsuccessful <NT>Workflows</NT>: [Rebuilding a failed Pipeline](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/rebuilding-a-failed-build#rebuilding-a-failed-pipeline).
- <NT>Rebuild with remote access</NT>: [Remote access](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/remote-access).
