:::important[<NT>GitHub</NT> App integration]

If you use the <NT>Bitrise</NT> <NT>GitHub</NT> App to connect a <NT>Bitrise</NT> <NT><GlossTerm baseform="workspace">Workspace</GlossTerm></NT> to a <NT>GitHub</NT> account or organization, you don't need to do anything described in this section: <NT>GitHub</NT> Checks will automatically work for you to provide status updates.

You can, however, continue to use the OAuth connection for your apps and use <NT>GitHub</NT> Checks with our <NT>GitHub</NT> App.

:::

The <NT>Bitrise</NT> <NT>GitHub</NT> App provides, among other things, an extended version of the classic build status checks that <NT>Bitrise</NT> sends back to <NT>GitHub</NT>. This extended version includes a <NT>Bitrise</NT> build summary and a check status. You can view the checks if you click the **<NT>Details</NT>** link of a pull request on <NT>GitHub</NT>. It unfolds the detailed build summary and build status our <NT>GitHub</NT> app attaches to your pull request on the **<NT>Check</NT>s** tab of <NT>GitHub</NT>.

![Bitrise_Checks_on_GitHub_Checks.png](/img/_paligo/uuid-d3db0692-cdd1-86cb-615a-955fd97587e9.png)

![Bitrise_Checks_on_GitHub_Checks.jpg](/img/_paligo/uuid-0268c55e-cc52-4a65-4f01-f1db5f281a4e.jpg)

There can be three different check statuses:

- Success.
- Failed.
- Action required (in the case of manual pull request approval).

To start using the <NT>Bitrise</NT> app to send status checks, you need to perform two separate actions:

1. [Install the Bitrise app](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/bitrise-checks-on-github#installing-the-bitrise-github-app-for-github-checks): This is only necessary for <NT>GitHub</NT>.com users. <NT>GitHub Enterprise Server</NT> users who [already set up the integration](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise) can proceed to enable the status checks with the app.
1. [Enable the Bitrise app](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/bitrise-checks-on-github#enabling-github-checks-on-bitrise): Turn on the feature on the <NT>Bitrise website</NT> and run a first build to be able to select the <NT>Bitrise</NT> app in your branch protection rules.


## Installing the <NT>Bitrise</NT> <NT>GitHub</NT> app for <NT>GitHub</NT> Checks

To use <NT>Bitrise</NT> Checks as a <NT>GitHub</NT>.com user, you need to install it as a <NT>GitHub</NT> app.

If the <NT>workspace</NT> that owns the <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT> has the <NT>Bitrise</NT> <NT>GitHub</NT> App connected and granted access to the <NT>Bitrise</NT> <NT>project</NT>'s repository, you can skip the procedure described here and start using <NT>GitHub</NT> Checks.

:::important[<NT>GitHub</NT>.com users only]

If you are a <NT>GitHub Enterprise Server</NT> user, you don't need to do this procedure. To enable <NT>GitHub</NT> Checks, you need to:

1. [Install the GitHub Enterprise Server integration](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise).
1. [Enable GitHub Checks on Bitrise](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/bitrise-checks-on-github#enabling-github-checks-on-bitrise).

:::

1. <Partial_GettingToTheProjectSettingsPage />
1. On the left, select **<NT>Integrations</NT>** from the menu options.
1. Click the link under the toggle. Don’t toggle the switch just yet, since you first need to install the <NT>Bitrise</NT> app.

   This link will take you to <NT>GitHub</NT>'s **<NT>Bitrise</NT>** installation page.

   ![github_checks_install.png](/img/_paligo/uuid-f07c26d9-ec9e-bcab-3f8e-184408ae7fc2.png)
1. Select the user or <NT>GitHub</NT> Organization you want to add the <NT>Bitrise</NT> app to.
1. <NT>Select an option</NT> to install the <NT>GitHub</NT> app to:

   - **<NT>All repositories</NT>**: Applies to all current and future repositories.
   - **<NT>Only select repositories</NT>**: Select the repositories that you need.

   With this, you authorize <NT>Bitrise</NT> to act on your behalf; for example, to check which repos you have access to, and use <NT>GitHub</NT> Checks to display check results.
1. Click **<NT>Install</NT> and authorize**.
1. In the <NT>GitHub</NT> prompt, provide your <NT>GitHub</NT> password.

If all goes well, you land on the **<NT>Bitrise</NT>** page of <NT>GitHub</NT>. You should see a blue note at the top-left corner of the page that <NT>Bitrise</NT> has been successfully installed on your <NT>GitHub</NT> account.


## Enabling <NT>GitHub</NT> Checks on <NT>Bitrise</NT>

Once [the Bitrise app is installed](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/bitrise-checks-on-github#installing-the-bitrise-github-app-for-github-checks) on <NT>GitHub</NT>, you need to enable <NT>GitHub</NT> Checks on the <NT>project settings</NT> page on [Bitrise](https://app.bitrise.io/dashboard/builds).

:::important[Enabling <NT>GitHub</NT> Checks is limited]

Please note that only <NT>workspace</NT> owners and project admins can enable this toggle on the **<NT>Project settings</NT>** page of the project.

:::

1. <Partial_GettingToTheProjectSettingsPage />
1. On the left, select **Repository**.
1. Scroll down to the **<NT>Build reporting</NT>** section and toggle on the **<NT>GitHub Checks</NT>** switch.

   :::important[Can't toggle the switch?]

   If you can't toggle the switch, check out [Can't enable GitHub Checks](#cant-enable-github-checks).

   :::
1. <NT><GlossTerm baseform="trigger">Trigger</GlossTerm></NT> a build. You can do so either automatically or manually. If you trigger a build manually, provide a specific commit hash to build.

   This build is necessary to perform the first <NT>Bitrise</NT> check. Once there is a finished check, you can select <NT>Bitrise</NT> Checks in your branch protection rules.

And you’re done! Now any pull request you open to your project on <NT>GitHub</NT> will be validated with <NT>Bitrise</NT> Checks and a build will get automatically started on <NT>Bitrise</NT> (if the [pull request trigger](/en/bitrise-ci/run-and-analyze-builds/build-triggers/yaml-syntax-for-build-triggers) is properly set on <NT>Bitrise</NT>).

### Can't enable <NT>GitHub</NT> Checks

There are two main reasons why you might not be able to toggle the switch to enable <NT>GitHub</NT> Checks:

- You cannot toggle the switch unless you install the <NT>Bitrise</NT> <NT>GitHub</NT> app FIRST. Once it’s done, you can go ahead and toggle the switch to the right to enable it.
- If you have renamed or moved the <NT>GitHub</NT> repository of an app which has already been added to <NT>Bitrise</NT> and you have installed <NT>Bitrise</NT> Checks on it. The repository’s <NT>GitHub</NT> URL has obviously changed. **<NT>GitHub Checks</NT>** switch can only work if the URL on <NT>Github</NT> and on <NT>Bitrise</NT> fully match (no redirect URL is allowed). So in this case, you have to [manually update the **Repository URL** of your app](/en/bitrise-platform/projects/configuring-the-repository-url-and-the-default-branch).


## Disabling <NT>GitHub</NT> Checks

To disable <NT>GitHub</NT> Checks for your project:

if you toggle the **<NT>ENABLE</NT> <NT>GITHUB CHECKS</NT>** switch to the left on the **<NT>Settings</NT>** page of the project.

1. <Partial_GettingToTheProjectSettingsPage />
1. Select **Repository** on the left.
1. Under **<NT>Build reporting</NT>**, disable the **<NT>GitHub Checks</NT>** toggle.


## <NT>GitHub</NT> Checks test summary

If your project’s repository contains any UI or unit tests, you can run them by using the appropriate testing <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s in your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>. With the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT>, your <NT>test results</NT> will be displayed in our **[test reports](/en/bitrise-ci/testing/deploying-and-viewing-test-results)** page and you can also see them on <NT>GitHub</NT> Checks, too.

For example, if you have an <NT>Android</NT> app, you can use the **Virtual Unit <NT>Testing</NT> for <NT>Android</NT>** <NT>Step</NT> to be able to see all your test results on <NT>GitHub</NT> Checks.

![test_reports.png](/img/_paligo/uuid-6fb2370d-7bd0-a531-147b-deeb7a89c996.png)

### <NT>Step</NT> by <NT>Step</NT> error report

If a CI build runs, <NT>GitHub</NT> Checks' test summary displays the <NT>Steps</NT>' successful and failed tests in separate tabs on the left sidebar. If you click on a <NT>Step</NT>, a more detailed report summary comes up where a number of tests are listed with actionable error report. The build summary is available on the <NT>Bitrise</NT> tab.

Please note that <NT>GitHub</NT> Checks' test summary can only print detailed information of the following <NT>Steps</NT>:

- **<NT>Xcode Test for iOS</NT>**
- **<NT>Android Unit Test</NT>**
- **<NT>iOS Device Testing</NT>**
- **<NT>Virtual Device Testing for Android</NT>**
- **<NT>Flutter Test</NT>**

You can export test results generated by any other <NT>Step</NT> to the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> in two ways:

- We strongly recommend using our **Export <NT>test results</NT> to the Test Reports** <NT>Step</NT>: this <NT>Step</NT> locates the test results based on your inputs.
- You can configure your own custom **Script** <NT>Step</NT> that creates and exports the necessary files.

### Checking the test results of a failed <NT>Step</NT>

If your build has some failed tests, here is how to access them in <NT>GitHub</NT> Checks:

1. Click the **<NT>Bitrise</NT> <NT>Check</NT>s** dropdown.
1. Click on a <NT>Step</NT> and view the error report of the specific <NT>Step</NT>.

You can click on other <NT>Steps</NT> from here or the build summary. The **View more details on <NT>Bitrise</NT> Checks** link takes you to the build page on <NT>Bitrise</NT>.

### Re-running a build using <NT>GitHub</NT> checks

:::important[Role requirement]

To re-run a build using <NT>GitHub</NT> Checks, [your GitHub account must connected to Bitrise](/en/bitrise-platform/repository-access/repository-access-with-oauth#connecting-a-git-provider-with-oauth-from-the-account-settings-page), and you need to have at least Developer role in your app's team. If these conditions are not met, re-run checks will fail without an error.

For a complete list of user roles and role cheatsheets, check [User roles on app teams](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).

:::

If you would like to re-run a build using <NT>GitHub</NT> Checks:

1. Open the pull request on <NT>GitHub</NT> where you would like to re-run checks.
1. Open the **<NT>Check</NT>s** tab.
1. Click **Re-run all checks**.
