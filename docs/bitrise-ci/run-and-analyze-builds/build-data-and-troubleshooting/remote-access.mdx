Remote access allows users to connect to a running <NT>build machine</NT> for easier debugging. A failed build can be rebuilt with remote access enabled to make troubleshooting a lot easier - for example, if the build logs don’t provide enough information about the error.

You can rebuild both standalone builds and [pipeline builds](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/pipelines-with-stages/configuring-a-pipeline-with-stages) with remote access.

:::important[Authorization]

Users who have the **<NT>Testers</NT>/QA** roles on the app CANNOT use remote access.

:::

There are three ways to use remote access:

- [Visual Studio Code](#remote-access-with-visual-studio-code): connect the <NT>Bitrise</NT> build to your local VSCode app as a remote repository.
- **SSH**: classic remote shell connection from your terminal.
- **<NT>Screen share</NT>**: remote screen sharing with our VNC client of choice. Works with macOS builds only.

With either method, you can access the <NT>build machine</NT> remotely during the build and for 10 minutes after the build is finished. If you would like to extend the availability of remote access, you can do so by adding a simple custom [Script](https://github.com/bitrise-io/steps-script) <NT><GlossTerm baseform="step">Step</GlossTerm></NT> after the failed <NT>Step</NT>. Check out [Extending the availability of remote access](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/remote-access#extending-the-availability-of-remote-access) for more information.

:::important[Build time]

When using remote access, 10 minutes will be added to your overall build time.

:::


## Remote access with Visual Studio Code

This remote access mechanism allows your local Visual Studio Code app to connect to the <NT>Bitrise</NT> <NT>build machine</NT> as a [remote repository](https://code.visualstudio.com/docs/remote/ssh).

You can browse and edit files in real time, use the built-in terminal to execute commands, and even install a subset of <NT>VSCode extensions</NT>.

1. Open the project on <NT>Bitrise</NT>.
1. On the main page of the project, find the build you want to re-run with remote access and click it.
1. On the **<NT>Rebuild</NT>** button, click the downward arrow to open the dropdown menu.
1. Select **<NT>Rebuild with remote access</NT>**. This starts a new build that you can access remotely.

   ![Rebuild with remote access option in the Rebuild dropdown menu](/img/run-and-analyze-builds/2026-07-14-rebuild-with-remote-access-button.png)
1. On the build's page, select the **VSCode** tab.
1. Follow the on-screen instructions to install the <NT>Bitrise CLI</NT>, its remote access extension and the connection command.

   ![VSCode tab with Bitrise CLI, remote access plugin, and connection command](/img/run-and-analyze-builds/2026-07-14-remote-access-vscode-tab.png)


## Remote access with SSH

To access a <NT>Bitrise</NT> <NT>build machine</NT> with SSH, you will need a command line interface and the correct command and password:

1. Open the project on <NT>Bitrise</NT>.
1. On the main page of the project, find the build you want to re-run with remote access and click it.
1. On the **<NT>Rebuild</NT>** button, click the downward arrow to open the dropdown menu.
1. Select **<NT>Rebuild with remote access</NT>**. This starts a new build that you can access remotely.

   ![Rebuild with remote access option in the Rebuild dropdown menu](/img/run-and-analyze-builds/2026-07-14-rebuild-with-remote-access-button.png)
1. On the build's page, select the **SSH** tab.
1. Find the **<NT>Command</NT>** and run it in a command line interface.

   ![SSH tab with the Command and Password fields](/img/run-and-analyze-builds/2026-07-14-remote-access-ssh-tab.png)

And done! You should be able to access the virtual machine where your build is running.


## Remote access with <NT>screen share</NT>

To access a <NT>Bitrise</NT> <NT>build machine</NT> with a <NT>screen share</NT> app, you will need the correct username, password, and URL, as well as a VNC <NT>screen share</NT> app:

:::note[macOS only]

Remote access with <NT>screen share</NT> is only available for <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s that run on a macOS-based <NT><GlossTerm baseform="stack">stack</GlossTerm></NT>. If the <NT>Workflow</NT>'s stack (or in absence of <NT>Workflow</NT>-specific stacks, the app's default stack) is a <NT>Linux</NT>-based one, you won't see the <NT>screen share</NT> instructions at all.

:::

1. Open the project on <NT>Bitrise</NT>.
1. On the main page of the project, find the build you want to re-run with remote access and click it.
1. On the **<NT>Rebuild</NT>** button, click the downward arrow to open the dropdown menu.
1. Select **<NT>Rebuild with remote access</NT>**. This starts a new build that you can access remotely.

   ![Rebuild with remote access option in the Rebuild dropdown menu](/img/run-and-analyze-builds/2026-07-14-rebuild-with-remote-access-button.png)
1. On the build's page, select the **<NT>Screen share</NT>** tab.
1. Find the required information:

   - URL
   - Username
   - Password

   ![Screen share tab with the URL, Username, and Password fields](/img/run-and-analyze-builds/2026-07-14-remote-access-screen-share-tab.png)
1. Open a VNC <NT>screen share</NT> application.

   The simplest option is using the default **Screen Sharing** application on macOS.
1. Fill out the required fields with the information from under the **<NT>Screen share</NT>** option.

And done! You should now be able to access the virtual machine where your build is running.


## Finding your files on the VM

Once you successfully logged in to the <NT>Bitrise</NT> virtual machine that ran your build, you can dive into the files themselves to see what’s happening in real time. After <NT>Bitrise</NT> finished cloning your app on to the VM, you can always access it at the following location:

```text
/Users/vagrant/git
```


## Extending the availability of remote access

Remote access is available while the build is running and for 10 minutes after the build is finished. If this is not enough, there’s a simple workaround to make sure remote access is available for a longer time.

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Add a **Script** <NT>Step</NT> after the <NT>Step</NT> that causes the build to fail.
1. Toggle the **<NT>Run even if previous Step(s) failed</NT>** option on to ensure the **Script** <NT>Step</NT> always runs.
1. Add a command to let the build “sleep” for a time specified in seconds:

   `sleep 5400`

   This example lets the build run for 90 minutes. It should be no more than your build time limit, of course.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

Add a `script` <NT>Step</NT> after the <NT>Step</NT> that causes the build to fail. Set `is_always_run` to `true` so the <NT>Step</NT> always runs, and add a command to let the build “sleep” for a time specified in seconds:

```yaml
workflows:
  primary:
    steps:
    - script:
        is_always_run: true
        inputs:
        - content: sleep 5400
```

This example lets the build run for 90 minutes. It should be no more than your build time limit, of course.

</TabItem>
</Tabs>

That’s it. While the build is still running, you can look around on the virtual machine for the possible issues that caused it to fail.
