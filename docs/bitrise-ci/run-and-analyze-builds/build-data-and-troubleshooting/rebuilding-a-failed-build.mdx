If a build fails, you can rebuild any time to quickly recover from any errors. You can rebuild either single <NT><GlossTerm baseform="Workflow">Workflow</GlossTerm></NT>s or an entire <NT><GlossTerm baseform="Pipeline">Pipeline</GlossTerm></NT>. A rebuild always uses the current <NT><GlossTerm baseform="bitrise.yml">configuration YAML</GlossTerm></NT> file: if a build fails and then you modify your configuration and rebuild, the rebuild will run with the new configuration.


## Rerunning a failed <NT>Workflow</NT> build

You can rebuild a failed <NT>Workflow</NT> at any time. Rebuilding means immediately launching a new build with the current <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file.

1. Open <NT>Bitrise CI</NT> and find your project's build page.
1. On the top right corner, click **<NT>Rebuild</NT>**.

   :::tip[Remote access]

   You can also rebuild with remote access: click the downward arrow to open the dropdown menu and click **<NT>Rebuild with Remote Access</NT>**. To find out more: [Remote access](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/remote-access)

   :::


## Rebuilding a failed <NT>Pipeline</NT>

You can rebuild an entire <NT>Pipeline</NT> or certain parts of it at any time. The <NT>Pipeline</NT> build page shows the status of every <NT>Workflow</NT> in your <NT>Pipeline</NT> when a build is running. If a <NT>Workflow</NT> fails, you can save time by re-running the failed <NT>Workflow</NT> instead of the whole <NT>Pipeline</NT>.

:::important[Time limit for partial reruns]

Partial pipeline reruns are available for 30 days after the original pipeline execution. Full reruns can be run at any time.

:::

### Rebuilding a failed <NT>Pipeline</NT>

You can rebuild a <NT>Pipeline</NT> either in full or its unsuccessful <NT>Workflows</NT> with the click of a button:

1. Open <NT>Bitrise CI</NT>.
1. Click a finished <NT>Pipeline</NT> build for your project.
1. On the <NT>Pipeline</NT> summary screen, click **<NT>Rebuild</NT>** to bring up the rebuild dropdown menu.
1. <NT>Select an option</NT>:

   - **<NT>Rebuild unsuccessful Workflows</NT>**: This option only rebuilds the <NT>Workflows</NT> that failed. This option might not be available for variant <NT>Workflows</NT> using the `parallel` property: [Rebuilding Pipelines using different Workflow variants](#rebuilding-pipelines-using-different-workflow-variants).
   - **<NT>Rebuild</NT> the entire <NT>Pipeline</NT>**: This option rebuilds the entire <NT>Pipeline</NT> from start to finish.
   - **<NT>Rebuild unsuccessful Workflows with remote access</NT>**: Remote access allows you to access the <NT>build machine</NT> remotely while the build is running: [Remote access](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/remote-access).
   - **<NT>Rebuild entire Pipeline with remote access</NT>**: See above.

### Rebuilding a <NT>Pipeline</NT> from a failed <NT>Workflow</NT>

You can rebuild a <NT>Pipeline</NT> from a specific failed <NT>Workflow</NT>. This will rebuild all <NT>Workflows</NT> that come after the failed <NT>Workflow</NT> in the <NT>Pipeline</NT>.

This option might not be available for variant <NT>Workflows</NT> using the `parallel` property: [Rebuilding Pipelines using different Workflow variants](#rebuilding-pipelines-using-different-workflow-variants).

1. Open <NT>Bitrise CI</NT>.
1. Click a finished <NT>Pipeline</NT> build for your project.
1. Scroll down to the graph of the <NT>Pipeline</NT>.
1. Find the failed <NT>Workflow</NT> and hover over it on the right of the card.
1. Click the **<NT>Rebuild from here</NT>** button.

### Rebuilding <NT>Pipelines</NT> using different <NT>Workflow</NT> variants

If your <NT>Pipelines</NT> [use different variants of the same Workflow(s)](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline#running-variations-of-the-same-workflow) with the `parallel` property, the available rebuild options depend on the Environment Variable defining the number of parallel runs.

- If the <NT>Workflow</NT> which sets the number of parallel runs has already finished during the failed build, you have to rebuild the whole <NT>Pipeline</NT>.
- If the number hasn't been set yet during the build, you can use a partial rebuild: either rebuilding from a failed <NT>Workflow</NT> or rebuild all unsuccessful <NT>Workflows</NT>.

This is because <NT>Bitrise</NT> only allows partial rebuilds if the rebuild runs with the same <NT>Pipeline</NT> configuration as the original build. Two <NT>Pipeline</NT> configurations are identical if they have the same number of <NT>Workflows</NT> and the <NT>Workflows</NT> have the same names. When the parallel count is set, <NT>Bitrise</NT> has no way of knowing if the number is going to be the same in the rebuild. If the number is different, the two <NT>Workflow</NT> configurations are also different so a full rebuild is the only option.
