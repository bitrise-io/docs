All <NT>Bitrise</NT> builds have a build number. The first build of your project is, by default, number 1, and the build number gets incremented with each build. You can get a build’s <NT>Bitrise build number</NT> on the website, as well as via the `$BITRISE_BUILD_NUMBER` Environment Variable. This variable can be used in any <NT><GlossTerm baseform="step">Step</GlossTerm></NT> or script where you need the <NT>Bitrise build number</NT> for any reason.

You can change the build number manually on the web UI. For details, see: [Changing the build number on the web UI](/en/bitrise-ci/run-and-analyze-builds/build-numbering-and-app-versioning#changing-the-build-number-on-the-web-ui).

<NT>Bitrise</NT> also helps you taking care of your mobile app’s versioning automatically, using <NT>Bitrise</NT>. We have two <NT>Steps</NT> to do it for you, one for iOS and one for <NT>Android</NT>. For details, see: [Setting up app versioning automatically on Bitrise](/en/bitrise-ci/run-and-analyze-builds/build-numbering-and-app-versioning#setting-up-app-versioning-automatically-on-bitrise).


## Changing the build number on the web UI 

The build number is automatically incremented whenever you run a build, but you can set it manually on the **<NT>Project settings</NT>** page of your <NT>Bitrise</NT> <NT>project</NT>. 

:::note[Negative build numbers]

The build number must be a positive integer. Zero or negative numbers are not accepted. You can, however, set any number you have already used.

:::

1. <Partial_GettingToTheProjectSettingsPage />
1. On the left, select **<NT>Builds</NT>**.
1. Find **<NT>Next build number</NT>** and click **<NT>Edit</NT>**.

   ![Next build number section on the Builds page of Project settings](/img/run-and-analyze-builds/2026-07-13-next-build-number-project-settings.png)
1. Add the desired build number then click **<NT>Save</NT>**.


## Setting up app versioning automatically on <NT>Bitrise</NT>

Track the version of your mobile app in its Git repository by modifying the file containing the essential information about the app (for example, the `Info.plist` file for iOS apps and either the `build.gradle` file or the `AndroidManifest.xml` file for <NT>Android</NT> apps).

<NT>Bitrise</NT> has two <NT>Steps</NT> to do this for you. These two <NT>Steps</NT> can insert the <NT>Bitrise build number</NT> or some other specified number into their respective files:

- **<NT>Change Android versionCode and versionName</NT>** for <NT>Android</NT> apps.
- **<NT>Set Xcode Project Build Number</NT>** for iOS apps.

Both <NT>Steps</NT> accept either numbers (integers and/or numeric strings) or <NT>environment variables</NT> in their relevant inputs. By default, both use the $<NT>BITRISE_BUILD_NUMBER</NT> Environment Variable as their <NT>default value</NT> for the build number.

### Setting the versionCode and the versionName of an <NT>Android</NT> app

For an <NT>Android</NT> app, the setting is used as an internal version number, to determine if a build of the app is more recent than another build. The `versionName` setting is a string used as the version number shown to users.

For in-depth information about <NT>Android</NT> versioning, please check out [the Android developer guide on the subject](https://developer.android.com/studio/publish/versioning).

:::note[Version information in the manifest file]

In this guide, we’re setting the version information in the `build.gradle` file. This is the recommended method: you can also set it directly in the `AndroidManifest.xml` file but be aware that any version info set in the manifest file before the build will be overwritten by the `build.gradle` file’s settings during the build.

:::

To configure <NT>Android</NT> versioning:

1. Add the **[Change Android versionCode and versionName](https://bitrise.io/integrations/steps/change-android-versioncode-and-versionname)** <NT>Step</NT> to your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>.
1. Set the the path to your `build.gradle` file in the **<NT>Path</NT> to the build.gradle file** input.

   ![change-android-version.png](/img/_paligo/uuid-fb7fb260-5679-1832-9448-228a048e2b80.png)
1. Add a value in the **<NT>New</NT> versionCode** input.

   The <NT>default value</NT> is the $<NT>BITRISE_BUILD_NUMBER</NT> Environment Variable.
1. Add a value in the **<NT>New</NT> versionName** input.

### Setting the CFBundleVersion and CFBundleShortVersionString of an iOS app

For an iOS app, the value of the `CFBundleVersion` key (“Bundle version” in <NT>Xcode</NT>) is the build number of the app while the value of the `CFBundleShortVersionString` key (“Bundle versions string, short” in <NT>Xcode</NT>) is the version number of the app.

For in-depth information about iOS versioning, including the functions of the `CFBundleVersion` and the `CFBundleShortVersionString` keys, please check out [this Apple technical note](https://developer.apple.com/library/archive/technotes/tn2420/_index.html); you can also look up the [summary of most important keys](https://developer.apple.com/library/archive/documentation/General/Reference/InfoPlistKeyReference/Articles/CoreFoundationKeys.html).

:::important[Setting up iOS versioning for apps using <NT>Xcode</NT> 13+]

To set up iOS versioning on <NT>Bitrise</NT> using <NT>Xcode</NT> 13+, update the following in <NT>Xcode</NT>:

- Set the **Generate <NT>Info.plist</NT> File** to `No`, under **PROJECT** and **<NT>TARGETS</NT>** on the **<NT>Build Settings</NT>** tab.
- Make sure you have all the necessary keys defined in the `Info.plist` file.

:::

1. Add the **<NT>Set Xcode Project Build Number</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Set the file path to the `Info.plist` file in the **<NT>Project path</NT>, scheme and Target** input.
1. Add a value in the **<NT>Build Number</NT>** input. This sets the `CFBundleVersion` key to the specified value in the `Info.plist` file. The <NT>default value</NT> is the $<NT>BITRISE_BUILD_NUMBER</NT> Environment Value.
1. Optionally, add a value in the **<NT>Version</NT> Number** input. This will set the `CFBundleShortVersionString` key to the specified value in the `Info.plist` file. This input is not required.


## Offsetting the build version

You can offset your app’s build version numbers if you tracked versions differently before using <NT>Bitrise</NT>. For example, if your app already had six builds before you started using <NT>Bitrise</NT>, you can offset the version code so it continues from where you left off, instead of restarting from 1.

<Tabs>
<TabItem value="android" label="Android" default>

1. Add the **<NT>Change Android versionCode and versionName</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. In the **<NT>New</NT> versionCode** input, use `$BITRISE_BUILD_NUMBER` (this is the input's <NT>default value</NT>).
1. In the **versionCode Offset** input, set the value to add to the build number. This can be a fixed integer or an Environment Variable.

:::tip[Offsetting an <NT>Android</NT> app's version example]

Let’s say you're about to run your app's fifth build on <NT>Bitrise</NT>, and the app already had six builds before you started using <NT>Bitrise</NT>. To keep versionCode numbering continuous:

- New versionCode: `$BITRISE_BUILD_NUMBER`
- versionCode Offset: `6`
- New versionName: `1.0.5`

`$BITRISE_BUILD_NUMBER` (5, in this example) is added to the versionCode Offset (6), so versionCode is set to 11. versionName isn't affected by the offset and stays 1.0.5 — so this build is versionName 1.0.5, versionCode 11 (the 11th build of version 1.0.5).

:::

</TabItem>
<TabItem value="ios" label="iOS">

1. Add the **<NT>Set Xcode Project Build Number</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. In the **<NT>Build Number</NT>** input, use `$BITRISE_BUILD_NUMBER` (this is the input's <NT>default value</NT>).
1. In the **<NT>Build Number</NT> Offset** input, set the value to add to the build number. This can be a fixed integer or an Environment Variable.

:::tip[Offsetting an iOS app's version example]

Let’s say you're about to run your app's fifth build on <NT>Bitrise</NT>, and the app already had six builds before you started using <NT>Bitrise</NT>. To keep the build number continuous:

- <NT>Build Number</NT>: `$BITRISE_BUILD_NUMBER`
- <NT>Build Number</NT> Offset: `6`
- Version Number: `1.1`

`$BITRISE_BUILD_NUMBER` (5, in this example) is added to the Build Number Offset (6), so CFBundleVersion is set to 11. CFBundleShortVersionString isn't affected by the offset and stays 1.1 — so this build is version 1.1, build 11.

:::

</TabItem>
</Tabs>
