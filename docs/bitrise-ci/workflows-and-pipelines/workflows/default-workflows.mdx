When you add a new project on [bitrise.io](https://www.bitrise.io), we create initial <NT><GlossTerm baseform="Pipeline">Pipeline</GlossTerm></NT>s and <NT><GlossTerm baseform="Workflow">Workflow</GlossTerm></NT>s for you. These are called default <NT>Pipelines</NT> and <NT>Workflows</NT>. A default <NT>Pipeline</NT> consists of <NT>default Workflows</NT>.

You can use these to run your tests or create installable binaries such as IPAs or APKs. Feel free to modify the default <NT>Pipelines</NT> and <NT>Workflows</NT> to suit your needs.

:::tip[Default <NT>Pipelines</NT>]

We also create default <NT>Pipelines</NT>: [Default Pipelines](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/default-pipelines).

:::


## <NT>Default Workflows</NT> for an iOS project

For a new iOS project, <NT>Bitrise</NT> automatically generates several <NT>Workflows</NT>. You can customize these <NT>Workflows</NT> or [create new ones](/en/bitrise-ci/workflows-and-pipelines/workflows/creating-a-workflow) based on them.

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your <NT>Xcode</NT> tests and get the test report. | The <NT>Workflow</NT> clones your Git repository, caches and installs your project’s dependencies if any, runs your <NT>Xcode</NT> tests and saves the test results. |
| `build_for_testing` | Builds your <NT>Xcode</NT> project ready for testing. | The <NT>Workflow</NT> clones your Git repository, builds your app ready for testing, apportions tests into each test shard and deploys the app plus tests to <NT>Bitrise</NT> so they can be used in subsequent <NT>Workflows</NT>.  This <NT>Workflow</NT> is part of a default <NT>Pipeline</NT>. |
| `test_without_building` | Tests your iOS app without needing to rebuild it. | Retrieves the app and tests from the preceding <NT>Workflow</NT> in a <NT>Pipeline</NT> and tests compiled bundles using <NT>Xcode</NT>’s `test_without_building` command.  This <NT>Workflow</NT> is part of a default <NT>Pipeline</NT>. |
| `archive_and_export_app` | Run your <NT>Xcode</NT> tests and create an IPA file to install your app on a device or share it with your team. | The <NT>Workflow</NT> clones your Git repository, caches and installs your project’s dependencies if there are any, runs your <NT>Xcode</NT> tests, and [exports an IPA file from the project](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects). |
| `build` | Builds your <NT>Xcode</NT> project.  This <NT>Workflow</NT> is generated only if the <NT><GlossTerm baseform="Project scanner">project scanner</GlossTerm></NT> didn't find any tests configured in your <NT>Xcode</NT> project. | The <NT>Workflow</NT> clones your Git repository, caches and installs your project’s dependencies if there are any, and builds your project. It uses <NT>Xcode</NT>'s [build for testing](/en/bitrise-ci/testing/testing-ios-apps/building-an-ios-app-for-testing) action. |


## <NT>Default Workflows</NT> for an <NT>Android</NT> project

For a new <NT>Android</NT> project, <NT>Bitrise</NT> automatically generates three new <NT>Workflows</NT>. You can customize these <NT>Workflows</NT> or [create new ones](/en/bitrise-ci/workflows-and-pipelines/workflows/creating-a-workflow) based on them.

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your <NT>Android</NT> unit tests and get the test report. | The <NT>Workflow</NT> clones your Git repository, [caches your Gradle dependencies](/en/bitrise-ci/dependencies-and-caching/android-dependencies), installs <NT>Android</NT> tools, runs your [Android unit tests](/en/bitrise-ci/testing/testing-android-apps/android-unit-tests) and saves the [test report](/en/bitrise-ci/testing/deploying-and-viewing-test-results). |
| `build_apk` | Run your <NT>Android</NT> unit tests and create an APK file to install your app on a device or share it with your team. | The <NT>Workflow</NT> clones your Git repository, installs <NT>Android</NT> tools, [sets the project’s version code based on the build number](/en/bitrise-ci/run-and-analyze-builds/build-numbering-and-app-versioning#setting-up-app-versioning-automatically-on-bitrise), [runs Android lint](/en/bitrise-ci/testing/testing-android-apps/running-lint-for-your-android-apps) and unit tests, builds the project’s APK file and save it. |
| `run_instrumented_tests` | Run your <NT>Android</NT> instrumented tests with test sharding and get a test report. | The <NT>Workflow</NT> clones your Git repository, [caches your Gradle dependencies](/en/bitrise-ci/dependencies-and-caching/android-dependencies), installs <NT>Android</NT> tools, boots up an <NT>Android</NT> emulator to run your <NT>Android</NT> instrumented tests and saves the [test report](/en/bitrise-ci/testing/deploying-and-viewing-test-results).  This <NT>Workflow</NT> is part of a default <NT>Pipeline</NT> which utilizes [parallelism](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/configuring-a-bitrise-pipeline#running-variations-of-the-same-workflow): it runs multiple copies of the <NT>Workflow</NT> to shard instrumented tests. |


## <NT>Default Workflows</NT> for a <NT>Node.js</NT> project

<NT>Bitrise</NT> generates one default <NT>Workflow</NT> for a <NT>Node.js</NT> project: a <NT>Workflow</NT> to run your project's lint and test scripts. You can customize this <NT>Workflow</NT> or create new ones based on it.

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your project’s tests. | The <NT>Workflow</NT> clones your Git repository, selects and installs the Node version, installs the `npm` packages, and, depending on the available scripts in your `package.json` file, runs the lint and test scripts. The <NT>Workflow</NT> also supports `node_modules` caching for better performance.  The `lint` and `test` steps use npm or Yarn depending on which package manager was detected. If neither lock file is present, the user selects the package manager interactively during setup. |


## <NT>Default Workflows</NT> for a <NT>Kotlin</NT> Multiplatform project

<NT>Bitrise</NT> generates three <NT>default Workflows</NT> for a <NT>Kotlin</NT> Multiplatform project: one to run tests, one to build an <NT>Android</NT> app, and one to build an iOS app.

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your project’s tests. | The <NT>Workflow</NT> clones your Git repository, runs the `test` <NT>Gradle</NT> task using the **<NT>Gradle</NT> Unit Test** <NT><GlossTerm baseform="step">Step</GlossTerm></NT>, and caches your <NT>Gradle</NT> task outputs. |
| `android_build` | Builds and signs an <NT>Android</NT> app from your <NT>Kotlin</NT> project. | The <NT>Workflow</NT> clones your Git repository, builds your app using the **<NT>Android Build</NT>** <NT>Step</NT>, [signs](/en/bitrise-ci/code-signing/android-code-signing/android-code-signing-using-the-android-sign-step) it with **<NT>Android Sign</NT>**, and caches your <NT>Gradle</NT> task outputs. It produces a downloadable APK file. |
| `ios_build` | Builds and signs an iOS app from your <NT>Kotlin</NT> project. | The <NT>Workflow</NT> clones your Git repository, builds and [signs](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) your app using the **<NT>Xcode Archive & Export for iOS</NT>** <NT>Step</NT>, and caches your <NT>Gradle</NT> task outputs. It produces a downloadable IPA file. |


## <NT>Default Workflows</NT> for a <NT>Java</NT> project

The project scanner can detect <NT>Java</NT> projects and create <NT>default Workflows</NT> for them. A new project is considered a generic <NT>Java</NT> project if:

- It uses <NT>Gradle</NT> (the code contains the usual <NT>Gradle</NT> configuration files and a <NT>Gradle</NT> Wrapper script in the project root directory) and it doesn't use <NT>Android</NT> or <NT>Kotlin</NT> Multiplatform required dependencies.
- It uses <NT>Maven</NT> with a `POM.xml` project configuration file in the root directory, and has a <NT>Maven</NT> Wrapper script.

For a generic <NT>Java</NT> project, <NT>Bitrise</NT> generates <NT>default Workflows</NT> based on the build tool.

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your projects tests. | The <NT>Workflow</NT> clones your Git repository and runs tests by using the **<NT>Run Gradle Tests</NT>** <NT>Step</NT> which runs the `test` <NT>Gradle</NT> task. |

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your projects tests. | The <NT>Workflow</NT> clones your Git repository and runs tests by using a <NT>Script</NT> step, running the <NT>Maven</NT> Wrapper’s test command: `./mvnw test`. |


## <NT>Default Workflows</NT> for a <NT>Ruby</NT> project

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your project’s tests. | The <NT>Workflow</NT> clones your Git repository and restores your gem cache with the `gem-{{ checksum "Gemfile.lock" }}` caching key. It installs dependencies depending on signals in your codebase:  - If MySQL gem is detected, it installs system dependencies. - If Bundler is detected, it installs dependencies with `bundle install`.  If a relation database gem is detected, it performs database setup: `db:create db:schema:load`.  If a test framework or a Rakefile is detected, the <NT>Workflow</NT> runs your tests.  The <NT>Workflow</NT> always saves the <NT>Ruby</NT> gem cache and includes the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT>. |


## <NT>Default Workflows</NT> for a Python project

For Python, a single `run_tests` <NT>Workflow</NT> is generated. The included <NT>Steps</NT> depend on the detected package manager and whether `pytest` was found in the project.

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your project’s tests. | The <NT>Workflow</NT> clones your Git repository and installs Python based on the detected version. If no version is detected, the <NT><GlossTerm baseform="stack">stack</GlossTerm></NT>'s default Python version is used. Installs dependencies: the exact command used depends on the detected package manager.  If `pytest` is detected, the <NT>Workflow</NT> runs your tests. |


## <NT>Default Workflows</NT> for a Flutter project

For Flutter projects, <NT>Bitrise</NT> creates two <NT>default Workflows</NT>: a `run_tests` <NT>Workflow</NT> and a `build_app` <NT>Workflow</NT>.

The exact <NT>Steps</NT> included in these <NT>default Workflows</NT> depend on the project type. However, the main goals of the two <NT>Workflows</NT> are the same, regardless of project type:

| <NT>Workflow</NT> ID | <NT>Workflow</NT> summary | <NT>Workflow</NT> description |
| --- | --- | --- |
| `run_tests` | Run your project’s tests. | The <NT>Workflow</NT> clones your Git repository, installs Flutter with the **Flutter Installer** <NT>Step</NT>, runs **<NT>Restore Dart cache</NT>** to pull your Dart cache. It runs:  - The **<NT>Flutter Test</NT>** <NT>Step</NT> when there is a `test/` directory in the project. - The **<NT>Flutter Analyze</NT>** <NT>Step</NT> when there is no `test/` directory in the project.  After running the tests, it saves the Dart cache and runs the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> to deploy any artifacts. |
| `build_app` | Builds both the iOS and the <NT>Android</NT> app from the Flutter project. | This <NT>Workflow</NT> is only generated when the project has an iOS or <NT>Android</NT> sub-project. For web projects, this <NT>Workflow</NT> is not generated.  The <NT>Workflow</NT> clones your Git repository, installs your code <NT>signing certificate</NT> for the iOS project, installs Flutter, runs the **<NT>Flutter Analyze</NT>** <NT>Step</NT>. It also runs the **<NT>Flutter Test</NT>** <NT>Step</NT> when a `test/` directory is present.  It builds the app with the **<NT>Flutter Build</NT>** <NT>Step</NT>. It builds all detected platforms. |
