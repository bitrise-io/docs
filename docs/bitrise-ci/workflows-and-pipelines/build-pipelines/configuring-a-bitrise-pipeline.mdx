:::important[<NT>Pipelines</NT> with stages]

If you have an existing <NT>Pipeline</NT> configuration from before December 2024, you probably have stages in your <NT>Pipeline</NT>. We strongly recommend to stop using stages and instead focus on configuring dependencies between <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s. These dependencies determine the order of execution in a <NT>Pipeline</NT>. This eliminates unnecessary waiting time in stages and allows you to create more flexible CI configurations.

You can keep using <NT>Pipelines</NT> with stages but they will receive no updates in the future. You can have both types of <NT>Pipelines</NT> in the same <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file but you can't mix and match the two types: you can't have both stages and <NT>Workflow</NT> dependencies in the same <NT>Pipeline</NT>.

You can find the documentation for <NT>Pipelines</NT> with stages here: [Pipelines with stages](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/pipelines-with-stages/configuring-a-pipeline-with-stages).

You can convert a <NT>Pipeline</NT> with stages into the new format: [Converting a Pipeline with stages into a graph Pipeline](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/converting-a-pipeline-with-stages-into-a-graph-pipeline).

:::

You can define your <NT>Pipelines</NT> under a `pipelines` element in a configuration YAML file. You can use the following elements to set up a <NT>Pipeline</NT>:

- A <NT>Pipeline</NT> name: the human readable identifier of the <NT>Pipeline</NT>.
- The `workflows` element: this contains a list of the <NT>Workflows</NT> that are part of the <NT>Pipeline</NT>.
- The `depends_on` property: This is a property of the `workflows element` and accepts an array of <NT>Workflow</NT> names. It defines the dependencies between <NT>Workflows</NT>.

For example, here's a simple <NT>Pipeline</NT> configuration:

![pipeline-example.png](/img/_paligo/uuid-e85b689a-0101-6664-7f58-6e9ae5c851a7.png)

In this <NT>Pipeline</NT>, we have four <NT>Workflows</NT>:

1. <NT>Workflow</NT> A runs first.
1. Once <NT>Workflow</NT> A is finished, <NT>Workflows</NT> B and C start at the same time.
1. <NT>Workflow</NT> D starts only when both <NT>Workflows</NT> B and C are successfully finished.

## Creating a <NT>Pipeline</NT>

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

Open the <NT>Workflow Editor</NT> and select **<NT>Pipelines</NT>** on the left. Click **<NT>Create Pipeline</NT>** and set a name, then click **<NT>Create Pipeline</NT>** again.

To add your first <NT>Workflow</NT>, click **<NT>Workflows</NT>**.

![add-workflow-pipeline.png](/img/_paligo/uuid-6f5bdb24-c8e6-3923-723f-07a33eae84bc.png)

You can keep adding <NT>Workflows</NT> this way. This method creates no dependencies between <NT>Workflows</NT>: they will all run at the same time.

![basic-pipeline.png](/img/_paligo/uuid-13e965a3-cef3-17c5-970b-b4308460ae26.png)

A <NT>Workflow</NT> can depend on one or more <NT>Workflows</NT>: if any of the <NT>Workflows</NT> it depends on fails, the <NT>Workflow</NT> won't run. To create <NT>Workflow</NT> dependencies, hover over the right end of a <NT>Workflow</NT> and click the plus sign:

![add-dependent-workflow.png](/img/_paligo/uuid-cc676406-6eb6-010a-1daa-9a687d3902ea.png)

You can add multiple dependent <NT>Workflows</NT> to the same <NT>Workflow</NT> this way:

![multiple-dependent-flows.png](/img/_paligo/uuid-9e8845d9-aa8e-6c98-642a-4e13fddf93ea.png)

The same <NT>Workflow</NT> can depend on two or more different <NT>Workflows</NT>: you can create a path by dragging from the plus sign to another <NT>Workflow</NT> already in the <NT>Pipeline</NT>:

![drag-and-drop-dep.png](/img/_paligo/uuid-636d0224-45fb-307d-a1d5-69a7d8e84d9f.png)

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

A bare minimum <NT>Pipeline</NT> configuration consists of a single <NT>Workflow</NT>:

```yaml
pipelines:
  example:
    workflows:
      A: {}
```

You can have multiple <NT>Workflows</NT> that all run in parallel:

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B: {}
      C: {}
```

A <NT>Workflow</NT> can depend on one or more <NT>Workflows</NT>: if any of the <NT>Workflows</NT> it depends on fails, the <NT>Workflow</NT> won't run. To create <NT>Workflow</NT> dependencies, you need to use the `depends_on` property:

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        depends_on:
        - A
      C:
        depends_on:
        - A
      D:
        depends_on:
        - B
        - C
```

If a <NT>Workflow</NT> depends on multiple <NT>Workflows</NT>, both YAML array syntaxes can be used:

```yaml
# Option 1
D:
  depends_on:
  - B
  - C
# Option 2
D:
  depends_on: [A, B]
```

</TabItem>
</Tabs>

## Configuration restrictions

- A <NT>Pipeline</NT> can have a maximum of 200 <NT>Workflows</NT>. Each must be an existing <NT>Workflow</NT> under the root level `workflows` element.
- [Utility Workflows](/en/bitrise-ci/workflows-and-pipelines/workflows/managing-workflows#utility-workflows) can't be added to a <NT>Pipeline</NT>. If you want to use utility <NT>Workflows</NT>, include them as part of a regular <NT>Workflow</NT>.
- Each <NT>Workflow</NT> can be added to the same <NT>Pipeline</NT> only once. You can't include the same <NT>Workflow</NT> in multiple different parts of a <NT>Pipeline</NT>.
- Any <NT>Workflow</NT> defined in the dependency list must be part of the same <NT>Pipeline</NT>.
- The dependency graph resulting from your configuration can't contain a circle or loop. You can't start builds with an invalid <NT>Pipeline</NT> configuration.

## Conditional <NT>Workflow</NT> execution in a <NT>Pipeline</NT>

Use `run_if` expressions to control <NT>Workflow</NT> execution: you can set conditions under which a <NT>Workflow</NT> should or should not run.

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. <Partial_WorkflowPropertiesInPipelines />
1. In the **<NT>Additional running conditions</NT>** input, add a valid [Go template](https://pkg.go.dev/text/template).

   The input accepts three helper functions:

   - `getenv`: Accesses an Environment Variable's value.
   - `enveq`: Compares an Environment Variable to a given value.
   - `envcontain`: Checks whether an Environment Variable contains a given string.

   ![run-if-template.png](/img/_paligo/uuid-4ababac0-1bb5-1f86-165a-f31d14fa9f28.png)
1. Close the <NT>Edit Workflow</NT> dialog, and click **<NT>Save changes</NT>**.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

In <NT>Pipelines</NT>, the `run_if` property needs an `expression` field that contains a [Go template](https://pkg.go.dev/text/template):

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        run_if:
          expression: {{ enveq "EXAMPLE_KEY" "example value" }}
        depends_on: [A]
```

The `expression` field accepts three helper functions:

- `getenv`: Accesses an Environment Variable's value.
- `enveq`: Compares an Environment Variable to a given value.
- `envcontain`: Checks whether an Environment Variable contains a given string.

The <NT>Bitrise CLI</NT> evaluates the expression during runtime: a <NT>Workflow</NT> only runs if its `run_if` expression evaluates to `true`. A <NT>Workflow</NT> that is skipped because of a `run_if` expression is counted as a successful <NT>Workflow</NT> so its dependent <NT>Workflows</NT> will run.

In this example, B will only run if the EXAMPLE_KEY Environment Variable has the value `example value`.

</TabItem>
</Tabs>

## Always executing a <NT>Workflow</NT> in a <NT>Pipeline</NT>

You can mark a <NT>Workflow</NT> to make sure it runs even if a <NT>Workflow</NT> it depends on failed.

:::important[Transitive dependency]

Even if a <NT>Workflow</NT> is configured to always run, its dependent <NT>Workflows</NT> might not run. If one or more of its parent <NT>Workflows</NT> failed, the <NT>Workflow</NT> in question will still run but its dependent <NT>Workflows</NT> might not.

For example, let's say we have <NT>Workflow</NT> C that depends on <NT>Workflow</NT> B. <NT>Workflow</NT> B depends on <NT>Workflow</NT> A. Of the three <NT>Workflows</NT>, only B is set to always run:

- If A fails, B will run but regardless of its result, C won't run because by depending on B, it also depends on A.
- C only runs if both A and B are successful.

:::

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. <Partial_WorkflowPropertiesInPipelines />
1. Set the **<NT>Always run</NT>** input to **<NT>Workflow</NT>**.

   This setting means the <NT>Workflow</NT> will run even if one or more of its parent <NT>Workflows</NT> fail. However, its dependent <NT>Workflows</NT> might not run!

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

Use the `should_always_run` property. It requires a string to specify the scope of the configuration. It has two available values:

- `none`: If a parent <NT>Workflow</NT> fails, the <NT>Workflow</NT> will not run. This is the <NT>default value</NT>.
- `workflow`: If a parent <NT>Workflow</NT> fails, the <NT>Workflow</NT> will run anyway.

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        depends_on: [ A ]
        should_always_run: workflow
      C:
        depends_on: [ B ]
      D:
        depends_on: [ C ]
        should_always_run: workflow
```

In this example:

- If A fails, B still runs.
- If either A or B fails, C won't run.
- If C won't run or runs but fails, D will still run.

</TabItem>
</Tabs>

## Aborting the <NT>Pipeline</NT> on <NT>Workflow</NT> failure

You can configure the <NT>Pipeline</NT> to immediately terminate when any given <NT>Workflow</NT> fails. By default, the <NT>Pipeline</NT> won't terminate

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. <Partial_WorkflowPropertiesInPipelines />
1. Toggle the **<NT>Abort Pipeline on failure</NT>** input on.

   If the <NT>Workflow</NT> in question fails, the <NT>Pipeline</NT> will immediately stop running.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

Add the `abort_on_fail` field set to `true` to the selected <NT>Workflows</NT>:

```yaml
pipelines:
  example:
    workflows:
      A: {}
      B:
        abort_on_fail: true
```

In this example, the <NT>Pipeline</NT> stops running if <NT>Workflow</NT> B fails: <NT>Workflow</NT> A will be aborted and no subsequent <NT>Workflows</NT> will run.

</TabItem>
</Tabs>


## <NT>Pipeline</NT> priority

You can set a priority for each <NT>Pipeline</NT>. The priority setting determines the position of the <NT>Pipeline</NT> build in the <NT>build queue</NT>: the higher the priority, the sooner the <NT>Pipeline</NT> build will run.

You can assign a priority either in the <NT>Workflow Editor</NT> or in the configuration YAML file of your project. The priority is always an integer between -100 and 100: the higher the number, the higher the priority. The default priority is 0.

For more information about <NT>build priority</NT>, and the order of precedence between different types of priorities, check out [Build priority](/en/bitrise-ci/configure-builds/configuring-build-settings/build-priority).


## Sharing files between <NT>Workflows</NT> in a <NT>Pipeline</NT>

To share files between <NT>Workflows</NT> in a <NT>Pipeline</NT>, you can use the **<NT>Deploy to Bitrise.io</NT>** <NT><GlossTerm baseform="step">Step</GlossTerm></NT> and the **<NT>Pull Pipeline intermediate files</NT>** <NT>Step</NT>. The <NT>Step</NT> downloads <NT>Pipeline</NT> intermediate files to a local folder.

To configure file sharing:

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Add the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> to the <NT>Workflow</NT> that generates the file(s) you need.
1. In the **<NT>Pipeline</NT> Intermediate File Sharing** input group, find the **Files to share between <NT>Pipeline</NT> <NT>Workflows</NT>** input and add the files as a newline-separated list of colon-separated items using the following structure: `<file_or_directory_path>:<environment_variable_key>`.

   You can use another Environment Variable as a filepath: for an iOS project, `$BITRISE_IPA_PATH:BITRISE_IPA_PATH` is a valid way to share a generated IPA with other <NT>Workflows</NT>.
1. Add the **[Pull Pipeline intermediate files](https://www.bitrise.io/integrations/steps/pull-intermediate-files)** <NT>Step</NT> to any <NT>Workflow</NT> that needs the generated files.
1. Use the **Intermediate file source** input to specify a set of <NT>Workflows</NT> from which you need files.

   ![pipeline-intermediate.png](/img/_paligo/uuid-5d9fd00b-0507-1ae7-80a2-b42a4377343e.png)

   You can use wildcards in the input. In the example above, we’re pulling all intermediate files from all <NT>Workflows</NT> which have a name starting with `workflow`.
1. When the <NT>Step</NT> finishes, your files and directories specified via the **[Deploy to Bitrise.io - Apps, Logs, Artifacts](https://www.bitrise.io/integrations/steps/deploy-to-bitrise-io)** <NT>Step</NT> should be available.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Add the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> to the <NT>Workflow</NT> that generates the file(s) you need.
1. Add the files to the `pipeline_intermediate_files` input as a newline-separated list of colon-separated items using the following structure: `<file_or_directory_path>:<environment_variable_key>`.

   You can use another Environment Variable as a filepath: for an iOS project, `$BITRISE_IPA_PATH:BITRISE_IPA_PATH` is a valid way to share a generated IPA with other <NT>Workflows</NT>.

   ```yaml
   steps:
     - deploy-to-bitrise-io: {
         inputs:        
         - pipeline_intermediate_files: "$BITRISE_IPA_PATH:BITRISE_IPA_PATH"
   ```
1. Add the **[Pull Pipeline intermediate files](https://www.bitrise.io/integrations/steps/pull-intermediate-files)** <NT>Step</NT> to any <NT>Workflow</NT> that needs the generated files:

   ```yaml
   steps:
     - pull-intermediate-files: {}
   ```
1. Use the `artifact_sources` input to specify a set of <NT>Workflows</NT> from which you need files:

   ```yaml
   steps:
     - pull-intermediate-files@1:
         inputs:
           - artifact_sources: workflow.*
   ```

   You can use wildcards in the input. In the example above, we’re pulling all intermediate files from all <NT>Workflows</NT> which have a name starting with `workflow`.
1. When the <NT>Step</NT> finishes, your files and directories specified via the **[Deploy to Bitrise.io - Apps, Logs, Artifacts](https://www.bitrise.io/integrations/steps/deploy-to-bitrise-io)** <NT>Step</NT> should be available.

</TabItem>
</Tabs>

For more information and details, check out the [Step repository](https://www.bitrise.io/integrations/steps/pull-intermediate-files).

## Sharing Env Vars between <NT>Workflows</NT>

You can reuse any environment variable from a <NT>Workflow</NT> and reuse it in subsequent <NT>Workflows</NT> using the [Share Pipeline variables](https://github.com/bitrise-steplib/bitrise-step-share-pipeline-variable) <NT>Step</NT>.

:::tip[Optional <NT>Workflows</NT> using run_if conditions]

You can easily combine the [**Share Pipeline variables**](https://github.com/bitrise-steplib/bitrise-step-share-pipeline-variable) <NT>Step</NT> with `run_if` expressions to create <NT>Pipelines</NT> with optional <NT>Workflows</NT>. For more information, check out [Enabling or disabling a Step conditionally](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally).

:::

To do so:

1. Add the [**Share Pipeline variables**](https://github.com/bitrise-steplib/bitrise-step-share-pipeline-variable) <NT>Step</NT> to the <NT>Workflow</NT>.
1. Optionally, you can define <NT>additional run conditions</NT> in the **<NT>Additional run conditions</NT>** input. The <NT>Step</NT> will only run if the conditions you specify here are true.
1. Add the Env Var(s) you would like to use in subsequent <NT>Workflows</NT> in the **<NT>Variable</NT>s to share between <NT>Pipeline</NT> <NT>Workflows</NT>** input.

   :::tip[Using environment variable keys]

   You can define Env Vars using a `{key}={value}` syntax. For example, `MY_ENV_KEY=value`, or `INSTALL_PAGE_URL=$BITRISE_PUBLIC_PAGE_URL`.

   If you want to use the default environment variable keys, you can use a shorthand syntax. For example, `EXISTING_ENV_KEY`.

   Sharing Env Vars using this <NT>Step</NT> does not override existing Env Vars defined in the app.

   :::


## Running variations of the same <NT>Workflow</NT>

Parallelism allows you to split the execution of a single <NT>Workflow</NT> in a <NT>Pipeline</NT> over many runners in a single instruction. This is particularly useful for test sharding: you can reuse a single testing <NT>Workflow</NT> several times over, without needing to specify boilerplate code for each shard. At runtime, <NT>Bitrise</NT> automatically creates a set of parallel <NT>Workflows</NT> and provides you with the shard number and total number of shards as <NT>environment variables</NT>.

You can also use a parallel configuration to build and test multiple white label apps from the same codebase, or to achieve any other goal that requires running the same CI jobs multiple times.

### Parallelism overview

:::important[Graph pipelines only]

Parallelism isn't available for <NT>Pipelines</NT> containing stages. To take advantage of the feature, convert your stage-based <NT>Pipelines</NT> into graph <NT>Pipelines</NT>: [Converting a Pipeline with stages into a graph Pipeline](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/converting-a-pipeline-with-stages-into-a-graph-pipeline).

:::

You can create a parallel configuration for a <NT>Workflow</NT> by using the `parallel` property. A <NT>Pipeline</NT> can have multiple <NT>Workflows</NT> with a parallel configuration. These <NT>Workflows</NT> can depend on other <NT>Workflows</NT>. If a <NT>Workflow</NT> depends on a <NT>Workflow</NT> with a parallel configuration, it will depend on the execution of all copies of the <NT>Workflow</NT>.

The `parallel` property takes an integer for a value: for example, if the value is 5, five copies of the <NT>Workflow</NT> will be executed. Each copy receives two new <NT>environment variables</NT>:

- $BITRISE_IO_PARALLEL_INDEX: a zero based index for each copy of the <NT>Workflow</NT>.
- $BITRISE_IO_PARALLEL_TOTAL: the total number of copies.

### Configuring parallelism

To set up the configuration:

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Open the <NT>Workflow Editor</NT>.
1. In the <NT>Pipeline</NT> graph, find your <NT>Workflow</NT> and click the gear icon to access the <NT>Workflow</NT> configuration.
1. On the **<NT>Configuration</NT>** tab, find the **<NT>Parallel copies</NT>** input and set a value.

   :::important[Value limitations]

   The value has to be a fixed number. You can only use integers. The maximum accepted value is 200.

   You can use an Environment Variable as the value: it allows you to dynamically calculate the optimal number of <NT>parallel copies</NT> during the build.

   :::

   If another <NT>Workflow</NT> depends on the <NT>Workflow</NT> with a parallel configuration, it will run only if all copies of the <NT>Workflow</NT> have been successful.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Open your [configuration YAML](/en/bitrise-ci/configure-builds/configuration-yaml/accessing-a-build-s-bitrise-yml-file) file.
1. In the `workflows` property of your <NT>Pipeline</NT>, find the <NT>Workflow</NT> you need.
1. Add the `parallel` property to it and set a value.

   :::important[Value limitations]

   The value has to be a fixed number. You can only use integers. The maximum accepted value is 200. You can use an Environment Variable as the value.

   You can use an Environment Variable as the value: it allows you to dynamically calculate the optimal number of <NT>parallel copies</NT> during the build.

   :::

   In this example, the `run-tests` <NT>Workflow</NT> is executed five times in parallel. The `report-results` <NT>Workflow</NT> depends on `run-tests` so it only runs once all 5 copies have been successful.

   ```yaml
   pipelines:
     my-ci-pipeline:
       workflows:
         build-without-testing: {}
         run-tests:
           depends_on: [build-without-testing]
           parallel: 5
         report-results:
           depends_on: [run-tests]
   ```

</TabItem>
</Tabs>

## Troubleshooting a <NT>Pipeline</NT>

If a <NT>Pipeline</NT> build fails, you have two main options:

- [Rebuild the Pipeline](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/rebuilding-a-failed-build#rebuilding-a-failed-pipeline). You can rebuild either the entire <NT>Pipeline</NT>, or the failed <NT>Workflows</NT>.
- Rebuild with [remote access](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/remote-access) which allows you to access the <NT>build machine</NT> while the build is running.
