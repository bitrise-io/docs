Configuring a <NT><GlossTerm baseform="Pipeline">Pipeline</GlossTerm></NT> that contains stages is only possible by directly editing the <NT><GlossTerm baseform="bitrise.yml">bitrise.yml</GlossTerm></NT> file. You can create and modify <NT><GlossTerm baseform="Workflow">Workflow</GlossTerm></NT>s in the graphical <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> but you need to define <NT>Pipelines</NT> and Stages in YAML format.


## Defining <NT>Pipelines</NT> with stages

The `bitrise.yml` file contains the full configuration of your <NT>Pipelines</NT>. As with all `bitrise.yml` files, first you need to define the format version and the project type.

```yaml
---
format_version: '8'
default_step_lib_source: 
project_type: android
```

This is a bare minimum `bitrise.yml` configuration. To define your <NT>Pipelines</NT>, you will need to use the `pipelines` attribute.

```yaml
pipelines:  
  pipeline-successful:
    stages:
    - stage-successful-1: {}
    - stage-successful-2: {}
    - stage-successful-3: {}
```

In this example, we have a <NT>Pipeline</NT> called `pipeline-successful`, with three Stages that will run consecutively. This means that if `stage-successful-1` finishes successfully, `stage-successful-2` starts. If any of the Stages fail, the subsequent Stage will not start: instead, the <NT>Pipeline</NT> will be aborted and marked as failed.

Each Stage has to be defined separately under the `stages` attribute. Defining a Stage means specifying the <NT>Workflows</NT> that are part of the Stage.

```yaml
stages:
  stage-successful-1:
    workflows:
    - test-1: {}
  stage-successful-2:
    workflows:
    - build-1: {}
    - build-2: {}
  stage-successful-3:
    workflows:
    - deploy-1: {}
    - deploy-2: {}
```

In this example, the Stages run the `test-1`, `build-1`, `build-2`, `deploy-1`, and `deploy-2` <NT>Workflows</NT>.


## Configuring a Stage to always run

By default, if a Stage fails - because one of its <NT>Workflows</NT> failed -, any other subsequent Stages of the <NT>Pipeline</NT> will not run. However, you can configure your <NT>Pipeline</NT> to run certain Stages unless the <NT>Pipeline</NT> is aborted.

To do so, you just need to set the should_always_run attribute of the Stage to true:

```yaml
stages:
  stage-always-run-successful-1:
    should_always_run: true
    workflows:
    - deploy-1: {}
    - deploy-2: {}
```

In the example above, the Stage called `stage-always-run-successful-1` will always run, regardless of the status of previous Stages. The only way these Stages will not run is if the <NT>Pipeline</NT> build is aborted by the user.


## Aborting the <NT>Workflows</NT> of a failed Stage

By default, if a <NT>Workflow</NT> in a particular Stage fails, the other <NT>Workflows</NT> in the same Stage aren’t automatically aborted: these <NT>Workflows</NT> will run but the next Stage won’t start. However, you can change this behavior to immediately and automatically abort all other <NT>Workflows</NT> in the same Stage.

To do so, you need to set the abort_on_fail attribute to `true`:

```yaml
stages:
  stage-abort-on-fail-1:
    abort_on_fail: true
    workflows:
    - deploy-1: {}
    - deploy-2: {}
```


## Sharing Env Vars between <NT>Pipeline</NT> Stages

You can reuse any environment variable from a <NT>Workflow</NT> and reuse it in subsequent <NT>Workflows</NT> using the [Share Pipeline variables](https://github.com/bitrise-steplib/bitrise-step-share-pipeline-variable) <NT><GlossTerm baseform="Step">Step</GlossTerm></NT>.

:::tip[Optional <NT>Workflows</NT> using run_if conditions]

You can easily combine the **<NT>Share Pipeline variables</NT>** <NT>Step</NT> with `run_if` expressions to create <NT>Pipelines</NT> with optional <NT>Workflows</NT>. For more information, check out [Enabling or disabling a Step conditionally](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally).

:::

To do so:

1. Add the **<NT>Share Pipeline variables</NT>** <NT>Step</NT> to the <NT>Workflow</NT>.
1. Optionally, you can define <NT>additional run conditions</NT> in the **<NT>Additional run conditions</NT>** input. The <NT>Step</NT> will only run if the conditions you specify here are true.
1. Add the Env Var(s) you would like to use in subsequent <NT>Workflows</NT> in the **<NT>Variable</NT>s to share between <NT>Pipeline</NT> <NT>Workflows</NT>** input.

   :::tip[Using environment variable keys]

   You can define Env Vars using a `{key}={value}` syntax. For example, `MY_ENV_KEY=value`, or `INSTALL_PAGE_URL=$BITRISE_PUBLIC_PAGE_URL`.

   If you want to use the default environment variable keys, you can use a shorthand syntax. For example, `EXISTING_ENV_KEY`.

   Sharing Env Vars using this <NT>Step</NT> does not override existing Env Vars defined in the app.

   :::

That's it! You can now use the Env Var in any subsequent Stage!
