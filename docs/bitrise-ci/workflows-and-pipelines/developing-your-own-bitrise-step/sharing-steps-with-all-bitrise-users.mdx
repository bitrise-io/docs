If you think your <NT><GlossTerm baseform="Step">Step</GlossTerm></NT> can be of use to others, you can share it! Before doing so, however, please check that there are no other <NT>Steps</NT> that solve the same problem. You can check:

- In the list of [released Steps](https://www.bitrise.io/integrations/steps).
- In the list of [open pull requests](https://github.com/bitrise-io/bitrise-steplib/pulls) in our StepLib.
- In the list of [closed pull requests](https://github.com/bitrise-io/bitrise-steplib/pulls?q=is%3Apr+is%3Aclosed) in our StepLib.

If you find a <NT>Step</NT> which is missing a particular feature, please try to contribute to it before developing or sharing a brand new <NT>Step</NT> for that feature. You can use our [Integrations page](https://www.bitrise.io/integrations/steps) to search for <NT>Steps</NT> in the <NT>Bitrise</NT> StepLib. By clicking the <NT>GitHub</NT> source button on a given <NT>Step</NT>’s page you will be taken to the <NT>Step</NT>’s repository, where you can submit a feature request or open a pull request.

Also, please keep in mind that shared <NT>Steps</NT> must be actively maintained and they must be open to contributors.


## Sharing a new <NT>Step</NT>

The sharing process is relatively straightforward - but please make sure to go through the process carefully when sharing a new <NT>Step</NT>.

:::note[The bitrise share command]

During <NT>Step</NT> development, you can get help anytime by simply running bitrise share. This will print a short guide on sharing.

:::

There are two ways to share a <NT>Step</NT>:

- You can do it manually, using the `bitrise share` command.
- If you created the <NT>Step</NT> with the <NT>Step</NT> plugin, you can run the `share-this-step` <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT> in your <NT>Step</NT>’s directory.

### Before you start

Before you start, make sure that:

- Your <NT>Step</NT> is in a public Git repository.
- The <GlossTerm baseform="step.yml">step.yml</GlossTerm> file contains a support_url property which points to a valid issue tracker (for example, the **Issues** page of your <NT>Step</NT>’s <NT>GitHub</NT> repository).
- The step.yml file contains a source_code_url property which points to the correct Git URL of your repository.

:::important[Sharing more than one <NT>Step</NT>]

Sharing more than one <NT>Step</NT> - that is, adding more than one new step.yml file to the <NT>Bitrise</NT> StepLib - must be done in separate pull requests! You cannot open a pull request that contains more than one new <NT>Step</NT>!

:::

If you’re ready, go ahead with your preferred sharing process!

### Sharing with the bitrise share command

1. Fork the [Bitrise StepLib](https://github.com/bitrise-io/bitrise-steplib.git) repository.
1. Prepare your forked StepLib locally for sharing:

   ```bash
   $ bitrise share start -c <https://github.com/[your-username]/bitrise-steplib.git>
   ```
1. Add the <NT>Step</NT> version tag to your <NT>Step</NT>’s repository.
1. Add the <NT>Step</NT> to your forked StepLib repository:

   ```bash
   $ bitrise share create --tag [step-version-tag] --git [step-git-uri].git --stepid [step-id]
   ```
1. Optionally, perform a complete health check on your forked StepLib:

   ```bash
   $ bitrise share audit -c <https://github.com/[your-username]/bitrise-steplib.git>
   ```
1. Review your <NT>Step</NT>’s step.yml file, and if you’re happy with it, finish the share process: This commits and pushes the step.yml file to the forked StepLib repository.

   ```bash
   $ bitrise share finish
   ```

   This commits and pushes the step.yml file to the forked StepLib repository.
1. Open a pull request in the official [Bitrise StepLib](https://github.com/bitrise-io/bitrise-steplib.git) repository.

### Sharing with the share-this-step <NT>Workflow</NT>

The share-this-step <NT>Workflow</NT> is included in the <NT><GlossTerm baseform="bitrise.yml">bitrise.yml</GlossTerm></NT> file that the <NT>Step</NT> plugin automatically generates when you create a new <NT>Step</NT> with it. Using it, sharing a <NT>Step</NT> is incredibly easy:

1. Fork the [Bitrise StepLib](https://github.com/bitrise-io/bitrise-steplib.git) repository.
1. Set the required <NT>Workflow</NT> <NT>Environment Variables</NT> as app level <NT>Environment Variables</NT> in the bitrise.yml file:

   ```yaml
   app:
     envs:
     - BITRISE_STEP_ID:
     - BITRISE_STEP_VERSION:
     - BITRISE_STEP_GIT_CLONE_URL:
     - MY_STEPLIB_REPO_FORK_GIT_URL:
   ```
1. Run the share-this-step <NT>Workflow</NT> in the <NT>Bitrise CLI</NT>:

   ```bash
   $ bitrise run share-this-step
   ```

### Fixing issues in a StepLib pull request

Once you submitted your <NT>Step</NT> version to the StepLib, wait for the <NT>Bitrise</NT> team to review it. If we ask for changes:

1. Close the pull request.
1. Delete the share branch from your fork of the <NT>Bitrise</NT> StepLib.
1. Fix the issues in the <NT>Step</NT> repository.
1. Add a new version tag to the commit that contains your fixes in the <NT>Step</NT> repository.
1. Run the [share process](/en/bitrise-ci/workflows-and-pipelines/developing-your-own-bitrise-step/sharing-steps-with-all-bitrise-users#sharing-a-new-step) again.

Hopefully, after fixing the issues, we’ll be able to merge your pull request and release your <NT>Step</NT> to the public!

## Adding a <NT>Step</NT> icon

You can add a <NT>Step</NT> icon to your <NT>Step</NT>: you will see it in the <NT>Workflow Editor</NT> and on our [Integrations](https://www.bitrise.io/integrations/) page. This is optional: you only need an icon if you want to add your <NT>Step</NT> to the official <NT>Bitrise Step</NT> Library. There are some requirements:

- Its background color should not be transparent.
- Size: 256x256 px.
- Margin: 60 px.
- Format: SVG.

To submit your <NT>Step</NT>’s icon:

- Add the .svg file into your StepLib fork repo at: `STEPLIB_FORK_ROOT/steps/YOUR_STEP_ID/assets/icon.svg`.
- Create a new pull request to the [StepLib repository](https://github.com/bitrise-io/bitrise-steplib).


## Abandoned <NT>Steps</NT>

If you are a <NT>Step</NT> maintainer, you should be reachable within a reasonable timeframe if anyone submits an issue or a pull request to your <NT>Step</NT>. If we try to contact you several times regarding an important fix/update in your <NT>Step</NT> and you refuse to answer for several weeks we might deprecate, remove or replace your <NT>Step</NT> in the collection! Abandoned <NT>Steps</NT> can be a threat for those who use it, please keep this in mind if you decide to share your <NT>Step</NT> with others!

The repository and issue tracker must not be removed, as there are permanent links to it included in the step.yml that is in the <NT>Bitrise</NT> StepLib. If they are removed, already shared <NT>Step</NT> versions will continue to function as they are also shared on a <NT>Bitrise</NT> managed file hosting service during the share process. A new version of the step may be released, managed by <NT>Bitrise</NT>.

If you shared a <NT>Step</NT> but you’re no longer able or willing to maintain it, please create a <NT>GitHub</NT> issue in this repository: [Bitrise StepLib](https://github.com/bitrise-io/bitrise-steplib)


## Reporting <NT>Step</NT> issues

If you’re a user of a <NT>Step</NT> which has critical (security or functionality) issues, please create a ticket in the <NT>Step</NT>’s Issue Tracker. Every <NT>Step</NT> declares the preferred way of reporting issues with the `support_url` attribute.

If you don’t get a response from the <NT>Step</NT>’s maintainer for an extended period (for more than a couple of weeks) please create a <NT>GitHub</NT> issue in this repository: [https://github.com/bitrise-io/bitrise-steplib](https://github.com/bitrise-io/bitrise-steplib) and we’ll try to resolve the issue, following the Abandoned <NT>Step</NT> policy. Please be patient and keep in mind that everyone who contributes to this collection does so to help you by providing a <NT>Step</NT> for you to use!
