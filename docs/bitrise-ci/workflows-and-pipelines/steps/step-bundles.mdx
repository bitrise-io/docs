<NT><GlossTerm baseform="Step">Step</GlossTerm></NT> bundles allow you to group multiple <NT>Steps</NT> into a single unit. With <NT>Step bundles</NT>, you can reuse <NT>Steps</NT> and sequences of <NT>Steps</NT> without manually copying and pasting from a YAML configuration file.

<NT>Step bundles</NT> are configured with a root level entity called `step_bundles`. You can insert <NT>Step bundles</NT> into any <NT><GlossTerm baseform="Workflow">Workflow</GlossTerm></NT>. Unlike utility <NT>Workflows</NT> and [Workflow chaining](/en/bitrise-ci/workflows-and-pipelines/workflows/managing-workflows#chaining-workflows-together), <NT>Step Bundles</NT> can be placed anywhere in a <NT>Workflow</NT>.

When running a build, each <NT>Step</NT> within the bundle will run at the position the bundle is inserted.

## Creating a <NT>Step bundle</NT>

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Open the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT>.
1. On the left, select **<NT>Step bundles</NT>**.
1. Click **<NT>Create Step bundle</NT>** if you don't have any bundles.

   If you already have at least one <NT>Step bundle</NT>, open the dropdown menu in the middle and click **<NT>Create Step bundle</NT>**.

   ![create-step-bundle.png](/img/_paligo/uuid-955d7e00-c03b-fa26-f239-dcf2f2b2708a.png)
1. In the dialog, provide an ID for your bundle.

   ![Create Step bundle dialog](/img/workflows-and-pipelines/2026-07-06-create-step-bundle-dialog.png)
1. If you have existing <NT>Step bundles</NT>, you can select one from the dropdown menu: your new <NT>Step bundle</NT> will be a copy of the existing one.
1. When done, click **<NT>Create Step bundle</NT>**.
1. Add your first <NT>Step</NT> by clicking **<NT>Add Step or Step bundle</NT>**.

   Adding a <NT>Step bundle</NT> within another <NT>Step bundle</NT> is called nesting: [Nested Step bundles](/en/bitrise-ci/workflows-and-pipelines/steps/step-bundles#nested-step-bundles).
1. Add additional <NT>Steps</NT> or bundles by hovering above or under an already added <NT>Step</NT> or bundle and clicking on the plus sign.
1. Configure new bundle inputs under the **<NT>Configuration</NT>** tab by clicking **+ <NT>Add input</NT>**. In the **<NT>New</NT> bundle input**dialog you can define input variables to manage multiple <NT>Steps</NT> within a bundle. Reference their keys in <NT>Steps</NT> and assign custom values for each <NT>Workflow</NT>.

   ![bundleinput.png](/img/_paligo/uuid-b51f2636-2f9d-89c2-595e-db6ab80d5416.png)

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. [Open the configuration YAML of your project](/en/bitrise-ci/configure-builds/configuration-yaml/editing-an-app-s-bitrise-yml-file).
1. Add a `step_bundles` entity to the top level of the YAML file and give it a name. You will refer to the bundle by this name.

   ```yaml
   format_version: 11
   step_bundles:
     install_deps:
   ```
1. Add a `steps` property under your bundle and add the <NT>Steps</NT> you want to run. You can [reference Steps](/en/bitrise-ci/references/steps-reference/step-reference-id-format) here the same way as in a <NT>Workflow</NT>.

   In this example, we add the *restore-cache* <NT>Step</NT> and an `npm` <NT>Step</NT>.

   ```yaml
   format_version: 11
   step_bundles:
     install_deps:
       steps:
       - restore-cache@1: {}
       - npm@1: {}
   ```
1. Add your <NT>Step bundle</NT> to an existing <NT>Workflow</NT>. The syntax works like this: `bundle::<bundle-name>`.

   ```yaml
   workflows:
     ci:
       steps:
       - git-clone@8.1: {}
       - bundle::install_deps: {}
       - deploy-to-bitrise-io@2: {}
   ```

</TabItem>
</Tabs>

## Creating a <NT>Step bundle</NT> based on a utility <NT>Workflow</NT>

You can create a <NT>Step bundle</NT> based on a [utility Workflow](/en/bitrise-ci/workflows-and-pipelines/workflows/managing-workflows#utility-workflows). The bundle will contain all the <NT>Steps</NT> that were part of the utility <NT>Workflow</NT>. But unlike a utility <NT>Workflow</NT>, you can place the <NT>Step bundle</NT> anywhere within another <NT>Workflow</NT>.

1. Open the <NT>Workflow Editor</NT>.
1. On the left, select **<NT>Step bundles</NT>**.
1. From the **<NT>Based on</NT>** dropdown menu, choose a utility <NT>Workflow</NT>.

   ![Screenshot_2025-03-25_at_09_59_54.png](/img/_paligo/uuid-8b6d8a17-8028-3064-a48d-9a43b0328391.png)
1. Click **<NT>Create Step bundle</NT>**.

## Naming <NT>Step bundle</NT> instances

You can add optional metadata fields to <NT>Step bundle</NT> instances to make your <NT>Workflow</NT> configs and build logs easier to read in <NT>Workflows</NT>.

:::note[<NT>Step bundle</NT> instance]

<NT>Step bundles</NT> are reusable config blocks - you define them once and can use them (instantiate them) across <NT>Workflows</NT> or in the same <NT>Workflow</NT>.

:::

You can set three optional metadata fields: **title**, **summary** and **description** in your <NT>Step bundle</NT>. These attributes override the generic information from the bundle's definition for that specific instance only, giving you granular control over how it's displayed in a particular <NT>Workflow</NT>. If you use these attributes, you don't need to add comments to <NT>Step bundles</NT> to distinguish two instances from each other.

A <NT>Step bundle</NT> instance can include:

- **title**: a human-readable name for this specific usage.
- **summary**: a brief description.
- **description**: detailed information about what this instance does.

![stepbundleinstance.png](/img/_paligo/uuid-9cf7ca34-d643-d13c-d241-42cefd6dc847.png)

These fields show up in the <NT>Bitrise</NT> UI and build logs, making it immediately clear what each instance is doing.

```yaml
- step-bundle-id:
    title: "A clear, human-readable title for this instance"
    summary: "A short summary of what this instance does."
    description: "A more detailed markdown-supported description if needed."
    inputs:
      - ...
```

**Additional attributes for <NT>Step bundle</NT> instances**

Let's say you have a <NT>Step bundle</NT> defined with the ID `run-my-tests` that builds and tests a specific scheme. You want to use it twice in your primary <NT>Workflow</NT>: once for unit tests on an iPhone simulator, and again for UI tests on an iPad simulator.

```yaml
workflows:
  primary:
    steps:
    - run-my-tests:
        title: "Run unit tests on iPhone"
        summary: Runs the UnitTests test plan.
        inputs:
        - scheme: "MyApp"
        - destination: "platform=iOS Simulator,name=iPhone 14"
        - test_plan: "UnitTests"
    - run-my-tests:
        title: "Run UI tests on iPad"
        summary: Runs the UITests test plan.
        inputs:
        - scheme: "MyApp"
        - destination: "platform=iOS Simulator,name=iPad Pro (12.9-inch)"
        - test_plan: "UITests"
```

The build log and <NT>Workflow editor</NT> now displays `"Run unit tests on iPhone"` and `"Run UI tests on iPad"` as the <NT>Step</NT> titles. If the UI tests fail, you'll know immediately without having to inspect the logs.


## Nested <NT>Step bundles</NT>

<NT>Step bundles</NT> allow nesting: you can add a <NT>Step bundle</NT> within another <NT>Step bundle</NT>. This allows you to create large, complex configurations with easily replaceable parts. You can add as many <NT>Step bundles</NT> within another <NT>Step bundles</NT> as you want.

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Open the <NT>Workflow Editor</NT>.
1. On the left, select **<NT>Step bundles</NT>**.
1. Select a <NT>Step bundle</NT> from the dropdown menu in the middle.
1. If it's an empty bundle, click **<NT>Add Step or Step bundle</NT>**.

   If the bundle already has <NT>Steps</NT> in it, hover between <NT>Steps</NT> and click the plus sign.

   ![add-bundle-to-bundle.png](/img/_paligo/uuid-e4db471e-3219-872f-8c6f-7aa194b7f156.png)
1. In the dialog, go to the **<NT>Step bundle</NT>** tab.

   ![bundle-tab.png](/img/_paligo/uuid-68c17140-7ed7-eae8-4101-12d403bee8e4.png)
1. Select a bundle to add it to the original bundle.
1. Use drag-and-drop to move it within the bundle.
1. When done, click **<NT>Save changes</NT>**

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. In the <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file, define the <NT>Step bundle</NT> you want to nest within a `step_bundles` property.

   ```yaml
   step_bundles:
     initialize:
       steps:
         - activate-ssh-key: {}
         - git-clone: {}
   ```
1. Nest this bundle within another one by using the `bundles` property.

   ```yaml
   step_bundles:
     initialize:
       steps:
         - activate-ssh-key: {}
         - git-clone: {}
     deploy_bundle:
       steps:
         - bundle::initialize: {}
         - deploy-to-bitrise-io: {}
   ```

</TabItem>
</Tabs>

## <NT>Step bundle</NT> inputs

In any <NT>Step bundle</NT>, a list of inputs can be defined under the `inputs` key. These are the only inputs you can set for a <NT>Step bundle</NT>: you can't define additional inputs outside the bundle definition.

You can, however, set values for the predefined inputs of a <NT>Step bundle</NT> outside the bundle definition: [Configuring Step bundle inputs](#configuring-step-bundle-inputs).

<NT>Step bundle</NT> inputs share the same model as <NT>Step inputs</NT>. You can read more about <NT>Step inputs</NT> and their format: [Step inputs reference](/en/bitrise-ci/references/steps-reference/step-inputs-reference).

1. In your configuration YAML file, add the `inputs` property to your <NT>Step bundle</NT>.

   ```yaml
   format_version: 11
   step_bundles:
     install_deps:
       inputs:
   ```
1. Under `inputs`, add key-value pairs that will be used for <NT>Step inputs</NT>.

   ```yaml
   format_version: 11
   step_bundles:
     install_deps:
       inputs:     
       - cache_key: "npm-cache-{{ checksum "package-lock.json" }}"
       - npm_command: install
   ```
1. Add the keys to the appropriate <NT>Step inputs</NT>.

   In our example, we configured default inputs for the `key` input of the `restore-cache` <NT>Step</NT> and the `command` input of the `npm` <NT>Step</NT>.

   ```yaml
   format_version: 11
   step_bundles:
     install_deps:
       inputs:     
       - cache_key: "npm-cache-{{ checksum "package-lock.json" }}"
       - npm_command: install
       steps:
       - restore-cache@1:
           inputs:
           - key: $cache_key
       - npm@1: {}
           inputs: 
           - command: $npm_command
   ```

## Configuring <NT>Step bundle</NT> inputs

You define <NT>Step bundle</NT> inputs in the <NT>Step bundle</NT> definition and you can't add <NT>new inputs</NT> outside the definition.

You can set new values for <NT>Step bundle</NT> inputs defined in the <NT>Step bundle</NT> in the <NT>Step</NT> list of the <NT>Workflow</NT>:

1. In your configuration YAML file, find your <NT>Workflow</NT> and the <NT>Step bundle</NT> in its `steps` list.
1. Add an `inputs` property where your <NT>Step bundle</NT> is referenced, and add a key-value pair to it.

   In this example, we set a new value of the `cache_key` input instead of its <NT>default value</NT> defined in the <NT>Step bundle</NT>. When the <NT>Workflow</NT> runs, the `restore-cache` <NT>Step</NT> will use the `npm-cache` value instead of `npm-cache-{{ checksum "package-lock.json" }}`.

   ```yaml
   format_version: 11
   step_bundles:
     install_deps:
       inputs:     
       - cache_key: "npm-cache-{{ checksum "package-lock.json" }}"
       - npm_command: install
       steps:
       - restore-cache@1:
           inputs:
           - key: $cache_key
       - npm@1:
           inputs: 
           - command: $npm_command
   workflows:
     ci:
       steps:
       - git-clone@8: {}
       - bundle::install_deps:
           inputs:
           - cache_key: "npm-cache"
       - deploy-to-bitrise-io@2: {}
   ```

## <NT>Step bundle</NT> outputs

<NT>Steps</NT> can generate output variables. When <NT>Steps</NT> are included in a <NT>Step bundle</NT>, their output variables are available to all <NT>Steps</NT> in the <NT>Workflow</NT> after the <NT>Step bundle</NT>.

For example, if the <NT>Step bundle</NT> used in the <NT>Workflow</NT> uses the `xcode-archive` <NT>Step</NT>, the next <NT>Steps</NT> after the bundle will have access to the BITRISE_IPA_PATH output variable which is exposed by the <NT>Step</NT>.

## <NT>Step Bundle</NT> conditional execution with run_if

Using `run_if` condition with <NT>Step bundles</NT> allows you to conditionally execute entire bundles based on <NT>Environment Variables</NT> or other criteria. When you apply `run_if` directly to <NT>Step bundle</NT> calls in your workflow, the condition is evaluated before the bundle executes. If it evaluates to false, the entire bundle and all its <NT>Steps</NT> are skipped.

The `run_if` syntax works the same as it does for individual <NT>Steps</NT>. All the `run_if` [template expressions and custom template functions](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally#examples-of-run_if-expressions), for example, `getenv`, `enveq`, and `envcontain` are supported for <NT>Steps</NT>.

**<NT>Skip</NT> deployment bundles for draft PRs**

In this example, the `deploy-to-staging` bundle would run only if the PR is not a draft PR. Similarly, the `notify-slack` bundle would run only if it is a successful build.

```yaml
workflows:
  deploy:
    steps:
      - bundle::build-and-test: {}
      - bundle::deploy-to-staging:
          run_if: '{{ not (envcontain "BITRISE_GIT_MESSAGE" "[draft]") }}'
      - bundle::notify-slack:
          run_if: '{{ not .IsBuildFailed }}'
```

1. Open your project's <NT>Workflow Editor</NT>.
1. Click **<NT>Step bundles</NT>** on the left.
1. Select the **<NT>Configuration</NT>** tab and enter a value under **<NT>Additional run conditions</NT>**.

   ![2025_08_15_step_bundle_conditional_execution_runif.png](/img/_paligo/uuid-d84d9f1d-68d7-2706-a7cb-b2023ffe96a5.png)

You can specify a `run_if` in the definition or in the <NT>Step bundle</NT> instance as well:

- When the `run_if` is in the definition, then it serves as the default run_if for all instances.
- When the `run_if` is in the instance, it overrides the definition, and that single instance will have the `run_if`.

  If a <NT>Step bundle</NT> is usually executed conditionally, it is better to define `run_if` on the definition, so that all instances will get the sensible default `run_if` . Like a repo clone and setup bundle that usually only make sense in a CI environment (that's why the git-clone <NT>Step</NT> itself has a `run_if: .IsCI`).

Learn more about how to [enable/disable and set conditions for a Step](/en/bitrise-ci/workflows-and-pipelines/steps/enabling-or-disabling-a-step-conditionally#examples-of-run_if-expressions) with `run_if` expressions.

### Migration

If you currently have identical `run_if` conditions on multiple <NT>Steps</NT> within a bundle, here is how you can streamline your workflow with `run_if` on <NT>Step bundles</NT>.

Before:

```yaml
workflows:
  test:
    steps:
      - bundle::my-test-suite: {}
step_bundles:
  my-test-suite:
    steps:
      - script:
          run_if: '{{ enveq "RUN_TESTS" "true" }}'
          # ...
      - gradle-runner:
          run_if: '{{ enveq "RUN_TESTS" "true" }}'
          # ...
      - deploy-to-bitrise-io:
          run_if: '{{ enveq "RUN_TESTS" "true" }}'
          # ...
```

After:

```yaml
workflows:
  test:
    steps:
      - bundle::my-test-suite:
          run_if: '{{ enveq "RUN_TESTS" "true" }}'
step_bundles:
  my-test-suite:
    steps:
      - script:
          # ...
      - gradle-runner:
          # ...
      - deploy-to-bitrise-io:
          # ...
```
