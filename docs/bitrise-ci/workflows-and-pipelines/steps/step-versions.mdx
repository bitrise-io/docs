<NT>Bitrise</NT> <NT><GlossTerm baseform="Step">Step</GlossTerm></NT> versions follow semantic versioning: a version number looks like MAJOR.MINOR.PATCH. For example, version 3.2.1 is the first patch of the second minor version of the third major version.

You can use any existing version of a <NT>Step</NT> in your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s, and different versions of the same <NT>Step</NT> in different <NT>Workflows</NT>.

We regularly update our <NT>Steps</NT> to make sure they are fully equipped for our users’ needs. However, you don’t have to use the <NT>latest version</NT> if you don’t want to: if an old version is stable and compatible with your build, feel free to continue using that, or roll back to it any time.


## Locking a <NT>Step</NT> to a major or minor version

On the graphical UI of the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT>, you can choose between locking a <NT>Step</NT> to either a major version or a minor version in any of your <NT>Workflows</NT>. This determines what version of the <NT>Step</NT> your <NT>Workflow</NT> will use.

Locking a <NT>Step</NT> to a version means that your <NT>Workflow</NT> is automatically updated to use the latest release of the <NT>Step</NT>’s selected version type, either major or minor, but it won’t get updated if a different major or minor version is released.

- If a <NT>Step</NT> is locked to a major version, it is automatically updated if a new minor version or a new patch for that major version is released. If a new major version is released, it won’t be updated.
- If a <NT>Step</NT> is locked to a minor version, it is automatically updated only if a new patch for that minor version is released. If a new minor or major version is released, it won’t be updated.

**<NT>Version</NT> locking**

The Example <NT>Step</NT>’s current version in the <NT>Workflow</NT> is 2.3.3. A new minor version comes out: 2.4.0.

- If the <NT>Step</NT> is locked to major version 2.x.x, the <NT>Step</NT> is updated to 2.4.0.
- If the <NT>Step</NT> is locked to minor version 2.3.x, the <NT>Step</NT> is NOT updated to 2.4.0.

Now let’s say the Example <NT>Step</NT> gets a new major version: 3.0.0!

In that case, the <NT>Step</NT> will not be automatically updated either way. If you want to use the new version, you need to lock the <NT>Step</NT> to either major version 3.x.x or minor version 3.0.x. This way you can be sure that a <NT>new update</NT> will not break your builds.

To set the update policy for a <NT>Step</NT>:

1. <Partial_OpeningTheWorkflowEditor />
1. Select the <NT>Step</NT>.
1. Go to the **<NT>Properties</NT>** tab.
1. Open the **<NT>Version</NT>** dropdown menu, and select your version.

   ![Step Version dropdown in the Properties tab](/img/workflows-and-pipelines/2026-07-06-step-version-dropdown.png)


## Using a specific <NT>Step</NT> version

You have the option of using a specific, static <NT>Step</NT> version. For example, version 3.2.2. This means that no matter what new versions are released for the <NT>Step</NT>, your <NT>Step</NT> version will not be updated in the <NT>Workflow</NT>.

:::important[YAML mode only!]

Please note that you cannot set a specific <NT>Step</NT> version on the graphical UI: you can only lock the <NT>Step</NT> to either a major or minor version. Setting a specific <NT>Step</NT> version is only possible in YAML mode.

:::

To set a specific <NT>Step</NT> version, you need to add that version to the <NT>Step</NT> reference in your app’s <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file:

1. <Partial_OpeningTheWorkflowEditor />
1. Switch to **YAML** at the top of the <NT>Workflow Editor</NT>.
1. Find the <NT>Workflow</NT> and the <NT>Step</NT> you need.
1. Set the <NT>Step</NT> version as part of the <NT>Step</NT> reference.

   In this example, we're setting the **<NT>Activate</NT> <NT>SSH key</NT>** <NT>Step</NT> of the **primary** <NT>Workflow</NT> to version 4.0.3:

   ```yaml
   workflows:
     primary:
       steps:
       - activate-ssh-key@4.0.3:
   ```


## Using the latest available version of the <NT>Step</NT>

Locking on to the latest release of a <NT>Step</NT> means that if a new version of the <NT>Step</NT> is released, the user’s <NT>Workflow</NT> is updated to use that. This includes automatic update to a new major version, potentially breaking the build. So be careful!

To make sure your <NT>Workflow</NT> will always use the latest available version of a given <NT>Step</NT>, all you have to do is remove any version information from the <NT>Step</NT> reference in the `bitrise.yml` file of your app:

1. <Partial_OpeningTheWorkflowEditor />
1. Switch to **YAML** at the top of the <NT>Workflow Editor</NT>.
1. Find the <NT>Workflow</NT> and the <NT>Step</NT> you need.
1. Make sure the <NT>Step</NT> has no version information:

   In this example, we're setting the **<NT>Activate</NT> <NT>SSH key</NT>** <NT>Step</NT> of the **primary** <NT>Workflow</NT> to the <NT>latest version</NT>:

   ```yaml
   workflows:
     primary:
       steps:
       - activate-ssh-key:
   ```
