Branch-based caching is a legacy technology that <NT>Bitrise</NT> has deprecated. Use the more efficient and flexible Key-based caching.

If your builds are using smaller cache archives than 15 GB then the branch-based caching <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s will keep working but we still recommend migrating the key-based caching as it results in better caching performance in almost all cases.

This guide will help you transition your projects to the key-based caching infrastructure.


## Differences between caching systems

Key-based caching is generally more efficient and far more powerful than branch-based caching. In the table, we compare the two caching methods based on their feature set.

| Features | Branch-based caching | Key-based caching |
| --- | --- | --- |
| **Cache scope** | One cache per branch | Multiple caches identified by keys |
| **Cache sharing across builds** | Limited to same branch and <NT><GlossTerm baseform="stack">stack</GlossTerm></NT> | Can be shared across <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s, branches, and stacks |
| **<NT>Steps</NT> used** | [Cache:Pull](https://bitrise.io/integrations/steps/cache-pull) and [Cache:Push](https://bitrise.io/integrations/steps/cache-push) | - The generic [Restore Cache](https://bitrise.io/integrations/steps/restore-cache) and [Save Cache](https://bitrise.io/integrations/steps/save-cache) <NT>Steps</NT> - [Dedicated caching steps for six dependency managers](/en/bitrise-ci/dependencies-and-caching/key-based-caching/dedicated-caching-steps-for-dependency-managers) |
| **<NT>Expiration</NT>** | Seven days without new build on branch | Seven days since last use |
| **Fallback mechanism** | <NT>Default branch</NT> cache used as fallback | Multiple fallback keys can be configured |
| **Maximum cache archive size** | No limit (until migration stated) | 15 GB |
| **Adding files/paths to the cache archive** | Configured manually or by other <NT>Bitrise steps</NT> supporting caching | Configured manually or via the dedicated caching <NT>Steps</NT> |
| **Ignoring files** | `Ignore paths` parameter | No separate ignore list |
| **Performance/speed** | Slower (larger gzip archives and slower storage backend) | Faster (state-of-the art [zstd](https://github.com/facebook/zstd) compression and faster storage backend) |
| **Dedicated caching <NT>Steps</NT> for dependency managers** | Not available | Available for [most dependency managers](/en/bitrise-ci/dependencies-and-caching/key-based-caching/dedicated-caching-steps-for-dependency-managers) |


## Replacing branch-based caching <NT>Steps</NT>

There are two ways to use key-based caching: either via the generic **<NT>Restore Cache</NT>** and **<NT>Save</NT> Cache** <NT>Steps</NT> or via the dedicated caching <NT>Steps</NT> for specific dependency managers. We strongly recommend using the dedicated <NT>Steps</NT>, unless your dependency manager is not supported or you need some special configuration.

### Using the dedicated caching <NT>Steps</NT>

Dedicated caching <NT>Steps</NT> cache dependencies for a specific dependency manager.

For example, if you run <NT>Gradle</NT> in your build, you can cache <NT>Gradle</NT> dependencies with the dedicated <NT>Steps</NT>. The dedicated <NT>Steps</NT> usually don't need any further configuration: you just add them to your <NT>Workflow</NT>:

:::tip[Full list of dedicated <NT>Steps</NT>]

The procedure below is only an example for using dedicated key-based caching <NT>Steps</NT>. The example uses <NT>Gradle</NT> but we have several dedicated <NT>Steps</NT>. For the detailed list of <NT>Steps</NT>, see: [Dedicated caching Steps for dependency managers](/en/bitrise-ci/dependencies-and-caching/key-based-caching/dedicated-caching-steps-for-dependency-managers).

:::

1. Replace the **Cache:Pull** <NT>Step</NT> with the **<NT>Restore Gradle Cache</NT>** <NT>Step</NT> in your <NT>Workflow</NT>. This <NT>Step</NT> will download your <NT>Gradle</NT> cache.
1. Replace the **Cache:<NT>Push</NT>** <NT>Step</NT> with the **<NT>Save Gradle Cache</NT>** <NT>Step</NT>. This <NT>Step</NT> updates the <NT>Gradle</NT> cache.

All key-based caching <NT>Steps</NT> follow a similar naming convention: the Restore <NT>Steps</NT> download the cache while the Save <NT>Steps</NT> update it. After you ran at least two builds with a key-based setup, you can find and download the cache archive from the **<NT>Project settings</NT>** page: [Accessing key-based cache archives](/en/bitrise-ci/dependencies-and-caching/key-based-caching/accessing-key-based-cache-archives).

### Using the generic key-based caching <NT>Steps</NT>

If the dedicated caching <NT>Steps</NT> don't produce the desired results, consider using the generic **<NT>Restore Cache</NT>** and **<NT>Save</NT> Cache** <NT>Steps</NT>. You can configure these <NT>Steps</NT> to perfectly fit your needs. You can:

- [Use key-based caching templates and functions](/en/bitrise-ci/dependencies-and-caching/key-based-caching/using-key-based-caching#key-based-caching-templates-and-functions).
- [Define multiple keys to match your cache archives](/en/bitrise-ci/dependencies-and-caching/key-based-caching/using-key-based-caching#key-matching-for-cache-archives).
- [Use conditional caching with dynamic keys](/en/bitrise-ci/dependencies-and-caching/key-based-caching/using-key-based-caching#using-conditional-caching-with-dynamic-keys).

Check out our key-based cache recipes utilising the generic <NT>Steps</NT>: [Advanced key-based cache recipes](https://github.com/bitrise-io/workflow-recipes/blob/main/recipes/key-cache-advanced.md).

:::
