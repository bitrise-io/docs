iOS <NT>code signing</NT> can be complicated, with several potential issues. We've collected some of the most common issues and their potential solutions. Let’s look into what you can do to make sure <NT>code signing</NT> works!


## <NT>Code signing</NT> and clean virtual machines

Every <NT>Bitrise</NT> build uses a clean virtual machine with no leftover files and configuration from previous builds. While you most likely have all the necessary <NT>code signing</NT> files (certificates and provisioning profiles) available on your local machine this is not the case with our virtual machines. They do not contain any <NT>code signing</NT> files relevant to your project, that is why you have to make sure:

- You choose the right <NT>code signing</NT> asset management option to upload the provisioning profiles to <NT>Bitrise</NT>. From here our <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s will download the files to the virtual machine at runtime.
- You upload the necessary <NT>signing certificate</NT>s to the **<NT>Code Signing</NT>** tab.
- <NT>You have</NT> connected your <NT>Bitrise</NT> account to the <NT>Apple Developer portal</NT>. (This step is not needed if you use the **Certificate and Profile Installer** <NT>Step</NT>.) to <NT>Apple Developer portal</NT>.

  If you suspect an error is related to <NT>code signing</NT>, there is almost certainly a problem with one of these three. When trying to build an iOS app on <NT>Bitrise</NT>, we strongly recommend generating an .ipa file of the app locally, on your own machine first. If that fails, the build will certainly fail on <NT>Bitrise</NT>, too.

  :::tip[Verbose logs]

  If you are getting any issues, make sure you enable the Verbose log input in your <NT>code signing</NT> asset management or building <NT>Step</NT> to get more information on the nature of the issue.

  :::


## iOS <NT>code signing</NT> <NT>Steps</NT> fail

Our [iOS code signing Steps](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) manage your provisioning profiles for you: they download the profiles from the <NT>Apple Developer portal</NT> and install them for you. Here’s what you can do if these <NT>Steps</NT> fail:

- Before trying to use this <NT>Step</NT>, generate an .ipa file locally - with the same export method you want to use on <NT>Bitrise</NT> - to ensure that the profiles are uploaded to the <NT>Apple Developer portal</NT>.
- Make sure that your <NT>Bitrise</NT> account is connected to the <NT>Apple Developer portal</NT> and that you have at least an Admin role in your <NT>Apple</NT> Developer team.


## File count limit on provisioning profiles

The maximum number of provisioning profiles and .p12 certificates you can upload to the **<NT>Code Signing</NT>** tab on <NT>Bitrise</NT> is 100. If you’ve already reached this limit and wish to use even more, then here are a few tips on how to use even more provisioning profiles:

- Use a <NT>Step</NT> with automatic provisioning, such as **<NT>Xcode Archive & Export for iOS</NT>**, **<NT>Export iOS and tvOS Xcode Archive</NT>**, **<NT>Xcode</NT> build for testing for iOS** , or **<NT>Manage iOS Code Signing</NT>**. These <NT>Steps</NT> only require the <NT>code signing</NT> identities (certificates with .p12 extension) to be uploaded to <NT>Bitrise</NT>. You can download the provisioning profiles from the <NT>Apple Developer portal</NT> on-the-fly during the build if you have [connected your Apple Developer account to Bitrise](/en/bitrise-platform/integrations/apple-services-connection/connecting-to-an-apple-service-with-api-key).
- You can create a .zip file containing the required certificates/profiles. In this case, you don’t need to upload any certificates/profiles on <NT>Bitrise</NT>. During the build you can download the .zip file and update the certificate/profile related inputs of the **Certificate and Profile Installer** <NT>Step</NT> to match the path to the certificate/profile on the <NT>build machine</NT>. Note that the **Certificate and Profile Installer** <NT>Step</NT> supports local paths and URLs for certificates and profiles.
- You can host the profiles and certificates yourself, and only add an URL that is pointing to a given certificate/profile to the workflow.

Also note that multiple URLs can be specified for both the certificate and profile inputs. Make sure you separate them with a pipe (`|`) character.


## Could not install the app on a device

To install iOS apps on a given device, you have to either:

- Export an IPA file with the `development` export method, with the device’s UDID registered in the provisioning profile used for the export.
- Export an IPA file with the `ad-hoc` export method and install the app via the public install page generated by the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT>.

If you can't install an app on a given device:

1. Check that the device UDID is included in the app’s provisioning profile.

   If you use manual provisioning, check the provisioning profile you uploaded to <NT>Bitrise</NT>.
1. Check the logs to see if the **<NT>Xcode Archive & Export for iOS</NT>** <NT>Step</NT> used the provisioning profile with the device’s UDID in it.
