Creating an <NT>Android</NT> App Bundle with <NT>Bitrise</NT> is almost the same as generating an APK. All you have to do is tweaking a few <NT><GlossTerm baseform="Step">Step</GlossTerm></NT> inputs to compile an <NT>Android</NT> App Bundle (.aab) file from your code, then get the bundle signed and deployed to <NT>Google Play Store</NT>.

:::important[<NT>Step</NT> versions supporting bundle creation]

The following <NT>Steps</NT> must be of the indicated version or newer - older versions of the <NT>Steps</NT> do NOT support bundle creation.

- [**Android Build**](https://github.com/bitrise-steplib/bitrise-step-android-build) 0.10.0 or newer
- [**Gradle Runner**](https://github.com/bitrise-io/steps-gradle-runner) 1.9.0 or newer
- [**Android Sign**](https://github.com/bitrise-steplib/steps-sign-apk) 1.3.0 or newer
- **<NT>Deploy to Google Play</NT>** 1.6.0 or newer

:::


## Generating an <NT>Android</NT> App Bundle file

You can create an <NT>Android</NT> App Bundle with either the **<NT>Gradle Runner</NT>** <NT>Step</NT> or with the **<NT>Android Build</NT>** <NT>Step</NT>.

### Using the <NT>Gradle Runner</NT> <NT>Step</NT>

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. <Partial_OpeningTheWorkflowEditor />
1. Insert the **<NT>Gradle Runner</NT>** <NT>Step</NT> after the **<NT>Android Unit Test</NT>** and **<NT>Android Lint</NT>** <NT>Steps</NT> in your <NT>Workflow</NT>.
1. Click the **Config** section of **<NT>Gradle Runner</NT>**.
1. In the **<NT>Gradle task to run</NT>** input field, set, for example, `bundleRelease` or `bundleDebug` to create a bundle of your project.

   ![Generating_and_deploying_Android_app_bundles.jpg](/img/_paligo/uuid-b0a21607-f817-bdff-5480-e82a6f9a0152.jpg)

   :::tip[APK and AAB in the same <NT>Workflow</NT>]

   If you wish to generate an <NT>Android</NT> App Bundle and an APK in one <NT>Workflow</NT>, you can specify an additional task in the **<NT>Gradle task to run</NT>** input field: set the input value to `bundleRelease assembleRelease` to generate release versions.

   :::

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Open your app's <NT><GlossTerm baseform="configuration yaml">Configuration YAML</GlossTerm></NT> file.
1. Insert the `gradle-runner` <NT>Step</NT> after the `android-unit-test` and `android-lint` <NT>Steps</NT> in your <NT>Workflow</NT>.

   ```yaml
   my-workflow:
     steps:
       - activate-ssh-key:
           run_if: '{{getenv "SSH_RSA_PRIVATE_KEY" | ne ""}}'
       - git-clone: {}
       - install-missing-android-tools: {}
       - android-lint: {}
       - android-unit-test: {}
       - gradle-runner:
   ```
1. In the `gradle_task` input of `gradle-runner`, set, for example, `bundleRelease` or `bundleDebug` to create a bundle of your project.

   ```yaml
   my-workflow:
     steps:
       - activate-ssh-key:
           run_if: '{{getenv "SSH_RSA_PRIVATE_KEY" | ne ""}}'
       - git-clone: {}
       - install-missing-android-tools: {}
       - android-lint: {}
       - android-unit-test: {}
       - gradle-runner:
           inputs:
             - gradle_task: bundleRelease
   ```

   :::tip[APK and AAB in the same <NT>Workflow</NT>]

   If you wish to generate an <NT>Android</NT> App Bundle and an APK in one <NT>Workflow</NT>, you can specify an additional task in the `gradle_task` input field: set the input value to `bundleRelease assembleRelease` to generate release versions.

   :::

</TabItem>
</Tabs>

This way the <NT>Step</NT> will generate an <NT>Android</NT> App Bundle instead of an APK.

### Using the <NT>Android Build</NT> <NT>Step</NT>

You can generate an <NT>Android</NT> App Bundle for your <NT>Android</NT> app with our **<NT>Android Build</NT>** <NT>Step</NT> as well:

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. <Partial_OpeningTheWorkflowEditor />
1. Add the **<NT>Android Build</NT>** <NT>Step</NT> after the **<NT>Android Unit Test</NT>** and **<NT>Android Lint</NT>** <NT>Steps</NT> in your <NT>Workflow</NT>.
1. Provide the root directory of your <NT>Android</NT> project in the **<NT>Project Location</NT>** input field.
1. Go to **<NT>Build</NT> type** and select `aab` as build type.

   ![Generating_and_deploying_Android_app_bundles.jpg](/img/_paligo/uuid-e786c907-9370-70be-0d2f-7c9c6f028ac7.jpg)

   :::tip[APK and AAB in the same <NT>Workflow</NT>]

   If you wish to generate an APK and an <NT>Android</NT> App Bundle in one <NT>Workflow</NT>, add two **<NT>Android Build</NT>** <NT>Steps</NT> after each other and configure one to build an <NT>Android</NT> App Bundle and the other to build an APK.

   :::

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Open your app's Configuration YAML.
1. Add the `android-build` <NT>Step</NT> after the `android-unit-test` and `android-lint` <NT>Steps</NT> in your <NT>Workflow</NT>.

   ```yaml
   my-workflow:
     steps:
       - activate-ssh-key:
           run_if: '{{getenv "SSH_RSA_PRIVATE_KEY" | ne ""}}'
       - git-clone: {}
       - install-missing-android-tools: {}
       - android-lint: {}
       - android-unit-test: {}
       - android-build:
   ```
1. Provide the root directory of your <NT>Android</NT> project in the `project_location` input field.
1. Set the value of the `build_type` input to `aab`.

   ```yaml
   my-workflow:
     steps:
       - activate-ssh-key:
           run_if: '{{getenv "SSH_RSA_PRIVATE_KEY" | ne ""}}'
       - git-clone: {}
       - install-missing-android-tools: {}
       - android-lint: {}
       - android-unit-test: {}
       - android-build:
           inputs:
             - build_type: aab
   ```

   :::tip[APK and AAB in the same <NT>Workflow</NT>]

   If you wish to generate an APK and an <NT>Android</NT> App Bundle in one <NT>Workflow</NT>, add two `android-build` <NT>Steps</NT> after each other and configure one to build an <NT>Android</NT> App Bundle and the other to build an APK.

   :::

</TabItem>
</Tabs>


## Signing an <NT>Android</NT> App bundle

Signing an <NT>Android</NT> App Bundle file works the same way as signing an APK: the most convenient method is to upload your keystore files to <NT>Bitrise</NT> and use the **<NT>Android Sign</NT>** <NT>Step</NT>:

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Upload your keystore file to <NT>Bitrise</NT>.
1. Open your <NT>Workflow</NT> in the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT>, and add the **<NT>Android Sign</NT>** <NT>Step</NT> AFTER the build <NT>Step</NT>.
1. Make sure that the **<NT>Keystore password</NT>**, **<NT>Key alias</NT>** and the **<NT>Key</NT> password** input fields are filled out.

   :::tip[Default input values]

   If you have uploaded your keystore file and filled out the required credentials, the **<NT>Android Sign</NT>** <NT>Step</NT>'s **<NT>Keystore url</NT>**, **<NT>Keystore password</NT>**, **<NT>Keystore alias</NT>**, and the **<NT>Private key password</NT>** inputs will get populated automatically!

   :::
1. Make sure the **<NT>App</NT> file path** input field displays the same output env var as the output of the build <NT>Step</NT> you've been using.

   For example, **<NT>Android Build</NT>** exports either a $BITRISE_APK_PATH or a $BITRISE_AAB_PATH Env Var that points to the APK and/or AAB file the <NT>Step</NT> generated. By default, this input points at these Env Vars.

   ![Generating_and_deploying_Android_app_bundles.jpg](/img/_paligo/uuid-8b24a99b-a80b-f23a-7708-af1136ffbedb.jpg)

   :::tip[Further configuration options]

   Check out all the available configuration options of the **<NT>Android Sign</NT>** <NT>Step</NT> in the <NT>Workflow Editor</NT>. You can:

   - Enable or disable memory page alignment with the **Page alignment** input.
   - Choose which tool signs the app with the **Signer tool** input: `automatic` (default, uses `apksigner` for APKs and `jarsigner` for AABs), `apksigner`, or `jarsigner`.
   - Enforce a specific [Signature Scheme](https://source.android.com/docs/security/features/apksigning#schemes) with the **APK Signature Scheme** input.

   :::

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Upload your keystore file to <NT>Bitrise</NT>.
1. In your app's Configuration YAML file, add the `sign-apk` <NT>Step</NT> AFTER the build <NT>Step</NT> - for example, `android-build` - in your <NT>Workflow</NT>.

   ```yaml
   sign-android-workflow:
     steps:
       - android-build: {}
       - sign-apk@1:
           inputs:
   ```
1. Make sure that the `keystore_url`, `keystore_password`, and `keystore_alias` inputs point to the correct location.

   ```yaml
   sign-android-workflow:
     steps:
       - android-build: {}
       - sign-apk@:
           inputs:
           - keystore_url: "$BITRISEIO_ANDROID_KEYSTORE_URL"
           - keystore_password: "$BITRISEIO_ANDROID_KEYSTORE_PASSWORD"
           - keystore_alias: "$BITRISEIO_ANDROID_KEYSTORE_ALIAS"
   ```

   :::tip[Default input values]

   If you have uploaded your keystore file to <NT>Bitrise</NT> and filled out the required credentials, you do not have to set the inputs at all: the <NT>default value</NT>s, defined in the <NT>Step</NT>'s `step.yml` configuration file, will point to the keystore file and the necessary credentials.

   :::
1. Make sure the `android_app` input field displays the same output Env Var as the output of the build <NT>Step</NT> you've been using.

   For example, `android-build` exports either a $BITRISE_APK_PATH or a $BITRISE_AAB_PATH Env Var that points to the APK and/or AAB file the <NT>Step</NT> generated. By default, this input points at these Env Vars.

   The <NT>Step</NT> will look for a binary to sign at the locations provided in this input.

   ```yaml
   sign-android-workflow:
     steps:
       - android-build: {}
       - sign-apk@1:
           inputs:
           - keystore_url: "$BITRISEIO_ANDROID_KEYSTORE_URL"
           - keystore_password: "$BITRISEIO_ANDROID_KEYSTORE_PASSWORD"
           - keystore_alias: "$BITRISEIO_ANDROID_KEYSTORE_ALIAS"
           - android_app: "$BITRISE_APK_PATH\\n$BITRISE_AAB_PATH"
   ```

   :::tip[Further configuration options]

   Check out all the available configuration options of the `android-sign` <NT>Step</NT> in [its step.yml file](https://github.com/bitrise-steplib/steps-sign-apk/blob/master/step.yml). You can:

   - Enable or disable memory page alignment with the `page_align` input.
   - Choose which tool signs the app with the `signer_tool` input: `automatic` (default, uses `apksigner` for APKs and `jarsigner` for AABs), `apksigner`, or `jarsigner`.
   - Enforce a specific [Signature Scheme](https://source.android.com/docs/security/features/apksigning#schemes) with the `signer_scheme` input.

   :::

</TabItem>
</Tabs>


## Deploying your <NT>Android</NT> App Bundle to <NT>Google Play</NT>

Deploying an AAB file isn't significantly different from deploying an APK to <NT>Google Play</NT>.

If you want to check the bundle prior to app store distribution, you can add the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> after the **<NT>Gradle Runner</NT> / <NT>Android Build</NT>** <NT>Steps</NT>. It uploads the bundle into the **<NT>Artifacts</NT>** tab of your <NT>Build</NT>’s page.

### Setting up <NT>Google Play</NT> deployment for the first time

Deploying to <NT>Google play</NT> publishes your app to Google's online store. When you do it for the first time, this requires a bit more work than simply deploying to [bitrise.io](https://www.bitrise.io/). Once the necessary configurations are in place, it becomes very simple.

When configuring <NT>Google Play</NT> deployment for the first time, you need to link your <NT>Google Play</NT> Developer account to an API project, set up API access, and upload the service account JSON key to <NT>Bitrise</NT>.

1. Upload the first AAB or APK manually to <NT>Google Play</NT> [using the Google Play Console](https://support.google.com/googleplay/android-developer/answer/113469?hl=en).
1. [Link](https://developers.google.com/android-publisher/getting_started) your <NT>Google Play</NT> Developer Console to an API project.
1. [Set up API Access Clients using a service account](https://developers.google.com/android-publisher/getting_started): Please note when you create your service account on the Google Developer Console, you have to choose `json` as **<NT>Key</NT> <NT>Type</NT>**.
1. Grant the necessary rights to the service account with your [Google Play Console](https://play.google.com/apps/publish). Go to **<NT>Settings</NT>**, then **<NT>Users & permissions</NT>**, then **<NT>Invite new user</NT>**. Due to the way the <NT>Google Play</NT> Publisher API works, you have to grant at least the following permissions to the service account:

   - Access level: View app information.
   - <NT>Release management</NT>: Manage production releases, manage testing track releases.
   - Store presence: Edit store listing, pricing & distribution.
1. As an optional step, you can add translations to your Store Listing: [Translate & localize your app](https://support.google.com/googleplay/android-developer/answer/3125566?hl=en).
1. [Connect your Google service account to Bitrise](/en/bitrise-platform/integrations/connecting-a-google-service-account-to-bitrise).

### Deploying to <NT>Google Play</NT>

Deploying to <NT>Google Play</NT> requires [a signed APK or AAB file](/en/bitrise-ci/code-signing/android-code-signing/android-code-signing-using-the-android-sign-step) and the **[Google Play Deploy](https://www.bitrise.io/integrations/steps/google-play-deploy)** <NT>Step</NT>.

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. [Upload your service account JSON key to the **Files** section](/en/bitrise-ci/run-and-analyze-builds/managing-build-files/uploading-files-for-your-builds).
1. In the **Files** section, copy the **<NT>Download URL</NT>** of your service account file.

   For example, `BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL`.
1. Open the <NT>Workflow Editor</NT> and select **<NT><GlossTerm baseform="secret">Secret</GlossTerm></NT>s**.
1. Create a Secret with the copied download URL as the value.

   If you uploaded the JSON key file to <NT>Bitrise</NT>, the download URL is an Environment Variable so check the **<NT>Replace variables in inputs</NT>** checkbox.

   :::note[Direct link]

   If you use a direct link to your keystore file without uploading it to <NT>Bitrise</NT>, you don't need to check the **<NT>Replace</NT> the variables in inputs** option.

   :::
1. Add the **<NT>Google Play Deploy</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. In the **<NT>Service Account JSON key file path</NT>** input, paste the Secret you created.
1. In the **<NT>Package name</NT>** input, add the <NT>package name</NT> of your app.
1. In the **Track** input, add the track where you want to deploy your APK (for example, alpha/beta/rollout/production or any custom track you set).

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. [Upload your service account JSON key to the **Files** section](/en/bitrise-ci/run-and-analyze-builds/managing-build-files/uploading-files-for-your-builds).

   :::note[Uploading the service account JSON key file]

   We recommend uploading the service account JSON key to <NT>Bitrise</NT> but it is not mandatory: you can store it elsewhere and provide a direct link to it.

   :::
1. Open your app's Configuration YAML file and add the `google-play-deploy` <NT>Step</NT> to it.

   ```yaml
   deploy-workflow:
     steps: 
     - google-play-deploy: 
         inputs:
   ```
1. In the `service_account_json_key_path` input, you need to provide the path to the service account JSON key file.

   [Create a Secret](/en/bitrise-ci/configure-builds/secrets) to the path and reference that here.

   ```yaml
   deploy-workflow:
     steps: 
     - google-play-deploy: 
         inputs:   
         - service_account_json_key_path: "$BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL"
   ```
1. In the `package` input, add [the package name](https://support.google.com/admob/answer/9972781?hl=en#:~:text=The%20package%20name%20of%20an,supported%20third%2Dparty%20Android%20stores.) of your app.

   ```yaml
   deploy-workflow:
     steps: 
     - google-play-deploy: 
         inputs:   
         - service_account_json_key_path: "$BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL"
         - package_name: myApp
   ```
1. In the `track` input, add the track where you want to deploy your app binary (for example, alpha/beta/rollout/production or any custom track you set).

   ```yaml
   deploy-workflow:
     steps: 
     - google-play-deploy: 
         inputs:   
         - service_account_json_key_path: "$BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL"
         - package_name: myApp
         - track: alpha
   ```

</TabItem>
</Tabs>

That’s all! Start or schedule a build and share the URL with external testers or distribute your app on an app store of your choice!
