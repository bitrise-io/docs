Visual Studio <NT>App Center</NT> has been taken offline on March 31, 2025. This guide is meant to help you migrating your projects from <NT>App Center</NT> to <NT>Bitrise</NT>.

We strongly recommend going through our [Getting started](/en/bitrise-build-cache/getting-started-with-the-build-cache/getting-started-with-the-build-cache) guide. The [Key Bitrise concepts](/en/bitrise-ci/getting-started/key-bitrise-concepts) document can also help understanding how <NT>Bitrise</NT> works.

In this guide, we'll be going through the similarities and differences between <NT>Bitrise</NT> and <NT>App Center</NT>.


## <NT>App Center</NT> orgs and <NT>Bitrise</NT> <NT>Workspaces</NT>

On <NT>App Center</NT>, your user account can own apps. Organizations are optional: <NT>App Center</NT> recommends creating an organization for any apps with multiple collaborators.

A [Bitrise Workspace](/en/bitrise-platform/workspaces/workspaces-overview) is a little different: when you sign up for the first time, we automatically create your first <NT>Workspace</NT>, too. This is because only <NT>Workspaces</NT> can own <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>s. Projects are not tied to your account but to <NT>Workspaces</NT>. A <NT>Workspace</NT> can own several projects but a project cannot be linked to multiple <NT>Workspaces</NT>.

A <NT>Bitrise</NT> <NT>Workspace</NT> has four main roles:

- **<NT>Owner</NT>**: The owner of the <NT>Workspace</NT>. Full administrative control over the <NT>Workspace</NT> without restrictions. A <NT>Workspace</NT> can have multiple Owners. The default Owner is the account that created the <NT>Workspace</NT>.
- **<NT>Manager</NT>**: The user can access and modify <NT>Workspace settings</NT> such as connected service accounts, can manage members but can't access billing details and can't delete the <NT>Workspace</NT>.
- **<NT>Contributor</NT>**: Can create new CI projects and add <NT>Release Management</NT> apps.
- **<NT>Viewer</NT>**: Can only view selected <NT>Workspace settings</NT> (Projects and Integrations).

On <NT>App Center</NT>, you can create teams within organizations. On <NT>Bitrise</NT>, <NT>Workspace</NT> members can be added to <NT>Workspace</NT> groups: this makes it easier to assign multiple people to projects at the same time.


## User roles and collaboration

<NT>App Center</NT> offers three different roles to manage access to your apps. On <NT>Bitrise</NT>, control is a little more granular: on each project's team, you can set five different roles to make sure that your team members have the exact right access to the project.

For more information, check out [Roles and permissions for Bitrise CI](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).


## <NT>App Center</NT> Build service and <NT>Bitrise</NT> builds

The <NT>App Center</NT> Build service helps you build your apps using a secure cloud infrastructure. A <NT>Bitrise</NT> build works the same way: you can [connect a repository](/en/bitrise-platform/repository-access/repository-access-with-oauth) to our service and build your project every time a code event happens in the repository.

<NT>Bitrise</NT> supports multiple mobile frameworks, including iOS, <NT>Android</NT>, <NT>React Native</NT>, Flutter, <NT>Ionic</NT>, and <NT>Cordova</NT>. When [adding a new project](/en/bitrise-ci/getting-started/adding-a-new-project), we automatically detect your framework and set up a configuration best suited for that particular framework.

### Repository connections

<NT>Bitrise</NT> supports connections to multiple code repository services:

- <NT>GitHub</NT> (including the <NT>GitHub</NT> app and <NT>GitHub Enterprise Server</NT>)
- <NT>GitLab</NT> (including self-hosted <NT>GitLab</NT>)
- <NT>Bitbucket</NT> (including <NT>Bitbucket Server</NT>)
- Any other Git repository via a generic Git connection

For most cases, you can use OAuth connections and [SSH keys](/en/bitrise-platform/repository-access/configuring-ssh-keys) to set up repository access. [HTTPS authorization](/en/bitrise-platform/repository-access/configuring-https-authorization-credentials) is also supported.

### <NT>GitHub</NT> App

Just like <NT>App Center</NT>, <NT>Bitrise</NT> offers a [GitHub App](/en/bitrise-platform/repository-access/github-app-integration) to integrate your <NT>Bitrise</NT> <NT>Workspace</NT> to a <NT>GitHub</NT> account or organization. With the <NT>GitHub</NT> App, you can:

- Connect <NT>Bitrise</NT> to <NT>GitHub</NT> without using an SSH key.
- Improve code security: the app relies on short-lived one-time tokens for access.
- [Link multiple repositories](/en/bitrise-platform/repository-access/github-app-integration#additional-linked-repositories-via-a-github-app) to the same project.
- Report your build status to <NT>GitHub</NT>.

### Configuring your build

On <NT>App Center</NT>, your build configuration is tied to the branches of your repository. On <NT>Bitrise</NT>, these are separate: the build configuration is independent of the branches of the repository. The process generally works like this:

1. You set up a build configuration: create [Workflows and Pipelines](/en/bitrise-ci/workflows-and-pipelines/workflows/workflows-overview) and configure the <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s inside them. Your configuration is stored in YAML format but you can create and modify it using the graphical <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT>. You can also set up [Environment Variables](/en/bitrise-ci/configure-builds/environment-variables) on the project level or on <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT> level.
1. You create [automatic build triggers](/en/bitrise-ci/run-and-analyze-builds/build-triggers/configuring-build-triggers): define the code events that should automatically start <NT>Bitrise</NT> builds. In addition to simple code push, you can create triggers for pull requests and Git tags. And you have further, more granular control: for example, you can configure a trigger that only starts a build if a certain file has changed in a pull request. You can also specify a branch or branches for a trigger so that only code events on the specified branch trigger a build.
1. You select the [stack](/en/bitrise-build-hub/infrastructure/build-stacks/about-build-stacks) and the [machine type](/en/bitrise-build-hub/infrastructure/build-machine-types) of your build.
1. On the **<NT>Project settings</NT>** page, you can configure additional options for your builds:

   - <NT>Set up</NT> [build notifications](/en/bitrise-ci/configure-builds/configuring-build-settings/configuring-email-notifications).
   - <NT>Set up</NT> [rolling builds](/en/bitrise-ci/configure-builds/configuring-build-settings/rolling-builds).
   - Configure [connection to services](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services) such as <NT>App Store Connect</NT> or <NT>Google Play</NT> to ensure your builds can can always access them.
   - Upload [code signing](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) files or any other files that your builds need.
   - Configure or download your [build cache](/en/bitrise-ci/dependencies-and-caching/key-based-caching/accessing-key-based-cache-archives).

### Running a build

Just like on <NT>App Center</NT>, every <NT>Bitrise</NT> build runs in a clean virtual machine that is discarded once the build is finished while generated artifacts are stored on our servers. On <NT>Bitrise</NT>, you can also select your build stack: for example, you can always choose between several different versions of <NT>Xcode</NT>. And we're always aiming to make new <NT>Xcode</NT> versions available as soon as possible.

You can [start builds manually](/en/bitrise-ci/run-and-analyze-builds/starting-builds/starting-builds-manually), trigger them automatically, or [schedule them](/en/bitrise-ci/run-and-analyze-builds/starting-builds/scheduling-builds). Once a build has run, you can check out detailed, structured [build logs](/en/bitrise-ci/run-and-analyze-builds/build-data-and-troubleshooting/build-logs).


## Testing on <NT>App Center</NT> and on <NT>Bitrise</NT>

<NT>App Center</NT> Test is a test automation service for mobile apps: you can upload your app binary and test files to execute tests.

<NT>Bitrise</NT> offers everything <NT>App Center</NT> Test does in our CI/CD service: we have multiple, platform-specific dedicated testing <NT>Steps</NT> that can find and run tests within your code. You can run your tests in simulators and on real devices. In addition to the dedicated testing <NT>Steps</NT>, you can run any script you want, so you can fully customize your tests.

Just like building and deploying, testing is also automatic: set up build triggers so that code events trigger builds with tests. Everything can be integrated: you can build, test, and deploy your app within the very same <NT>Workflow</NT>, if you choose to.

You can view all your test results and generated test artifacts in one place: [Deploying and viewing test results](/en/bitrise-ci/testing/deploying-and-viewing-test-results).

:::note[Test distribution solution]

An enhanced <NT>Release Management</NT> solution with test distribution features is coming soon. You will be able to:

- Easily access and distribute different versions of your app for testing from a directory-like page for IPA, APK, and AAB files.
- Share your app via either a public or a private install page, or distribute builds to your testers early, in the pre-production phase.

:::


## <NT>App Center</NT> Distribute and <NT>Bitrise</NT> <NT>Release Management</NT>

The <NT>App Center</NT> Distribute service allows you to manage app distribution across multiple platforms in one place. <NT>Bitrise</NT> [Release Management](/en/release-management) offers exactly that: a one-stop shop for your release requirements. You can:

- Automate all your releases.
- Release your apps across multiple platforms at the same time.
- Have full granular control over the entire <NT>release process</NT> without leaving <NT>Release Management</NT>.

To get started, you just need to [connect an app](/en/release-management/getting-started-with-release-management/connecting-an-app). Once you successfully connected an app, you don't have to go through any of the online stores to manage your releases.

We also offer [a fully featured REST API](/en/release-management/release-management-api) to make the most of <NT>Release Management</NT>.
