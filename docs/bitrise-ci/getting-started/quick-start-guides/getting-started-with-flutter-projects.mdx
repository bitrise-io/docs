Flutter is a mobile app SDK that allows developers to create native apps for both iOS and <NT>Android</NT>. [Bitrise](https://app.bitrise.io/users/sign_in) supports Flutter apps: we have dedicated <NT><GlossTerm baseform="Step">Step</GlossTerm></NT>s to help you with all your Flutter needs. This guide walks you through setting up, testing, building and deploying a simple Flutter <NT><GlossTerm baseform="Project">project</GlossTerm></NT> on <NT>Bitrise</NT>.


## Adding a Flutter project to <NT>Bitrise</NT>

Add your project the same way as any other: see [Adding a new project](/en/bitrise-ci/getting-started/adding-a-new-project) for the full walkthrough of the wizard. This section covers what's specific to Flutter.

### How the scanner detects a Flutter project

The <NT><GlossTerm baseform="project scanner">project scanner</GlossTerm></NT> looks for a `pubspec.yaml` file anywhere in your repository, excluding `node_modules`. Every directory that has one is a candidate Flutter project.

For each candidate, the scanner also checks for:

- A `test/` directory with at least one `*_test.dart` file, to enable testing.
- An `ios/Runner.xcworkspace` directory, to enable iOS builds.
- An `android/build.gradle` or `android/build.gradle.kts` file, to enable <NT>Android</NT> builds.
- A `web/` directory, to enable web builds.

It suppresses the standalone iOS, macOS, and <NT>Android</NT> scanners for the same project, since the generated Flutter <NT>Workflow</NT> already handles those platforms.

### What you can configure

During the **<NT>CI configuration</NT>** stage, the wizard shows a single option: **<NT>Project location</NT>**, the directory that contains your `pubspec.yaml` file. Everything else, including which platforms to build, is detected automatically and baked into the generated <NT>Workflow</NT>.

## Testing a Flutter app

You can write and run unit-, widget-, and integration tests with Flutter. For more information, check out [Flutter’s official documentation](https://flutter.io/docs/testing).

You can use our automatically generated **primary** <NT>Workflow</NT> to test your Flutter app. It can include the **<NT>Flutter Test</NT>** <NT>Step</NT> that runs Flutter tests and can also generate code coverage reports.

:::note[Test reports]

The **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> exports the results of the **<NT>Flutter Test</NT>** <NT>Step</NT> to **<NT>Tests</NT>** tab by default.

:::

To run tests on a Flutter app:

1. Open your app’s <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> and open the **primary** <NT>Workflow</NT>, or any of your <NT>Workflows</NT> that you want to use to run tests.
1. In the **<NT>Flutter Install</NT>** <NT>Step</NT>, fill in the **<NT>Flutter SDK version or bundle URL</NT>** input.

   You can specify either tags or branches of the Flutter SDK’s git repository. The <NT>default value</NT> is `stable`. This will use the latest stable branch of Flutter.

   - To find the available version tags, check: [https://github.com/flutter/flutter/releases](https://github.com/flutter/flutter/releases).
   - To see the the available branches, check: [https://github.com/flutter/flutter/branches](https://github.com/flutter/flutter/branches).
1. Add the **<NT>Flutter Test</NT>** <NT>Step</NT>. In the **<NT>Additional parameters</NT>** input, enter any flags you wish to use to.

   The <NT>Step</NT> runs the `flutter test` command with the specified flags. To check the available flags, open a command line interface on your own machine and run `flutter test --help`.

   ![Getting_started_with_Flutter_apps.png](/img/_paligo/uuid-3c8c65d8-1de1-0a50-62cf-acf76dc76bf9.png)
1. Make sure the **<NT>Project Location</NT>** input of the **<NT>Flutter Test</NT>** <NT>Step</NT> is correct.

   The <NT>default value</NT> is the the Environment Variable (Env Var) created for your Flutter project’s location.
1. If you want to generate code coverage reports, set the **Generate code coverage files?** input to `yes`.

   This runs the `flutter test` command with the `--coverage` flag.
1. To export the test results to the **<NT>Tests</NT>** tab, add the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> to the end of your <NT>Workflow</NT>.

   For more information, check out [Deploying and viewing test results](/en/bitrise-ci/testing/deploying-and-viewing-test-results).


## Deploying a Flutter app

To build and deploy a Flutter app, a <NT>Workflow</NT> must contain these <NT>Flutter</NT> <NT>Steps</NT>:

- **<NT>Flutter Install</NT>**
- **<NT>Flutter Build</NT>**

If you have platforms specified in your repository, a deployment <NT>Workflow</NT> will be automatically generated when adding the project on <NT>Bitrise</NT>. The content of this <NT>Workflow</NT> depends on the platforms: for example, if your Flutter project contains only an iOS project, the <NT>Workflow</NT> will contain the **<NT>Certificate and profile installer</NT>** <NT>Step</NT>.

You can build both iOS and <NT>Android</NT> projects at the same time or you can build them separately, each using their own <NT>Workflow</NT>. You can set this in the **<NT>Platform</NT>** input of the **<NT>Flutter Build</NT>** <NT>Step</NT> any time. By default, the <NT>Step</NT> is configured according to the platform or platforms that the scanner detected when adding the app on <NT>Bitrise</NT>.

In these examples, we're assuming that you are building and deploying both the iOS and <NT>Android</NT> versions of your app in the same <NT>Workflow</NT>, performing the necessary operations serially. However, you can do both versions in parallel with a single build <NT><GlossTerm baseform="trigger">trigger</GlossTerm></NT> by using <NT>Pipelines</NT>:

:::

### Deploying a cross-platform app to bitrise.io

The **<NT>Deploy to bitrise.io</NT>** <NT>Step</NT> uploads all the artifacts related to your build into the [**Artifacts**](/en/bitrise-ci/run-and-analyze-builds/managing-build-files/build-artifacts-online) tab on your <NT>Build</NT>’s page.

You can share the generated binary file (APK for <NT>Android</NT> or an IPA file for iOS) with your team members using the public install page. The public install page is a URL you can share with others who can install the generated app binary on their device. You can also notify user groups or individual users that your APK or IPA file has been built.

:::important[Publishing to expo.io]

The **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> does not use <NT>Expo</NT> commands and doesn’t publish to [expo.io](https://docs.expo.dev/workflow/publishing/). This <NT>Step</NT> publishes artifacts to <NT>Bitrise</NT> and is not specific to a particular platform.

If you need to publish to [expo.io](https://docs.expo.dev/workflow/publishing/), set the **<NT>Run</NT> expo publish after eject?** input of the **<NT>Expo Eject</NT>** <NT>Step</NT> to `yes`. Be aware that in that case you have to provide your username and password for your <NT>Expo</NT> account to publish to [expo.io.](http://expo.io/)

:::

1. <Partial_OpeningTheWorkflowEditor />
1. Make sure you have the **<NT>Deploy to bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. In the **<NT>Notify: User Roles</NT>**, add the role so that only those get notified who have been granted with this role. Or fill out the **<NT>Notify: Emails</NT>** field with email addresses of the users you want to notify.

   Make sure you set those email addresses as [Secrets](/en/bitrise-ci/configure-builds/secrets)! These details can be also modified under **<NT>Notifications</NT>** if you click the **eye** icon next to your generated binary in the **<NT>Artifacts</NT>** tab.
1. If you want the <NT>Step</NT> to generate a public install page for you, set the **<NT>Enable</NT> public page for the App?** input to `true`.

### Deploying a Flutter app to <NT>App Store Connect</NT>

To deploy your iOS Flutter project to the App Store, you’ll need to build the app, export an IPA file and submit it to the App Store.

Unlike testing, this requires <NT>code signing</NT> files:

- An iOS Distribution Certificate (a .p12 file).
- An App Store Provisioning Profile.

For Flutter applications, <NT>code signing</NT> requires setting a Team ID in the <NT>project settings</NT> in <NT>Xcode</NT>.

#### Configuring Team ID for Flutter apps

Once you created your iOS project locally, you will need to review the <NT>project settings</NT> for it in <NT>Xcode</NT>. More specifically, you need to set a valid Team ID: without that, your build will fail on <NT>Bitrise</NT>.

1. In <NT>Xcode</NT>, open **Runner.xcworkspace** in <NT>your app</NT>’s **ios** folder.
1. To view your app’s settings, select the **Runner** project in the <NT>Xcode</NT> project navigator. Then, in the main view sidebar, select the **Runner** target.
1. Select the **General** tab.
1. In the **Signing** section, find the **<NT>Team</NT>** menu and set it to the team associated with your registered <NT>Apple</NT> Developer account.
1. Commit the change to your repository!

   :::important[Don't forget to commit your changes!]

   If you only set the Team ID locally, your build will still FAIL on <NT>Bitrise</NT>!

   :::

#### Configuring Flutter deployment to the App Store on <NT>Bitrise</NT>

To deploy your app to the App Store, you need to upload the <NT>code signing</NT> files. You have two options:

- Upload the code <NT>signing certificate</NT>(s) to <NT>Bitrise</NT> and use <NT>automatic code signing</NT> with the [Manage iOS Code signing](https://bitrise.io/integrations/steps/manage-ios-code-signing) <NT>Step</NT>.
- Upload the provisioning profile(s) and the code <NT>signing certificate</NT>(s) to <NT>Bitrise</NT> and use manual <NT>code signing</NT> with the [Certificate and profile installer](https://bitrise.io/integrations/steps/certificate-and-profile-installer) <NT>Step</NT>.

In this guide, we will focus on the second option, manual <NT>code signing</NT>:

1. Make sure you have the **<NT>Certificate and profile installer</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. Upload the required <NT>code signing</NT> files to [Bitrise](https://app.bitrise.io/users/sign_in).
1. Open the **<NT>Flutter Build</NT>** <NT>Step</NT> and find the **iOS <NT>Platform</NT> Configs** input group.
1. Make sure the **<NT>Additional parameters</NT>** input has the value `--release`.
1. Check the **<NT>Platform</NT>** input of the <NT>Step</NT>: make sure it’s set to either `iOS` or `both`.
1. Set the **iOS output artifact type** input to **archive**.

   ![flutter-build-ios-config.png](/img/_paligo/uuid-821c4edf-e9a6-a85e-a6e5-8cce61ab5f42.png)
1. Add the **Export iOS and tvOS <NT>Xcode archive</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.

   It should be after the **<NT>Flutter Build</NT>** <NT>Step</NT>.
1. Set the **<NT>Distribution method</NT>** input of the <NT>Step</NT> to **app-store**.
1. Add the **<NT>Deploy to App Store Connect</NT>** <NT>Step</NT> to the end of the <NT>Workflow</NT>.
1. Provide your <NT>Apple</NT> credentials in the respective input fields.

   - <NT>Apple ID</NT>.
   - password or, if you use two-factor authentication on <NT>App Store Connect</NT>, your application password.

   Don’t worry, the password will not be visible in the logs or exposed .
1. Start a build!

If all goes well, the <NT>Step</NT> will submit the app to <NT>App Store Connect</NT>. You can, from the <NT>App Store Connect</NT> page, distribute the app to external testers via Testflight, or release it to the App Store itself.

### Deploying a Flutter app to <NT>Google Play</NT>

To deploy your app to <NT>Google Play</NT>, you need to export an App Bundle file and sign it. You have two options:

- You can [configure code signing](https://flutter.dev/docs/deployment/android#configure-signing-in-gradle) in the app’s `build.gradle` file and then Flutter will sign your app during the build phase.
- Sign your AAB or APK file on <NT>Bitrise</NT> with our dedicated <NT>Step</NT>.

The scope of this guide is the second option: signing your AAB/APK file with the **<NT>Android Sign</NT>** <NT>Step</NT>.

1. [Generate a keystore file](https://flutter.io/docs/deployment/android#create-a-keystore).

   The keystore file is required for <NT>code signing</NT>.
1. Open an app on <NT>Bitrise</NT> and go to **<NT>Workflow</NT>** > **<NT>Code signing</NT>** > **<NT>Android</NT> <NT>Code Signing</NT>**.
1. Drag and drop the keystore file, and fill out the metadata.
1. [Configure Google Play access.](/en/bitrise-ci/deploying/android-deployment/generating-and-deploying-android-app-bundles#setting-up-google-play-deployment-for-the-first-time)

   You only need to do this for your very first <NT>Google Play</NT> deployment of the app.
1. In the **<NT>Flutter Build</NT>** <NT>Step</NT>, find the **<NT>Android</NT> <NT>Platform</NT> Configs** input group and make sure the **<NT>Additional parameters</NT>** input has the value `--release`.
1. In the **<NT>Android</NT> output artifact type** input field, select either **APK** or **appbundle** depending on your deployment requirements.
1. In the **Output (.apk, .aab) pattern** input, set the path where the **<NT>Deploy to Google Play</NT>** <NT>Step</NT> will be able to access the generated binary.

   The path should be relative to the <NT>project source</NT> directory, stored in the BITRISE_SOURCE_DIR Environment Variable.
1. Make sure you have the **<NT>Deploy to Google Play</NT>** <NT>Step</NT> after the **<NT>Android Sign</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. Fill out the required input fields as follows:

   - **<NT>Service Account JSON key file path</NT>**: This field can accept a remote URL so you have to provide the Env Var which contains your uploaded service account JSON key. For example: `$BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL`.
   - **<NT>Package name</NT>**: The <NT>package name</NT> of your <NT>Android</NT> app.
   - **Track**: The track where you want to deploy your APK (for example, alpha/beta/rollout/production or any custom track you set).
