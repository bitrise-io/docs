## Adding a <NT>React Native</NT> project to <NT>Bitrise</NT>

Add your project the same way as any other: see [Adding a new project](/en/bitrise-ci/getting-started/adding-a-new-project) for the full walkthrough of the wizard. This section covers what's specific to <NT>React Native</NT>.

### How the scanner detects a <NT>React Native</NT> project

The <NT><GlossTerm baseform="project scanner">project scanner</GlossTerm></NT> looks for a `package.json` file that lists `react-native` as a dependency, skipping anything under `node_modules`. It then checks for an `ios/` and an `android/` directory next to it, and only registers the project as <NT>React Native</NT> if at least one of them exists.

The scanner also checks whether your project uses <NT>Expo</NT>: if `package.json` lists `expo` as a dependency and there's an `app.json`, `app.config.js`, or `app.config.ts` file alongside it, <NT>Bitrise</NT> treats the project as <NT>Expo</NT>-based instead, and skips the native `ios/`/`android/` check. See [Getting started with Expo projects](/en/bitrise-ci/getting-started/quick-start-guides/getting-started-with-expo-projects) for that flow.

### What you can configure

During the **<NT>CI configuration</NT>** stage, the wizard shows:

- **<NT>React Native</NT> project directory**: the directory containing your `package.json` file.
- <NT>For the</NT> <NT>Android</NT> part of your project: **Project root directory**, **<NT>Module</NT>** (defaults to `app`), and **Variant** (defaults to `Debug`).
- <NT>For the</NT> iOS part of your project: **<NT>Project or workspace path</NT>**, **<NT>Scheme name</NT>**, and **<NT>Distribution method</NT>**.

## <NT>Code signing</NT> for <NT>React Native</NT> projects

A <NT>React Native</NT> project can consist of an <NT>Android</NT> and an iOS project. Both have different signing procedures. Follow our platform-specific instructions to code sign your mobile app for both iOS and <NT>Android</NT>.

### Signing your <NT>Android</NT> project

All <NT>Android</NT> apps must be digitally signed with a certificate before they can be installed on <NT>Android</NT> devices. On <NT>Bitrise</NT>, you can use our dedicated <NT><GlossTerm baseform="step">Step</GlossTerm></NT> for this purpose but first you'll need a keystore file.

1. [Generate a keystore file](https://reactnative.dev/docs/signed-apk-android#generating-an-upload-key).
1. <Partial_GettingToTheProjectSettingsPage />
1. On the left, select **<NT>Code signing</NT>** from the menu.
1. On the **<NT>Android</NT>** tab, click **<NT>Add keystore file</NT>**.
1. In the dialog, drag-and-drop the file and fill out the required fields then click **<NT>Continue</NT>**

   ![keystore-alias.png](/img/_paligo/uuid-0b79fc48-3861-0744-0f53-53a6c65ffc2c.png)
1. Open the <NT><GlossTerm baseform="Workflow Editor">Workflow Editor</GlossTerm></NT>.
1. Add the **<NT>Android Sign</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.

   If you uploaded your keystore file and filled out the metadata, the required inputs of the <NT>Step</NT> are already filled and require no more configuration from you.

   :::important[Prerequisite of the <NT>Android Sign</NT> <NT>Step</NT>]

   Make sure to add the **<NT>Android Sign</NT>** <NT>Step</NT> after a build <NT>Step</NT> (**<NT>Android Build</NT>** <NT>Step</NT> or **<NT>Gradle Runner</NT>** <NT>Step</NT>) in your <NT>Workflow</NT>.

   :::

### Creating a signed IPA for <NT>Xcode</NT> projects

:::note[Overview on iOS <NT>code signing</NT> in <NT>Bitrise</NT>]

For a comprehensive overview on what <NT>Steps</NT> are available for <NT>code signing</NT> asset management, visit the [iOS code signing page](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects).

:::

You can easily create a signed IPA file for your <NT>Xcode</NT> project with <NT>Bitrise</NT>.

- <NT>You have</NT> set up [Apple service connection](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services) on <NT>Bitrise</NT>.
- Your [code signing files are managed correctly](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects).
- You set the relevant inputs of our **<NT>Xcode Archive & Export for iOS</NT>** <NT>Step</NT>.

:::important[Upload the distribution AND the development <NT>signing certificate</NT>s]

We strongly recommend uploading BOTH the development and distribution <NT>signing certificate</NT>s for your project. If you don't have an uploaded development <NT>signing certificate</NT>, <NT>Steps</NT> with automatic provisioning options will generate one on the fly every time you start a build. This can eventually lead to reaching the maximum number of certificates, blocking you from starting new builds.

:::

If you’re all set, proceed to setting up IPA export in your <NT>Workflow</NT>.

#### Signing an IPA file with a different team’s <NT>code signing</NT> file

You might want to sign the IPA file with a different team’s <NT>code signing</NT> files. For example:

- If you use your company’s <NT>code signing</NT> files for internal builds, but your client’s <NT>code signing</NT> files are used for App Store distribution.
- If you use <NT>Apple ID</NT> for <NT>automatic code signing</NT> and the <NT>Apple ID</NT> belongs to multiple teams, use The Developer Portal team to use for this export input to specify which team should be used for <NT>automatic code signing</NT> asset management.

To do so:

1. Make sure the right <NT>code signing</NT> files of the new development team are uploaded to <NT>Bitrise</NT>.
1. Set the The Developer Portal team to use for this export option as well (in addition to the **<NT>Distribution method</NT>**).
1. Set the **<NT>Distribution method</NT>**.

## Deploying a <NT>React Native</NT> app

You can deploy your <NT>React Native</NT> app to:

- bitrise.io: This allows you download the generated binary, and to share it with others via the public install page.
- Online stores: we have integrations for multiple different online stores. In these guides, we'll show you how to publish to <NT>Google Play</NT> and to <NT>Apple</NT>'s App Store.

:::note[CodePush]

You can deploy updates to your users' devices with <NT>Bitrise</NT> CodePush. CodePush is part of [Release Management](/en/release-management) and it is supported for <NT>React Native</NT> and <NT>Expo</NT> apps.

For more information about CodePush, check out the official guides: [CodePush](/en/release-management/codepush/about-codepush).

:::

### Deploying a cross-platform app to bitrise.io

The **<NT>Deploy to bitrise.io</NT>** <NT>Step</NT> uploads all the artifacts related to your build into the [**Artifacts**](/en/bitrise-ci/run-and-analyze-builds/managing-build-files/build-artifacts-online) tab on your <NT>Build</NT>’s page.

You can share the generated binary file (APK for <NT>Android</NT> or an IPA file for iOS) with your team members using the public install page. The public install page is a URL you can share with others who can install the generated app binary on their device. You can also notify user groups or individual users that your APK or IPA file has been built.

:::important[Publishing to expo.io]

The **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> does not use <NT>Expo</NT> commands and doesn’t publish to [expo.io](https://docs.expo.dev/workflow/publishing/). This <NT>Step</NT> publishes artifacts to <NT>Bitrise</NT> and is not specific to a particular platform.

If you need to publish to [expo.io](https://docs.expo.dev/workflow/publishing/), set the **<NT>Run</NT> expo publish after eject?** input of the **<NT>Expo Eject</NT>** <NT>Step</NT> to `yes`. Be aware that in that case you have to provide your username and password for your <NT>Expo</NT> account to publish to [expo.io.](http://expo.io/)

:::

1. <Partial_OpeningTheWorkflowEditor />
1. Make sure you have the **<NT>Deploy to bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. In the **<NT>Notify: User Roles</NT>**, add the role so that only those get notified who have been granted with this role. Or fill out the **<NT>Notify: Emails</NT>** field with email addresses of the users you want to notify.

   Make sure you set those email addresses as [Secrets](/en/bitrise-ci/configure-builds/secrets)! These details can be also modified under **<NT>Notifications</NT>** if you click the **eye** icon next to your generated binary in the **<NT>Artifacts</NT>** tab.
1. If you want the <NT>Step</NT> to generate a public install page for you, set the **<NT>Enable</NT> public page for the App?** input to `true`.

### Deploying the app to <NT>App Store Connect</NT>

Keep in mind that every time you want to push an app to <NT>App Store Connect</NT>, it must have a unique build and version number: [increment either or both](/en/bitrise-ci/run-and-analyze-builds/build-numbering-and-app-versioning) before deploying.

1. Make sure you have a working [connection to your Apple Developer account](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).
1. Generate an IPA file on your own machine at least once.
1. [Upload all necessary code signing files](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) to <NT>Bitrise</NT>.

   To deploy an app to <NT>App Store Connect</NT>, you need a Distribution type certificate and an App Store type provisioning profile.
1. Make sure the **<NT>Xcode Archive & Export for iOS</NT>** <NT>Step</NT> is in your <NT>Workflow</NT>.
1. Set the **<NT>Automatic code signing</NT>** input to the <NT>Apple service connection</NT> you want to use for <NT>code signing</NT>. The available options are:

   - `off` if you don’t use <NT>automatic code signing</NT>, or if you exclusively use <NT>Step inputs</NT> for <NT>Apple</NT> service authentication.
   - `api-key` if you use [API key authorization](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).
   - `apple-id` if you use [Apple ID authorization](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services).
1. Set the **<NT>Distribution method</NT>** input of the <NT>Step</NT> to `app-store`.

   The <NT>Step</NT> will store the path of the exported .ipa file in the $BITRISE_IPA_PATH Environment Variable.
1. Add the **<NT>Deploy to App Store Connect</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Fill the required inputs.

   - Either the app’s <NT>Apple ID</NT> or its Bundle ID is a required input. One of the two must be provided.
   - If you set the **<NT>Submit</NT> for <NT>Review</NT>** to `yes`, the <NT>Step</NT> will wait for your submission to be processed on <NT>App Store Connect</NT> and then submit the given version of the app for review.
   - The <NT>default value</NT> of the **<NT>Skip</NT> <NT>App</NT> <NT>Version</NT> Update** input is `No`. Change it only if you incremented the app version number in another way.
   - If you use an <NT>App Store Connect</NT> account that is linked to multiple teams, provide either a Team ID or a Team name!
1. Start a build.

If all goes well, your app will be submitted to App Store and you can distribute it via Testflight or via the App Store!

### Deploying your <NT>Android</NT> project to <NT>Google Play</NT>

You can use the **<NT>Deploy to Google Play</NT>** <NT>Step</NT> in your <NT>Workflow</NT> to upload your digitally signed AAB/APK to the <NT>Google Play Store</NT>.

1. Make sure your <NT>Workflow</NT> includes the **<NT>Android Build</NT>** <NT>Step</NT> and that it runs before the **<NT>Android Sign</NT>** <NT>Step</NT>.
1. [Configure code signing](/en/bitrise-ci/getting-started/quick-start-guides/getting-started-with-react-native-projects#signing-your-android-project) for your app.
1. [Configure Google Play access.](/en/bitrise-ci/deploying/android-deployment/generating-and-deploying-android-app-bundles#setting-up-google-play-deployment-for-the-first-time)

   You only need to do this for your very first <NT>Google Play</NT> deployment of the app.
1. Make sure you have the **<NT>Deploy to Google Play</NT>** <NT>Step</NT> after the **<NT>Android Sign</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. Fill out the required input fields as follows:

   - **<NT>Service Account JSON key file path</NT>**: This field can accept a remote URL so you have to provide the Env Var which contains your uploaded service account JSON key. For example: `$BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL`.
   - **<NT>Package name</NT>**: The <NT>package name</NT> of your <NT>Android</NT> app.
   - **Track**: The track where you want to deploy your APK (for example, alpha/beta/rollout/production or any custom track you set).
