You can use <NT>Cordova</NT> and <NT>Ionic</NT> frameworks to develop cross-platform apps. <NT>Bitrise</NT> can help you with its automated testing, <NT>code signing</NT> and deployment procedures so that you can ship your iOS and/or <NT>Android</NT> app/s to the respective marketplace in no time! If your <NT><GlossTerm baseform="workspace">Workspace</GlossTerm></NT> has more than one concurrency, you can have <NT>Android</NT> and iOS builds run simultaneously. Now let us guide you through the process!

## Adding an <NT>Ionic</NT>/<NT>Cordova</NT> project to <NT>Bitrise</NT>

Add your project the same way as any other: see [Adding a new project](/en/bitrise-ci/getting-started/adding-a-new-project) for the full walkthrough of the wizard. This section covers what's specific to <NT>Ionic</NT> and <NT>Cordova</NT>.

### How the scanner tells <NT>Ionic</NT> and <NT>Cordova</NT> apart

The <NT><GlossTerm baseform="project scanner">project scanner</GlossTerm></NT> checks for an `ionic.config.json` file first, or the legacy `ionic.project` file. If it finds one, it registers the project as <NT>Ionic</NT> and doesn't check for `config.xml` at all.

If neither <NT>Ionic</NT> file is present, the scanner looks for a `config.xml` file with a `cordova.apache.org` namespace and registers the project as <NT>Cordova</NT>. If it also finds an <NT>Ionic</NT> config file alongside that `config.xml`, it defers to <NT>Ionic</NT> instead.

### What you can configure

During the **<NT>CI configuration</NT>** stage, the wizard shows:

- **Directory of the config.xml file**: shown only if your config file isn't at the root of your repository.
- **<NT>Platform</NT>**: which platform to build for, <NT>Android</NT>, iOS, or both.

## Testing <NT>Ionic</NT>/<NT>Cordova</NT> apps

You can run unit tests for <NT>Ionic</NT>/<NT>Cordova</NT> apps on <NT>Bitrise</NT> by using our **<NT>Karma Jasmine Test Runner</NT>** or **<NT>Jasmine Test Runner</NT>** <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s.

If your <NT>Cordova</NT>/<NT>Ionic</NT> project has a Karma Jasmine dependency in its **package.json** file and a `karma.conf.js` file in the project root, our scanner will detect it when you're adding your app and automatically insert the respective testing <NT>Step</NT> into your <NT>Workflow</NT>. If this dependency is missing from your project, you can manually insert one of our testing steps to your <NT>Workflow</NT> using our <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT>.

1. <Partial_OpeningTheWorkflowEditor />
1. Make sure you have a testing <NT>Step</NT> in your <NT>Workflow</NT>.

   :::important[Installing dependencies]

   Make sure you've completed [Installing JavaScript dependencies](#installing-javascript-dependencies) before you run tests.

   :::

   You can choose between the **<NT>Karma Jasmine Test Runner</NT>** and the **<NT>Jasmine Test Runner</NT>** <NT>Steps</NT>.

**<NT>Cordova</NT> app configuration with <NT>Karma Jasmine Test Runner</NT>**

In this example, you can find a `bitrise.yml` configuration that includes a <NT>Workflow</NT> called **primary**. This <NT>Workflow</NT> includes the **<NT>Karma Jasmine Test Runner</NT>** <NT>Step</NT>.

```yaml
primary:
    steps:
    - activate-ssh-key@4.0.3:
        run_if: '{{getenv "SSH_RSA_PRIVATE_KEY" | ne ""}}'
    - git-clone@4.0.11: {}
    - script@1.1.5:
        title: Do anything with Script step
    - npm@0.9.1:
        inputs:
        - command: install
    - karma-jasmine-runner@0.9.1: {}
    - deploy-to-bitrise-io@1.3.15: {}
```


## <NT>Code signing</NT> <NT>Ionic</NT>/<NT>Cordova</NT> projects

If you want to build an app for iOS or <NT>Android</NT>, you need to upload the platform-specific files on the **<NT>Project settings</NT>** page. You can also generate builds for both platforms which requires uploading all <NT>code signing</NT> files of the platforms.

### iOS <NT>code signing</NT> for <NT>Ionic</NT> and <NT>Cordova</NT> projects

Naturally, <NT>Bitrise</NT> supports iOS applications built with either **<NT>Ionic</NT>** or **<NT>Cordova</NT>**. However, the <NT>code signing</NT> process is slightly different compared to a native <NT>Xcode</NT> project.

<NT>Bitrise</NT> supports both manual and automatic provisioning for <NT>Ionic</NT> and <NT>Cordova</NT> apps as well - and once again, the processes are somewhat different.

#### <NT>Ionic</NT>/<NT>Cordova</NT> <NT>code signing</NT> with manual <NT>code signing</NT> asset management

1. Generate the native <NT>Xcode</NT> project locally from your <NT>Ionic</NT> or <NT>Cordova</NT> project by calling `cordova platform add ios` or `ionic cordova platform add ios`.
1. Upload the files to [bitrise.io](https://www.bitrise.io): open the **<NT>Project settings</NT>** page and select **<NT>Code signing</NT>** on the left. Upload a certificate and a provisioning profile.

   ![upload-code-signing.png](/img/_paligo/uuid-88a3d6bf-be6f-1811-a908-9072b4b88a9a.png)
1. Make sure you have the **<NT>Certificate and profile installer</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. Add the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT> to your <NT>Workflow</NT>. It must come after the **<NT>Certificate and profile installer</NT>** <NT>Step</NT>.
1. Fill in the required inputs for the <NT>Step</NT>. Please note that both the **<NT>Code Signing</NT> Identity** and the **Provisioning <NT>Profile</NT>** are required inputs for iOS apps even though they are not marked as such.

   - **<NT>Build</NT> configuration**: you can set it to either `debug` or `release`.
   - **Code Sign Identity**: enter a Developer or a Distribution identity.
   - **Provisioning <NT>Profile</NT>**: enter the appropriate provisioning profile.
   - **Packaging <NT>Type</NT>**: this controls what type of build is generated by <NT>Xcode</NT>. Set the type of <NT>code signing</NT> you need.

     ![gen-cordova-build.png](/img/_paligo/uuid-bc068222-9d71-6b17-2d07-9bdca30fa368.png)
1. Add the **<NT>Cordova</NT> archive** or the **<NT>Ionic archive</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Fill in the required inputs.

   - The **<NT>Platform</NT>** input needs to be set to: `device`.
   - The **<NT>Build command configuration</NT>** input must match the **<NT>Build</NT> configuration** input of the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT>.

   This <NT>Step</NT> must come after the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT> in the <NT>Workflow</NT>.
1. Run your build!

#### <NT>Ionic</NT>/<NT>Cordova</NT> <NT>code signing</NT> with <NT>automatic code signing</NT> asset management

1. Make sure your .p12 <NT>signing certificate</NT>s are uploaded to [bitrise.io](https://www.bitrise.io).
1. Add the **<NT>Cordova prepare</NT>** or the **<NT>Ionic prepare</NT>** <NT>Step</NT> to your <NT>Workflow</NT>. These <NT>Steps</NT> call the `platform rm` and `platform add` commands.
1. Add the **[Manage iOS Code Signing](https://www.bitrise.io/integrations/steps/manage-ios-code-signing)** <NT>Step</NT> to your <NT>Workflow</NT>. If you have both the **Certificate and Profile Installer** and the **<NT>Manage iOS Code Signing</NT>** <NT>Steps</NT> in your <NT>Workflow</NT>, your build might encounter unexpected issues.

   The <NT>Step</NT> will export:

   - The project’s development team.
   - The installed codesign identity’s name.
   - The installed provisioning profile.

   :::caution[One <NT>code signing</NT> <NT>Step</NT> only]

   If you have both the **<NT>Certificate and profile installer</NT>** and the **<NT>Manage iOS Code Signing</NT>** <NT>Steps</NT> in your <NT>Workflow</NT>, your build might encounter unexpected issues.

   :::
1. Select the **<NT>Apple service connection method</NT>** (based on the [Apple service you have set up in Bitrise](/en/bitrise-platform/integrations/apple-services-connection/about-connecting-to-apple-services)) and the **<NT>Distribution method</NT>**.

   ![manage-ios-code.png](/img/_paligo/uuid-de743a8c-7a79-dda4-b9de-e3c71f2ac32a.png)
1. Add the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Configure the <NT>Step</NT> to use the <NT>code signing</NT> settings exported by the **<NT>Manage iOS Code Signing</NT>** <NT>Step</NT>:

   **Development distribution example**:

   ```yaml
   - generate-cordova-build-configuration:
       inputs:
       - development_team: $BITRISE_DEVELOPER_TEAM 
       - package_type: development 
       - code_sign_identity: iPhone Developer
       - configuration: debug
   ```

   **Production distribution example**:

   ```yaml
   - generate-cordova-build-configuration:
       inputs:
       - development_team: $BITRISE_DEVELOPER_TEAM 
       - package_type: app-store 
       - code_sign_identity: iPhone Developer
       - configuration: release
   ```
1. Add the **<NT>Cordova Archive</NT>** or the **<NT>Ionic</NT> Archive** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Fill in the required inputs.

   - The **<NT>Platform</NT>** input needs to be set to: `device`.
   - The **<NT>Build command configuration</NT>** input must match the **<NT>Build</NT> configuration** input of the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT>.

     ![gen-cordova-build.png](/img/_paligo/uuid-bc068222-9d71-6b17-2d07-9bdca30fa368.png)
1. Set the **Should remove platforms as prepare step?** to `false`. This is crucial: it ensures the <NT>Step</NT> will not remove and re-add the platform of the native projects generated in the **<NT>Cordova prepare</NT>** or the **<NT>Ionic prepare</NT>** <NT>Step</NT>.
1. Run your build!

### <NT>Android</NT> <NT>code signing</NT> using the <NT>Android Sign</NT> <NT>Step</NT>

You can create a signed APK using the **<NT>Android Sign</NT>** <NT>Step</NT> in your <NT>Bitrise</NT> <NT>Workflow</NT>. This <NT>Step</NT> is configured to run if you have already uploaded your [keystore file](https://developer.android.com/studio/publish/app-signing#generate-key) to <NT>Bitrise</NT>.

The **<NT>Android Sign</NT>** <NT>Step</NT> is not required if signing is configured in your project’s `build.gradle` file. If so, running the **<NT>Android Build</NT>** <NT>Step</NT> (or the **<NT>Gradle Runner</NT>** <NT>Step</NT>) signs the output (APK or AAB) automatically. Nevertheless, we recommend that you use the **<NT>Android Sign</NT>** <NT>Step</NT> to sign your project in an easy and secure way.

:::note[jarsign and apksigner]

APKs can be signed with either `jarsigner` or `apksigner`. For APKs, if you wish to use apksigner to sign your project, then in the <NT>Android Sign</NT> <NT>Step</NT> you have to first set the Enables `apksigner` input to true and leave the APK Signature Scheme input on automatic. This way `apksigner` checks your APK’s minimum and target SDK versions and chooses the required schemes. It signs your project with V1 scheme if your minimum supported version is low and it also signs with other schemes for newer systems.

Please note that AAB files can only be signed with jarsigner. The <NT>Step</NT> uses `jarsigner` if it detects a file ending with `.aab`

:::

1. [Upload your keystore file to Bitrise](/en/bitrise-ci/code-signing/android-code-signing/uploading-android-keystore-files-to-bitrise).
1. Add the **<NT>Android Sign</NT>** <NT>Step</NT> to your <NT>Workflow</NT> after the <NT>Step</NT> that builds your APK or AAB file.

   <NT>Bitrise</NT> uses the above <NT>Environment Variables</NT> and sets them as inputs into the respective fields of the **<NT>Android Sign</NT>** <NT>Step</NT>. Once the <NT>Step</NT> runs, it produces either a signed APK or an AAB. The signed APK or AAB is used in deploy <NT>Steps</NT>, for example, the**<NT>Google Play Deploy</NT>** <NT>Step</NT> or the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT>. The latter deploys the APK/AAB on the **<NT>Artifacts</NT>** tab. You can also use [Release Management](/en/release-management) to deploy your app once you built an installable artifact.

:::note[Downloading your keystore file]

You can download your keystore file to the project directory using the **[File Downloader](https://www.bitrise.io/integrations/steps/file-downloader)** <NT>Step</NT>:

```yaml
- file-downloader:
   inputs:
   - source: $BITRISEIO_ANDROID_KEYSTORE_URL
   - destination: "$HOME/keystores/my_keystore.jks" #native android#
```

If a <NT>Step</NT> requires the keystore file, make sure to include that <NT>Step</NT> AFTER the **<NT>File Downloader</NT>** <NT>Step</NT>.

After this <NT>Step</NT>, `my_keystore.jks` will be available at `$HOME/keystores/my_keystore.jks`.

:::


## Deploying <NT>Ionic</NT>/<NT>Cordova</NT> apps

To build and deploy an <NT>Ionic</NT> or <NT>Cordova</NT> app on <NT>Bitrise</NT>, you need to digitally sign both the <NT>Android</NT> and iOS project (if you're building both) and then use the <NT>Cordova Archive</NT> or the <NT>Ionic</NT> Archive <NT>Step</NT> to build the app before deploying it.

You can deploy the successfully built app to:

- Online stores, such as the <NT>Google Play Store</NT> or <NT>Apple</NT>'s App Store.
- <NT>Bitrise.io</NT>: the generated binaries will be available on the **<NT>Artifacts</NT>** tab of the build's page. You can download them from there or share them with others via the public install page.

### Deploying a cross-platform app to bitrise.io

The **<NT>Deploy to bitrise.io</NT>** <NT>Step</NT> uploads all the artifacts related to your build into the [**Artifacts**](/en/bitrise-ci/run-and-analyze-builds/managing-build-files/build-artifacts-online) tab on your <NT>Build</NT>’s page.

You can share the generated binary file (APK for <NT>Android</NT> or an IPA file for iOS) with your team members using the public install page. The public install page is a URL you can share with others who can install the generated app binary on their device. You can also notify user groups or individual users that your APK or IPA file has been built.

:::important[Publishing to expo.io]

The **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT> does not use <NT>Expo</NT> commands and doesn’t publish to [expo.io](https://docs.expo.dev/workflow/publishing/). This <NT>Step</NT> publishes artifacts to <NT>Bitrise</NT> and is not specific to a particular platform.

If you need to publish to [expo.io](https://docs.expo.dev/workflow/publishing/), set the **<NT>Run</NT> expo publish after eject?** input of the **<NT>Expo Eject</NT>** <NT>Step</NT> to `yes`. Be aware that in that case you have to provide your username and password for your <NT>Expo</NT> account to publish to [expo.io.](http://expo.io/)

:::

1. <Partial_OpeningTheWorkflowEditor />
1. Make sure you have the **<NT>Deploy to bitrise.io</NT>** <NT>Step</NT> in your <NT>Workflow</NT>.
1. In the **<NT>Notify: User Roles</NT>**, add the role so that only those get notified who have been granted with this role. Or fill out the **<NT>Notify: Emails</NT>** field with email addresses of the users you want to notify.

   Make sure you set those email addresses as [Secrets](/en/bitrise-ci/configure-builds/secrets)! These details can be also modified under **<NT>Notifications</NT>** if you click the **eye** icon next to your generated binary in the **<NT>Artifacts</NT>** tab.
1. If you want the <NT>Step</NT> to generate a public install page for you, set the **<NT>Enable</NT> public page for the App?** input to `true`.

### Deploying your <NT>Android</NT> project to <NT>Google Play</NT>

You can use the **<NT>Deploy to Google Play</NT>** <NT>Step</NT> in your <NT>Workflow</NT> to upload your digitally signed AAB/APK to the <NT>Google Play Store</NT>.

1. Add the **<NT>Cordova</NT> archive** or the **<NT>Ionic archive</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.

   Note that if you’re building for both iOS and <NT>Android</NT> in one project, and either of your apps fails, the whole **<NT>Cordova</NT> archive/<NT>Ionic archive</NT>** <NT>Step</NT> will fail.
1. Fill in the required inputs.

   - Set **<NT>Platform</NT> to use in cordova-cli commands** (<NT>Cordova</NT>) or **<NT>Platform</NT> to use in ionic-cli commands** (<NT>Ionic</NT>) to **android** (or **ios,android** if building for both platforms).
   - Set **<NT>Build</NT> command target** to **device**.
   - The **<NT>Build command configuration</NT>** input must match the **<NT>Build</NT> configuration** input of the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT>.

   The archive <NT>Step</NT> must come after the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT> in the <NT>Workflow</NT>.
1. [Configure code signing](/en/bitrise-ci/getting-started/quick-start-guides/getting-started-with-react-native-projects#signing-your-android-project) for your app.
1. [Configure Google Play access.](/en/bitrise-ci/deploying/android-deployment/generating-and-deploying-android-app-bundles#setting-up-google-play-deployment-for-the-first-time)

   You only need to do this for your very first <NT>Google Play</NT> deployment of the app.
1. Make sure you have the **<NT>Deploy to Google Play</NT>** <NT>Step</NT> after the **<NT>Android Sign</NT>** and **<NT>Cordova Archive</NT>** or **<NT>Ionic</NT> Archive** <NT>Step</NT> in your <NT>Workflow</NT>.
1. Fill out the required input fields as follows:

   - **<NT>Service Account JSON key file path</NT>**: This field can accept a remote URL so you have to provide the Env Var which contains your uploaded service account JSON key. For example: `$BITRISEIO_SERVICE_ACCOUNT_JSON_KEY_URL`.
   - **<NT>Package name</NT>**: The <NT>package name</NT> of your <NT>Android</NT> app.
   - **Track**: The track where you want to deploy your APK (for example, alpha/beta/rollout/production or any custom track you set).

### Deploying your iOS project to the App Store

1. Add the **<NT>Cordova</NT> archive** or the **<NT>Ionic archive</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.

   Note that if you’re building for both iOS and <NT>Android</NT> in one project, and either of your apps fails, the whole **<NT>Cordova</NT> archive/<NT>Ionic archive</NT>** <NT>Step</NT> will fail.
1. Fill in the required inputs.

   - The **<NT>Platform</NT>** input needs to be set to **device**.
   - The **<NT>Build command configuration</NT>** input must match the **<NT>Build</NT> configuration** input of the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT>.

   The archive <NT>Step</NT> must come after the **<NT>Generate cordova build configuration</NT>** <NT>Step</NT> in the <NT>Workflow</NT>.
1. [Configure iOS code signing](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects) for your iOS project.
1. Add the **<NT>Deploy to App Store Connect</NT> - Application Loader (formerly iTunes Connect)** <NT>Step</NT> to your <NT>Workflow</NT>, after the **<NT>Xcode Archive & Export for iOS</NT>** <NT>Step</NT> but preferably before the **<NT>Deploy to Bitrise.io</NT>** <NT>Step</NT>.
1. Provide your <NT>Apple</NT> credentials in the **<NT>Deploy to App Store Connect</NT> - Application Loader (formerly iTunes Connect)** <NT>Step</NT>.

   The <NT>Step</NT> will need your:

   - <NT>Apple ID</NT>.
   - Password or, if you use two-factor authentication on <NT>App Store Connect</NT>, your application password.

   Don’t worry, the password will not be visible in the logs or exposed.
1. [Start a build.](/en/bitrise-ci/run-and-analyze-builds/starting-builds/approving-pull-request-builds)
