<NT>Bitrise</NT> manages your builds with <NT><GlossTerm baseform="step">Step</GlossTerm></NT>s, <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s, and <NT><GlossTerm baseform="pipeline">Pipeline</GlossTerm></NT>s.

- <NT>Steps</NT> are individual build tasks: for example, cloning your repository, running unit tests, or creating an installable binary are all individual build tasks.
- <NT>Workflows</NT> are sequences of <NT>Steps</NT>: when running a <NT>Bitrise</NT> build, each <NT>Step</NT> is executed in the order that is defined in a <NT>Workflow</NT>.
- <NT>Pipelines</NT> represent the top level of a <NT>Bitrise CI</NT>/CD configuration. <NT>Pipelines</NT> can be used to organize the entire CI/CD process and to set up advanced configurations with multiple different tasks running parallel and/or sequentially.

The configuration is defined entirely in a `bitrise.yml` file that you can store on <NT>Bitrise</NT> or in your own repository. It also defines the infrastructure used for the build: <NT>Bitrise</NT> builds run in a clean virtual machine, with preinstalled tools and services for most mobile development use cases. These are defined by our <NT><GlossTerm baseform="stack">stack</GlossTerm></NT>s which are updated regularly.

## <NT>Steps</NT> and <NT>Workflows</NT>

[Workflows](/en/bitrise-ci/workflows-and-pipelines/workflows/workflows-overview) are a collection of [Steps](/en/bitrise-ci/workflows-and-pipelines/steps/steps-overview) which are executed in order. This order can be configured in the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT>, or by editing the `bitrise.yml` file directly.

Each <NT>Bitrise</NT> build runs at least one <NT>Workflow</NT>. You can start builds by triggering a specific <NT>Workflow</NT>: in such a build, <NT>Workflows</NT> can be chained together but they run sequentially. In a <NT>Pipeline</NT>, <NT>Workflows</NT> are organized in stages, and in each stage, <NT>Workflows</NT> run in parallel.

When you add a new app on <NT>Bitrise</NT>, we automatically create <NT>default Workflows</NT> for you. They contain the most frequently used <NT>Step</NT> for the app's platform. You can add or remove <NT>Steps</NT> any time from a <NT>Workflow</NT>. <NT>Steps</NT> can be configured via <NT>Step inputs</NT> and they can generate data that can be used by subsequent <NT>Steps</NT> in the same <NT>Workflow</NT>.

<NT>Bitrise Steps</NT> are open source: you can suggest improvements to existing <NT>Steps</NT> or develop, share, and use your own.

## <NT>Pipelines</NT>

[Pipelines](/en/bitrise-ci/workflows-and-pipelines/build-pipelines/pipelines-with-stages/configuring-a-pipeline-with-stages) enable a complex CI/CD configuration for advanced use cases. <NT>Pipelines</NT> can be used to organize the entire CI/CD process and to set up advanced configurations with multiple different tasks running parallel and/or sequentially.

A <NT>Pipeline</NT> allows you to configure dependencies between <NT>Workflows</NT>. Each <NT>Workflow</NT> starts executing when its parent <NT>Workflows</NT> are done. <NT>Workflows</NT> on the same level are executed in parallel.

You can configure build <NT><GlossTerm baseform="trigger">trigger</GlossTerm></NT>s to trigger a <NT>Pipeline</NT> directly. When a <NT>Pipeline</NT> is triggered, the <NT>build machine</NT> starts executing all the <NT>Workflows</NT> of the first stage.

## <NT>Environment Variables</NT> and <NT>Secrets</NT>

You can store and use configuration data, or any other data in key-value pairs called [Environment Variables](/en/bitrise-ci/configure-builds/environment-variables). This enables, among other things, the reuse of build configurations across multiple different apps: in each app, Env Vars with the same key can store different values.

Env Vars have different scopes: You can configure app-level and <NT>Workflow</NT>-level Env Vars, as well as create new variables during the build, and pass these on to subsequent <NT>Steps</NT> and <NT>Workflows</NT> in the build. Any <NT>Bitrise Step</NT> that generates some sort of output makes that output available to subsequent <NT>Steps</NT> in the form of Env Vars.

<NT>Secrets</NT> are specific types of Env Vars: they are stored in encrypted format and their values are never exposed in build logs or in the `bitrise.yml` file.

## <NT>Build machines</NT> and stacks

<NT>Bitrise</NT> builds run in clean virtual machines that are destroyed once the build is finished, ensuring the security of your code. You can select the type of the <NT>build machine</NT>: the operating system (macOS or <NT>Linux</NT>) and the amount of computing resources the machine offers.

A build stack defines what tools and services are installed on the VM that runs your build. For example, our <NT>Xcode</NT> stacks each come with a different version of <NT>Xcode</NT>.
