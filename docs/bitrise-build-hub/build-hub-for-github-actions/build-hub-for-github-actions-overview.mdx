<NT>Bitrise</NT> Build Hub is a high-performance build infrastructure for <NT>GitHub</NT> Actions, purpose-built for mobile app development. It provides fully managed, zero-maintenance runners that execute your <NT>GitHub</NT> Actions workflows on the industry's fastest <NT>Apple</NT> silicon and <NT>Linux</NT> machines.

Find out how to set up Build Hub for <NT>GitHub</NT> Actions: [Configuring Build Hub for GitHub Actions](/en/bitrise-build-hub/build-hub-for-github-actions/configuring-build-hub-for-github-actions).

## Key capabilities

- `M4 Pro` <NT>Apple</NT> silicon and `AMD EPYC Zen4/Zen5` <NT>Linux</NT> machines optimized for iOS and <NT>Android</NT> builds.
- Latest <NT>Xcode</NT> versions within 24 hours of <NT>Apple</NT> release, including betas.
- Mobile-optimized <NT><GlossTerm baseform="stack">stack</GlossTerm></NT>s with preinstalled tooling (`Xcode`, `Android SDK`, `Flutter`, `React Native`, and more).
- Co-located caching for near-zero latency and no network egress costs.
- US and EU data centers for data residency requirements.
- Pre-warmed VM pools for instant build start with no queue times.

Build Hub works with both <NT>GitHub Cloud</NT> and <NT>GitHub Enterprise Server</NT> repositories. Your existing <NT>GitHub</NT> Actions workflow files stay unchanged—you only update the runs-on label to route builds to <NT>Bitrise</NT> infrastructure.

## Requirements

To run your <NT>GitHub</NT> Actions workflows on <NT>Bitrise</NT> Build Hub infrastructure, you need:

- A <NT>Bitrise</NT> <NT><GlossTerm baseform="workspace">workspace</GlossTerm></NT>.
- A <NT>GitHub</NT> account.
- A way to authenticate the Build Hub machines to <NT>GitHub</NT> Actions. You can either use a personal access token or the <NT>Bitrise</NT> - Build Hub app. We recommend using the app.
- A machine pool on <NT>Bitrise</NT>: you can select the <NT>machine type</NT>, the amount of machines, and the system image that contains the software configuration required to launch your instance.


## Authentication

<NT>Bitrise</NT> Build Hub requires either the <NT>Bitrise</NT> - Build Hub <NT>GitHub</NT> app or a [GitHub personal access token](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens) to authenticate the runners and enable proper management of the <NT>GitHub</NT> Action workflows and runner configurations.

We recommend using the <NT>GitHub</NT> app integration.

### <NT>GitHub</NT> app authentication

The <NT>Bitrise</NT> - Build Hub app can be installed to a <NT>GitHub</NT> account or a <NT>GitHub</NT> organization. The app can be scoped to access either:

- <NT>All repositories</NT>: All current and future repositories owned by the resource owner. Includes public repositories.
- Select repositories: At least one repository must be selected. Build Hub won't be able to access other repositories owned by the same resource owner.

The app has the following access rights:

- Read access to actions, metadata, and organization events.
- Read and write access to organization self hosted runners.

You can install the app to the <NT>GitHub</NT> account or organization of your choice when [creating a machine pool](/en/bitrise-build-hub/build-hub-for-github-actions/configuring-build-hub-for-github-actions#creating-a-machine-pool).

### <NT>GitHub</NT> personal access token

You can use two types of tokens:

- [Classic tokens](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens#creating-a-personal-access-token-classic)
- [Fine-grained access tokens](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens#creating-a-fine-grained-personal-access-token)

The different token types require different access types, depending on your [target scope](/en/bitrise-build-hub/build-hub-for-github-actions/build-hub-for-github-actions-overview#target-scope). Check out the table below to see the exact permissions for the respective target scopes:

| Target scope | Required permission for a classic token | Required permissions for a fine-grained access token |
| --- | --- | --- |
| <NT>GitHub</NT> Enterprise (GHE) Cloud: https://github.com/enterprises/&lt;enterprise&gt;  GHE Server: https://&lt;hostname&gt;/enterprises/&lt;enterprise&gt; | `manage_runners:enterprise` | Not supported |
| <NT>GitHub Cloud</NT> organization: https://github.com/&lt;org&gt;  GHE organization: https://&lt;hostname&gt;/&lt;org&gt; | `org:admin` | `read/write` permission to **<NT>Self-hosted runners</NT>** |
| <NT>GitHub Cloud</NT> repository: https://github.com/&lt;owner&gt;/&lt;repo&gt;  GHE repository: https://&lt;hostname&gt;/&lt;owner&gt;/&lt;repo&gt; | `workflow` | Not supported |

[Create a personal access token](/en/bitrise-build-hub/build-hub-for-github-actions/configuring-build-hub-for-github-actions#creating-github-personal-access-tokens-for-build-hub) before [creating a machine pool](/en/bitrise-build-hub/build-hub-for-github-actions/configuring-build-hub-for-github-actions#creating-a-machine-pool).


## Provisioning machines for <NT>GitHub</NT> Actions

When [creating a machine pool](/en/bitrise-build-hub/build-hub-for-github-actions/configuring-build-hub-for-github-actions#creating-a-machine-pool), you must request a certain number of machines that will run your builds. The number of machines we keep running at a time depends on the [authentication method](/en/bitrise-build-hub/build-hub-for-github-actions/build-hub-for-github-actions-overview#authentication):

- If you use our <NT>GitHub</NT> app: We create and start machines on-demand. When a <NT>GitHub</NT> Actions workflow starts, we provision a machine. If you start multiple <NT>GitHub</NT> Actions workflows at the same time, we start a machine for each, up to the maximum number of machines defined by the machine pool configuration. When the <NT>GitHub</NT> Actions build ends, the machines are shut down.
- If you use a <NT>GitHub</NT> personal access token: We always keep running as many machines as you requested in the machine pool. If you need fewer machines at a given time, you need to manually reduce the number of machines in the machine pool configuration.


## Target scope

When using a <NT>GitHub</NT> personal access token to authenticate the Build Hub, you need to set a target scope when [creating a machine pool](/en/bitrise-build-hub/build-hub-for-github-actions/configuring-build-hub-for-github-actions#creating-a-machine-pool). The target scope is the context in which the runner is allowed to operate, such as a specific repository or a <NT>GitHub</NT> organization.

The accepted target scopes are:

- <NT>GitHub Cloud</NT> organization: https://github.com/&lt;org&gt;
- <NT>GitHub Cloud</NT> repository: https://github.com/&lt;owner&gt;/&lt;repo&gt;
- <NT>GitHub</NT> Enterprise (GHE) Cloud: https://github.com/enterprises/&lt;enterprise&gt;
- GHE Server: https://&lt;hostname&gt;/enterprises/&lt;enterprise&gt;
- GHE organization: https://&lt;hostname&gt;/&lt;org&gt;
- GHE repository: https://&lt;hostname&gt;/&lt;owner&gt;/&lt;repo&gt;


## Warmup script

Use a warmup script to customize your build environment and improve speed and performance. You can add any script to your configuration when creating a machine pool. The script will run when the machine is set up.

Make sure that it returns a non-zero exit code in case of an error. The script will only fail if your script returns with a non-zero exit code.


## Rolling update percentage

When creating a machine pool for Build Hub, you can set a rolling update percentage. This is the ratio of machines that are immediately and simultaneously rebooted after you reconfigure an existing machine pool.

This allows users to update the runner configuration without losing ongoing builds. If you set the value to 100, all machines are rebooted immediately and ongoing builds are aborted.
