You must create and connect a <NT>Cloud Controller</NT> to take advantage of our Controller-managed <NT>AWS</NT> offering. The process has two stages:

1. [Configuring the Controller on Bitrise.](#configuring-the-controller-on-bitrise)
1. Setting up an instance running the controller agent on <NT>AWS</NT>.

## Configuring the controller on <NT>Bitrise</NT>

1. Get access to the <NT>AWS</NT> controller feature from the <NT>Bitrise</NT> team.
1. <NT>Log in</NT> to your <NT>Bitrise</NT> <NT><GlossTerm baseform="workspace">Workspace</GlossTerm></NT> as a <NT>Workspace</NT> owner.

   Only <NT>Workspace</NT> owners can access and configure this feature.
1. Open the **<NT>Workspace settings</NT>** page of the <NT>Workspace</NT>.
1. Select **<NT>Self-hosted infrastructure</NT>** and go to the **<NT>Bitrise on AWS</NT>** tab.

   ![create-controller-1.png](/img/_paligo/uuid-bfadf366-bdfc-4440-d2db-5e9f68843f9f.png)
1. Click **<NT>Connect with AWS</NT>**. This should bring up a dialog box.
1. Set a name for your controller and specify if you want to use the controller in an air-gapped network.

   For more information on air-gapped mode, see [Bitrise on AWS: Cloud Controller](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/aws-cloudformation-templates).

   ![create-controller-1.png](/img/_paligo/uuid-a4204c31-4165-2af9-290e-e3116ef784a3.png)
1. <NT>Copy and save your token</NT>.

   :::important[Save the token]

   This token is only accessible only right after creation. If the token is lost, you need to delete the controller and create a new one which would generate a <NT>new token</NT>.

   :::

After successful creation, the <NT>Bitrise on AWS</NT> screen should indicate that the controller on the <NT>Bitrise</NT> side is prepared, and it’s waiting for an <NT>AWS</NT> connection. In all cases the Controller and the <NT>build machines</NT> will initiate network calls toward the <NT>Bitrise</NT> control plane, no external inbound traffic is required.

## Setting up the controller instance on <NT>AWS</NT>

After a successful controller creation on <NT>Bitrise</NT>, go to <NT>AWS</NT> and create an instance running the Controller Agent.

:::note[New or existing VPC]

The configuration will need a Virtual Private Cloud (VPC). Every <NT>AWS</NT> region contains a default VPC but if necessary, you can create your own VPC within each region. If you wish to share your VPC between <NT>Bitrise</NT>-specific resources and your other resources you can reuse the already existing and configured VPC. If you prefer to separate resources, you can create a new VPC with the provided template.

For more information, read [the official AWS documentation on VPCs](https://docs.aws.amazon.com/vpc/latest/userguide/configure-your-vpc.html).

:::

<TabItem value="setup-with-an-existing-vpc" label="Setup with an existing VPC" default>

1. <NT>Log in</NT> to your <NT>AWS</NT> account.
1. Select the template option you need and you'll get the HTTP link to your template. You can read more about templates on the [AWS CloudFormation templates](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/aws-cloudformation-templates) page.

   - [Link to the base template](https://bitrise-cloudformation-templates.s3.amazonaws.com/cloud-controller-with-existing-vpc.yaml).
   - [Link to the airgapped template](https://bitrise-cloudformation-templates.s3.amazonaws.com/cloud-controller-with-existing-vpc-airgapped.yaml). Use this if the <NT>Bitrise</NT> Controller is expected to run [in airgapped mode](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/aws-cloudformation-templates).

   :::note[Additional manual configuration]

   The airgapped support requires some manual configuration in <NT>AWS</NT> that is not automated via CloudFormation. For details, please check the [AWS CloudFormation templates](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/aws-cloudformation-templates) page.

   :::
1. Navigate to CloudFormation and select **Create <NT><GlossTerm baseform="stack">stack</GlossTerm></NT> with new resources (standard)**.

   ![controller-create-stack.png](/img/_paligo/uuid-336544f8-0f74-08ad-8a5d-616f33d62736.png)
1. Create a stack using the **Template is ready** and **<NT>Amazon S3 URL</NT>** options.
1. In the **<NT>Amazon S3 URL</NT>** field, provide the URL gathered in the previous step and click **<NT>Next</NT>**.

   ![create-stack-template.png](/img/_paligo/uuid-d2e802df-0b0d-afba-f36c-ad6a9af244bb.png)
1. On the following prompt, specify the following fields:

   - **<NT>Stack name</NT>**: Any value works here. This helps to identify the stack if you have multiple stacks.
   - **<NT>Latest AMI ID</NT>**: This AMI ID is used as a base AMI to run the <NT>Cloud Controller</NT> on.
   - **<NT>Bitrise</NT> Controller Token**: The token is generated as part of controller creation on <NT>Bitrise</NT>: [Configuring the controller on Bitrise](#configuring-the-controller-on-bitrise).
   - **<NT>Bitrise</NT> <NT>Workspace</NT> ID**: [Identifying Workspaces and apps with their slugs](/en/bitrise-ci/api/identifying-workspaces-and-apps-with-their-slugs).
   - **Controller <NT>Log</NT> Group Class** : The <NT>Cloud Controller</NT> generates logs that are accessible to the customers in <NT>AWS</NT>. This configuration specifies what features the customer wants to use during log analysis. The default is **<NT>Infrequent access</NT>**, which provides a limited log analyzing capability at a lower cost.
   - **Controller <NT>Log</NT> Retention In Days** : Determines how long <NT>AWS</NT> stores the <NT>Cloud Controller</NT> logs.
   - **Controller <NT>SSH Key</NT>** : Select one previously configured SSH key. With this key, you can log into the created instance and, if needed, investigate.
   - **<NT>Subnet ID</NT>s** : Choose at least two subnets in two different Availability Zones from the previously configured subnets, but make sure not to select more than one subnet from the same Availability Zone. The selected subnets should belong to the selected VPC. The subnets should have a route table configuration enabling the created instances to communicate to the <NT>Bitrise</NT> Services. There is no incoming communication required for the Controller.
   - **VPC CIDR <NT>Block</NT>** : <NT>Default value</NT> is provided. Make sure you select a CIDR Block matching the VPC’s CIDR Block. For more info, see [the official AWS docs](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-cidr-blocks.html).
   - **VPC ID** : Select from the previously configured VPCs.

   :::note[Only one controller per VPC/region]

   We do not recommend running more than one Controller in one VPC/one region. These Controllers could start more resources than necessary which can cost money.

   :::
1. The other stack option configurations are optional. For details, check [AWS’s official documentation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/cfn-console-add-tags.html).
1. Review your previous settings on the **<NT>Review</NT>** screen. If everything looks correct, acknowledge your CloudFormation capabilities at the bottom and then submit your request.
1. Once the <NT>AWS</NT> CloudFormation successfully creates the instance running the Controller (this process can take 5-7 minutes), go back to your <NT>Workspace</NT> page on <NT>Bitrise</NT> to confirm the connectivity between the instance and <NT>AWS</NT>. If the Controller’s status is **<NT>Connected</NT>**, the connection works correctly.

   ![controller-ready.png](/img/_paligo/uuid-85d4125e-c0ef-ea64-d916-d04725b00581.png)

## Enabling CloudWatch <NT>Bitrise</NT> Agent logging

Setting up CloudWatch is optional, but we recommend doing so: it enables better build troubleshooting capabilities. <NT>Bitrise</NT> can provide you with faster support in case of an issue if you share the <NT>Bitrise Agent logs</NT> from your CloudWatch setup.

To enable your <NT>AWS</NT> resources, such as <NT>EC2</NT> instances, to send logs to CloudWatch, you must configure it in [AWS CloudFormation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/Welcome.html). You need the following settings:

- **CreatedBitriseAgentLogs**: When enabled, this setting initiates the creation of a <NT>CloudWatch log group</NT> for the log stream, along with the required role and an instance profile named `bitrise-agent-log-instance-profile`.
- **<NT>Bitrise Agent Logs</NT> Group Class**: The <NT>Bitrise</NT> Agent generates logs that are accessible to the customers in <NT>AWS</NT>. This configuration specifies what features the customer wants to use during log analysis. The default is **<NT>Infrequent access</NT>**, which provides a limited log analyzing capability at a lower cost.
- **<NT>Bitrise Agent Logs</NT> Retention In Days** (required): The days until <NT>AWS</NT> stores the <NT>Bitrise Agent logs</NT>.
