The <NT>Cloud Controller</NT> enables you to enjoy the benefits of a <NT>Bitrise</NT>-managed infrastructure on your <NT>AWS</NT> environment. With the <NT>Cloud Controller</NT>, you can manage your build infrastructure even without deep knowledge about <NT>AWS</NT>.


## <NT>AWS</NT> offering types

There are two basic types of <NT>Bitrise</NT> on <NT>AWS</NT> AMIs:

- **Bare metal**: Your builds run directly on an <NT>AWS</NT> machine. This means that the build environment is persistent, the same way it would be on your own computer. Artifacts or cache items from a build can have an effect on subsequent builds, unless you clean up the environment.
- **Virtualized/VM-based:** Just like on <NT>Bitrise</NT>, every build runs in its own virtual machine that is destroyed once the build is finished. It is powered by <NT>Bitrise</NT>’s virtualization solution that has run millions of builds on the <NT>Bitrise</NT>-managed machines built on top of <NT>Apple</NT>’s virtualization framework. Our macOS offerings, including the virtualized solution, are only available on <NT>Apple</NT> silicon machines.

:::important[Updating the AMIs]

The <NT>Bitrise</NT> AMIs are constantly updated with the <NT>latest version</NT>s of the pre-installed tools. However, providing the latest <NT>Xcode</NT> and <NT>Android</NT> Studio versions in the form of AMIs are slower than on the <NT>Bitrise</NT>-managed machines, because of the <NT>AWS</NT> AMI review process. You can check out the currently available versions on the [AWS stack reports](https://stacks.bitrise.io/stack_reports/aws/).

:::

:::note[<NT>AWS</NT> pricing for Mac dedicated hosts]

For macOS instances, a dedicated host must be allocated. On-Demand <NT>Amazon EC2</NT> Mac Dedicated Hosts have a minimum host allocation and billing duration of 24 hours. For more details, check out the official <NT>AWS</NT> documentation: [Amazon EC2 Dedicated Hosts Pricing](https://aws.amazon.com/ec2/dedicated-hosts/pricing/#Pricing_for_Dedicated_Hosts).

:::

### Bare metal macOS

[The bare metal macOS AMI](https://aws.amazon.com/marketplace/pp/prodview-gvebicfl7c37w) is only available on <NT>Apple</NT> silicon machines, on the following <NT>Amazon EC2</NT> Mac <NT>instance type</NT>s:

- `mac-m4.metal`
- `mac-m4pro.metal`
- `mac2.metal`
- `mac2-m2.metal`
- `mac2-m2pro.metal`

<NT>Android</NT> emulators are supported on all the <NT>instance type</NT>s above.

### Virtualized macOS

[The virtualized macOS AMI](https://aws.amazon.com/marketplace/pp/prodview-aqc5tyfdeozky) is only available on <NT>Apple</NT> silicon machines, on the following <NT>Amazon EC2</NT> Mac <NT>instance type</NT>s:

- `mac-m4.metal`
- `mac-m4pro.metal`
- `mac2.metal`
- `mac2-m2.metal`
- `mac2-m2pro.metal`

Virtualization offers an ephemeral build environment: the virtual machine is destroyed at the end of every build. This means that all builds run in a completely clean environment, and previous builds have no effect on them.

You have the option of running either one or two VMs per build. Depending on how resource intensive a build is, running two VMs can mean significant savings in hardware costs.

:::caution[No <NT>Android</NT> emulation]

<NT>Android</NT> emulation is not available as the <NT>Apple</NT> silicon architecture doesn't support nested virtualization.

:::

### Bare metal <NT>Linux</NT>

[The bare metal Linux AMI](https://aws.amazon.com/marketplace/pp/prodview-7h5yynaygkhls) is available on the following <NT>Amazon EC2</NT> <NT>Linux</NT> <NT>instance type</NT>s:

- `t2.2xlarge`
- `t2.xlarge`
- `c5n.metal`
- `c5.xlarge`
- `c5.metal`
- `c5.4xlarge`
- `c5.2xlarge`

Different <NT>instance type</NT>s suit different computing needs. To find out more about the functions and capabilities of the different <NT>instance type</NT>s, check out [the official AWS documentation](https://aws.amazon.com/ec2/instance-types/).

You can run <NT>Android</NT> emulators on the `metal` <NT>instance type</NT>s only. The other <NT>instance type</NT>s don't support the use of emulators.


## When to choose the <NT>AWS</NT> option

If you need to run <NT>Bitrise</NT> builds on infrastructure that you control, we offer two options: using <NT>Amazon EC2</NT> Mac or <NT>Linux</NT> instances or our on-premise option.

We recommend using <NT>Amazon EC2</NT> instances to run <NT>Bitrise</NT> builds if you or your company already has an <NT>AWS</NT> account to operate important infrastructure. Using an <NT>Amazon EC2</NT> instance comes with several advantages:

- Plug and play: we provide the Amazon Machine Image (AMI) as a mobile-specific build environment, with all required tools preinstalled. No specific setup needed: you only need to configure the <NT>EC2</NT> Mac or <NT>Linux</NT> instance.
- Control your environment: you can configure network and storage settings during the launch of the instance (for example, you can configure a separate subnet for your <NT>Bitrise</NT> builds) to make sure all <NT>build machines</NT> run in their own <NT>AWS</NT> environment and operate according to company security policies.
- Security: your code base will remain entirely in your control.

  :::caution[Information leaving the <NT>AWS</NT> environment]

  Using certain features means leaving the <NT>AWS</NT> environment: for example, using the <NT>build cache</NT>, generating build artifacts or test results.

  :::

If you don't have an <NT>AWS</NT> account, use some other cloud provider, or already manage your own machines, we recommend checking out [our on-premise runner offering](/en/bitrise-platform/infrastructure/running-bitrise-builds-on-premise).


## <NT>Cloud controller</NT> overview

The <NT>Cloud Controller</NT> enables you to enjoy the benefits of a <NT>Bitrise</NT>-managed infrastructure on your <NT>AWS</NT> environment. With the <NT>Cloud Controller</NT>, you can manage your build infrastructure even without deep knowledge about <NT>AWS</NT>.

The <NT>Cloud Controller</NT> enables <NT>Bitrise</NT> on <NT>AWS</NT> users to operate <NT>Amazon EC2</NT> infrastructure at scale:

- The automation supports both the bare metal and the VM-based macOS offerings, providing complete automation on reserving macOS dedicated hosts, starting the instances with the chosen AMI, and connecting to the <NT>Bitrise</NT> pool.
- Controller's automation saves your mobile DevOps team time and removes an error-prone manual process that doesn't scale above a certain number of instances.
- Changing the number of instances in the Agent Pool definition enables <NT>Amazon EC2</NT> cost savings. Scaling down even a bigger macOS machine pool for the weekend will be as easy as changing a single number in the configuration.
- Updating the machines to a newer build environment - provided in the form of an AMI by <NT>Bitrise</NT> - will be seamless and won't disturb the mobile developers. The configuration can define the percentage of machines affected by the update at any given time.
- You can monitor the instances in each <NT>Bitrise</NT> pool, see their state, and the builds they are currently running.

:::important[No inbound traffic required]

We’ve created the <NT>Bitrise</NT> on <NT>AWS</NT> offering with the highest security standards in mind, to match even the most strict company policy requirements. In all cases the Controller and the <NT>build machines</NT> will initiate network calls toward the <NT>Bitrise</NT> control plane, so no external inbound traffic is required.

:::

## Air-gapped mode for the controller

In order to provide the core functionalities of <NT>Bitrise</NT> in <NT>AWS</NT> environment, both the controller and the <NT>build machines</NT> need to access the <NT>Bitrise</NT> Control Plane on three addresses:

- https://cloud-controller-aws-internal.services.bitrise.io
- https://exec.bitrise.io
- https://build-log.services.bitrise.io

Some additional quality-of-life features (like the <NT>build machine</NT> self-update or the external IP collection) require further communication with additional resources on different addresses. To achieve the maximum security in <NT>AWS</NT>, you can disable these extra functionalities by enabling the **<NT>Air-gapped network</NT> mode** during controller creation.

Please note that this setting doesn't affect your <NT>AWS</NT> configuration. You need to ensure that the network you use is configured securely.
