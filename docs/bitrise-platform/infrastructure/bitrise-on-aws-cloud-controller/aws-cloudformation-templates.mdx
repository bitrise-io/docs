[AWS CloudFormation](https://aws.amazon.com/cloudformation/) simplifies the one-click provisioning and management of the <NT>Bitrise</NT> <NT>Cloud Controller</NT> on <NT>AWS</NT>. When the CloudFormation template is deployed into a customer VPC, it creates the <NT>Bitrise</NT> <NT>Cloud Controller</NT> and the accompanying infrastructure.

![cloudformation.png](/img/_paligo/uuid-f52635e0-c0c6-f09a-042a-4d8e785fb35c.png)

<NT>Bitrise</NT> maintains CloudFormation templates for different use cases:

- **Deploy <NT>Cloud Controller</NT> to an existing VPC:** Choose this template if the user has an established <NT>AWS</NT> presence with already built-out networking.
- **Deploy <NT>Cloud Controller</NT> into a brand new VPC**: Choose this template if the user is new to <NT>AWS</NT> or doesn’t have an already built-out networking. This template creates a new VPC first (based on customer preferences) and then deploys the controller into it.
- **Deploy Airgapped <NT>Cloud Controller</NT> to an existing VPC**: Choose this template if the user has an established <NT>AWS</NT> presence with already built-out networking and would like to run the controller in a strictly [airgapped environment](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/aws-cloudformation-templates). This is not fully automated: some [manual configuration is needed](#deploy-airgapped-cloud-controller-to-an-existing-vpc).
- **Deploy Airgapped <NT>Cloud Controller</NT> into a brand new VPC**: Choose this template if the user is new to <NT>AWS</NT> or doesn’t have an already built-out networking and would like to run the controller in a strictly [airgapped environment](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/aws-cloudformation-templates). This template creates a new VPC first with the correct airgapped configuration and then deploys the controller into it.

## Deploy CloudController to an existing VPC

Template parameters:

- **<NT>Stack Name</NT>** (required): Identifies the <NT>Bitrise</NT> <NT>Cloud Controller</NT> parent stack within the <NT>AWS</NT> Account.
- **<NT>Latest AMI ID</NT>**: This AMI ID is used as a base AMI to run the <NT>Cloud Controller</NT> on.
- **BitriseControllerToken** (required): The controller token the user receives when creating a controller on the <NT>Bitrise Website</NT>.
- **BitriseWorkspaceID** (required): The <NT>workspace</NT> ID belonging to the customer’s <NT>Bitrise</NT> account.
- **ControllerLogGroupClass** (required): The controller saves error logs to a <NT>CloudWatch log group</NT> within the customer <NT>AWS</NT> Account. The customer controls which [Log Group Class](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/CloudWatch_Logs_Log_Classes.html) fits their budget the most. The <NT>default value</NT> should be suitable for most of the cases. Default: `INFREQUENT_ACCESS`.
- **ControllerLogRetentionInDays** (required): The number of days CloudWatch should retain the Controller error logs. Default: 7 (days).
- **ControllerSshKey** (required): Provides SSH access for the <NT>Cloud Controller</NT> instance.
- **SubnetIds** (required): At least two subnets within the same region but in different AZs. We recommend using private subnets, but public subnets work as well.
- **VpcId** (required): The VPC where the controller will be deployed.
- **VpcCidrBlock** (required): The CIDR block of the selected VPC.
- **CustomBashScript** (optional): Custom bash script to run on instance startup
- **UseHostNetwork (required)**: Use host network mode for <NT>Docker</NT> container (--net=host)

### Infrastructure provisioned by the template

- **<NT>Internal Application Load Balancer</NT>**: Servers have no traffic and only perform periodic health checks on the <NT>Cloud Controller</NT> instance. In all cases the Controller and the <NT>build machines</NT> will initiate network calls toward the <NT>Bitrise</NT> control plane, no external inbound traffic is required.
- **<NT>Autoscaling Group</NT>**: Makes sure that a healthy <NT>Cloud Controller</NT> instance is running at a time.Instance: a t2.small instance on which the <NT>Cloud Controller</NT> runs.
- **LaunchTemplate**: <NT>Cloud Controller</NT> instance configuration. Also needed for the controller self-update feature.
- **<NT>IAM role, instance profile, and policy</NT>**: Certain permissions are necessary for the controller to query the build node states.
- **<NT>AWS Secrets Manager</NT>**: The template creates two secrets, respectively, for storing the WorkspaceID and the Controller Token.
- **<NT>CloudWatch log group</NT>**: The template creates a <NT>CloudWatch log group</NT> for storing Controller error logs.
- **<NT>Security groups</NT>**: The template creates two security groups: one for the LoadBalancer and one for the instance.

## Deploy <NT>Cloud Controller</NT> to a new VPC

Template parameters:

- **<NT>Stack Name</NT>** (required): Identifies the <NT>Bitrise</NT> <NT>Cloud Controller</NT> parent stack within the <NT>AWS</NT> Account.
- **<NT>Latest AMI ID</NT>**: This AMI ID is used as a base AMI to run the <NT>Cloud Controller</NT> on.
- **BitriseControllerToken** (required): The controller token the user receives when creating a controller on the <NT>Bitrise Website</NT>.
- **BitriseWorkspaceID** (required): The <NT>workspace</NT> ID belonging to the customer’s <NT>Bitrise</NT> account.
- **ControllerLogGroupClass** (required): The controller saves error logs to a <NT>CloudWatch log group</NT> within the customer <NT>AWS</NT> Account. The customer controls which [Log Group Class](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/CloudWatch_Logs_Log_Classes.html) fits their budget the most. The <NT>default value</NT> should be suitable for most of the cases. Default: `INFREQUENT_ACCESS`.
- **ControllerLogRetentionInDays** (required): The number of days CloudWatch should retain the Controller error logs. Default: 7 (days).
- **ControllerSshKey** (required): Provides SSH access for the <NT>Cloud Controller</NT> instance.
- **EnvironmentName** (optional): Adds a prefix to each piece of <NT>Bitrise</NT>-related infrastructure with the Environment Name. In case the customer has a vast number of resources, it might come in handy to be able to distinguish <NT>Bitrise</NT>-related resources from the rest.
- **PrivateSubnet1CIDR** (required): CIDR range of the first private subnet of the new VPC. Default: 10.192.32.0/20.
- **PrivateSubnet2CIDR** (required): CIDR range of the second private subnet of the new VPC. Default: 10.192.64.0/20.
- **PublicSubnet1CIDR** (required): CIDR range of the first public subnet of the new VPC. Default: 10.192.0.0/20.
- **PublicSubnet2CIDR** (required): CIDR range of the second public subnet of the new VPC. Default: 10.192.16.0/20
- **VpcCidrBlock** (required): The CIDR block of the selected VPC.
- **CustomBashScript** (optional): Custom bash script to run on instance startup
- **UseHostNetwork (required)**: Use host network mode for <NT>Docker</NT> container (--net=host)

### Infrastructure provisioned by the template

- **<NT>Internal Application Load Balancer</NT>**: Servers have no traffic and only perform periodic health checks on the <NT>Cloud Controller</NT> instance. In all cases the Controller and the <NT>build machines</NT> will initiate network calls toward the <NT>Bitrise</NT> control plane, no external inbound traffic is required.
- **<NT>Autoscaling Group</NT>**: Makes sure that a healthy <NT>Cloud Controller</NT> instance is running at a time.
- **<NT>Instance</NT>**: a `t2.small` instance on which the <NT>Cloud Controller</NT> runs.
- **LaunchTemplate**: <NT>Cloud Controller</NT> instance configuration. Also needed for the controller self-update feature.
- **<NT>IAM role, instance profile, and policy</NT>**: Certain permissions are necessary for the controller to query the build node states.
- **<NT>AWS Secrets Manager</NT>**: The template creates two secrets, respectively, for storing the WorkspaceID and the Controller Token.
- **<NT>CloudWatch log group</NT>**: The template creates a <NT>CloudWatch log group</NT> for storing Controller error logs.
- **<NT>Security groups</NT>**: The template creates two security groups: one for the LoadBalancer and one for the instance.
- **VPC**: A standard, general-purpose VPC based on best practices. Choosing the default CIDR block results in subnets with 4096 available IP addresses. The list of VPC-related resources the template creates:

  - One VPC.
  - Two public subnets.
  - Two private subnets.
  - One NAT Gateway + one Elastic IP.
  - One Internet Gateway + one Elastic IP.
  - Routing tables.
- **<NT>Bitrise Agent Logs</NT>**: This functionality allows for seamless sending of <NT>Bitrise</NT> Agent build logs to <NT>AWS</NT> CloudWatch. It leverages CloudFormation to automatically set up the necessary IAM roles and policies, ensuring the <NT>Bitrise</NT> Agent has appropriate permissions. Moreover, it creates a specific log group in CloudWatch named `bitrise-agent-log`, facilitating organized log management and real-time analysis within the <NT>AWS</NT> environment.

### Manual configuration for the airgapped template

1. Create the following interface-type VPC endpoints using the configured subnet:

   Ensure that the private DNS is enabled for all.

   - `com.amazonaws.${AWS::Region}.autoscaling`: Used for controller self-update.
   - `com.amazonaws.${AWS::Region}.ec2`: Used for <NT>EC2</NT> instance and dedicated host management.
   - `com.amazonaws.${AWS::Region}.ecr.api`: Used for downloading controller binary from <NT>Bitrise</NT> private ECR.
   - `com.amazonaws.${AWS::Region}.ecr.dkr`: Used for downloading controller binary from <NT>Bitrise</NT> private ECR.
   - `com.amazonaws.${AWS::Region}.logs`: Used for sending controller logs to CloudWatch.
   - `com.amazonaws.${AWS::Region}.secretsmanager`: Used for accessing secrets.
1. Apply the created endpoints to the Instance Security group.

## Necessary <NT>AWS</NT> permissions and connectivity

The controller needs certain <NT>AWS</NT> permissions to perform actions on the build nodes. We did our best to limit the required permissions to a minimal scope. We even made [our CloudFormation template repository](https://github.com/bitrise-io/cloud-controller-cloudformation/blob/production/iam/roles.yaml) public to build trust. Please see the [entire list of (up-to-date) required permissions](https://github.com/bitrise-io/cloud-controller-cloudformation/blob/production/iam/roles.yaml) in the repository.

The controller requires connectivity to certain <NT>Bitrise</NT> endpoints. See more [in the controller documentation](/en/bitrise-platform/infrastructure/bitrise-on-aws--cloud-controller/creating-and-configuring-a-controller).
