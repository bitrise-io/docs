<NT><GlossTerm baseform="step">Step</GlossTerm></NT> execution containers can be defined for any <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>. You define the container in the top level of the configuration file and then refer to it in a <NT>Step</NT> or a [Step bundle](/en/bitrise-ci/workflows-and-pipelines/steps/step-bundles). The <NT>Step</NT> or the <NT>Step bundle</NT> will run in the referred execution container. Within the same <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT> or <NT>Pipeline</NT>, you can run different <NT>Steps</NT> and <NT>Step bundles</NT> in different execution containers.

:::important

Containers apply to individual <NT>Steps</NT> and <NT>Step bundles</NT>, not to entire <NT>Workflows</NT> or <NT>Pipelines</NT>.

:::

To define the execution container, you need to set the following:

- The ID of the container. It will be used to reference this container.
- The type of the container: `execution`.
- The name and version of the <NT>Docker</NT> image you want to use.

  :::important[Getting images]

  You can use any public <NT>Docker</NT> image from [Docker Hub](https://hub.docker.com/).

  :::

After the containers are defined, you can refer to them in a <NT>Step</NT> or <NT>Step bundle</NT> which will then run in the referred container during your build.

## File sharing across containers

To enable file sharing across different containers and the host <NT>Bitrise</NT> environment, the following folders are shared each time you run a <NT>Docker</NT> container on <NT>Bitrise</NT>:

- `/bitrise`
- `/root/.bitrise:/root/.bitrise/`
- `tmp:/tmp`

By default, <NT>Bitrise</NT> will use `/bitrise/src` as its working directory, and everything created in either of these folders will be available across all <NT>Step</NT> execution containers.

:::important[<NT>Step</NT> execution containers only]

This applies only to <NT>Step</NT> execution containers. Volumes and file sharing with [service containers](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/service-containers) are not supported.

:::


## Defining an execution container

Use the `container` property to define an execution container. You can define them either on the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> or directly in your <NT><GlossTerm baseform="configuration yaml">configuration YAML</GlossTerm></NT> file.

When you have at least one execution container defined, you can run <NT>Steps</NT> and <NT>Step bundles</NT> in them: [Running an execution container](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/execution-containers#running-an-execution-container).

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. <Partial_OpeningTheWorkflowEditor />
1. On the left, select **<NT>Containers</NT>**.
1. Go to the **<NT>Execution containers</NT>** tab and then click **<NT>Add container</NT>**.

   ![SCR-20260313-pnzl.png](/img/_paligo/uuid-35998534-7cfb-c0c6-7277-bb12dd6bc345.png)
1. Set the required properties for the container. They are:

   - **<NT>Unique ID</NT>**: This ID is used to refer to the container in your configuration YAML file.
   - **<NT>Image</NT>:** The name and version of the <NT>Docker</NT> image. You can use any image found on [Docker Hub](https://hub.docker.com/) or you can use other registries.

     For a <NT>Docker</NT> Hub image, use the `[name]:[version]` format. For other registries, use `[registry server]/[owner]/[name]:[version]`.
1. Optionally, add port mappings in the **<NT>Ports</NT>** field in the `[HostPort01]:[ContainerPort01]` format.

   Read more about port publishing and port mapping in [Docker's official documentation](https://docs.docker.com/engine/network/port-publishing/).
1. Click **Show <NT>more options</NT>** to access <NT>Docker</NT> authentication credentials, <NT>Environment Variables</NT> for your container, and <NT>Docker create options</NT>.
1. Optionally, set up your <NT>Docker</NT> credentials.

   The credentials are used for the `docker login` command. You can set up:

   - **<NT>Registry server</NT>**: It should be a fully qualified registry server URL. This is optional if the server is already part of the image reference.
   - **<NT>Username</NT>**.
   - **<NT>Password</NT>**.

   :::tip

   You must use [secrets](/en/bitrise-ci/configure-builds/secrets) to pass your <NT>Docker</NT> credentials to your configuration.

   :::
1. Optionally, add options in the **<NT>Docker create options</NT>** field.

   These are <NT>Docker</NT> container resource options: parameters that will be passed to the `docker container create` command.

   ![SCR-20260313-pruq.png](/img/_paligo/uuid-683647fc-3131-1c5a-8667-2fa719d617cd.png)

   For a list of all available options, see [the Docker documentation](https://docs.docker.com/reference/cli/docker/container/create/#options).
1. When done, click **<NT>Create</NT> container**.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Add the `containers` property to the top level of your configuration YAML.

   :::tip[YAML syntax reference]

   For detailed YAML syntax reference, see [Docker container properties](/en/bitrise-ci/references/configuration-yaml-reference#docker-container-properties).

   :::
1. Set an ID for your container.

   It must be unique within the configuration. The ID is used to refer to the container in <NT>Steps</NT> or <NT>Step bundles</NT>.

   ```yaml
   containers:
     node-18:
   ```
1. Set the required properties for the container. They are:

   - The name and version of the <NT>Docker</NT> image, in a `name:version` format. You can use any image found on [Docker Hub](https://hub.docker.com/).
   - The container type: `execution`.

   ```yaml
   containers:
     node-21:
       type: execution
       image: node:21.6
     node-18: 
       type: execution
       image: node:18
   ```
1. Optionally, set up a port mapping in the `[HostPort01]:[ContainerPort01]` format.

   Read more about port publishing and port mapping in [Docker's official documentation](https://docs.docker.com/engine/network/port-publishing/).

   ```yaml
   containers:
     node-21:
       type: execution
       image: node:21.6
       ports: 
       - 3000:3000
   ```
1. Optionally, set up your <NT>Docker</NT> credentials.

   The credentials are used for the `docker login` command. You can set up:

   - A server: It should be a fully qualified registry server URL. This is optional if the server is already part of the image reference.
   - A username.
   - A password.

   :::tip

   We recommend using secrets to pass your <NT>Docker</NT> credentials to your configuration.

   :::

   ```yaml
   containers:
     node-21:
       type: execution
       image: node:21.6
       credentials:
         username: $DOCKER_USERNAME
         password: $DOCKER_PASSWORD
         server: us-central1-docker.pkg.dev
   ```
1. Optionally, use the `options` property to configure additional <NT>Docker</NT> container resource options: parameters that will be passed to the `docker container create` command.

   In this example, the service is configured to have [healthchecks](https://docs.docker.com/engine/reference/run/#healthchecks).

   ```yaml
   containers:
     node-21:
       type: execution
       image: node:21.6
       options: >-
         --health-cmd pg_isready
         --health-interval 10s
         --health-timeout 5s
         --health-retries 5
   ```

   For a list of all available options, see [the Docker documentation](https://docs.docker.com/reference/cli/docker/container/create/#options).

</TabItem>
</Tabs>


## Running an execution container

You can run a <NT>Step</NT> or a <NT>Step bundle</NT> in an execution container. To do so, refer to [a defined container](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/execution-containers#defining-an-execution-container) in the <NT>Step</NT> or <NT>Step bundle</NT> configuration.

You can:

- Refer to the container in a <NT>Step</NT> or a <NT>Step bundle</NT> within a <NT>Workflow</NT>. When that <NT>Workflow</NT> runs, the <NT>Step</NT> or the <NT>Step bundle</NT> will run within that container.
- [Refer to the container in the Step bundle definition](#running-an-execution-container). When that <NT>Step bundle</NT> is added to the <NT>Workflow</NT>, it will run within that container by default, unless you override it within the <NT>Workflow</NT>.

This guide covers both options.

:::note[Container nesting]

You can define containers on different levels: <NT>Steps</NT> and <NT>Step bundles</NT> both support their own container configuration. This creates a hierarchy between different levels. For the principles governing the hierarchy, see [Container nesting rules](/en/bitrise-platform/infrastructure/docker-containers-on-bitrise/about-docker-containers-on-bitrise#container-nesting-rules).

:::

<Tabs>
<TabItem value="workflow-editor" label="Workflow Editor" default>

1. Open the <NT>Workflow Editor</NT>.
1. Select a <NT>Step</NT> or a <NT>Step bundle</NT> in a <NT>Workflow</NT>.

   Alternatively, you can select **<NT>Step Bundles</NT>** on the left navigation menu and set a container in the <NT>Step bundle</NT> definition. The subsequent steps in the procedure are the same.

   :::important[Bundle definition override]

   By default, if you set a container in the <NT>Step bundle</NT> definition the <NT>Step bundle</NT> will run in the referred container in any <NT>Workflow</NT> you add it to.

   However, if you set a container in a <NT>Step bundle</NT> instance (the <NT>Step bundle</NT> within a <NT>Workflow</NT>), it overrides the container set in the <NT>Step bundle</NT> definition.

   :::
1. Select **<NT>Containers</NT>**.
1. Under **Execution Container**, click **<NT>Add container</NT>** and select a container from the menu.
1. If your container is already running but you want to run a clean instance of the container, check the **<NT>Recreate container</NT>** option.

   By default, the <NT>Step</NT> or <NT>Step bundle</NT> will use an already running container if there is one.

</TabItem>
<TabItem value="configuration-yaml" label="Configuration YAML">

1. Open your configuration YAML file and find your <NT>Workflow</NT>.
1. Add the `execution_container` property with a container name to the <NT>Step</NT> or <NT>Step bundle</NT> you need. You can only refer to a single execution container.

   Add an execution container to a <NT>Step</NT> within a <NT>Workflow</NT> format:

   ```yaml
   workflows:
     ci:
       steps:
       - git-clone: {}
       - script:
           execution_container: node-21
   ```

   Add an execution container to a <NT>Step bundle</NT> within a <NT>Workflow</NT>:

   ```yaml
   workflows:
     ci:
       steps:
       - git-clone: {}
       - script: {}
       - bundle::test-bundle-id:
           execution_container: node-21
   ```

   Add the execution container to a <NT>Step bundle</NT> definition:

   ```yaml
   step_bundles:
     test-bundle-id:
       steps:
       - git-clone@8: {}
       - restore-cache@2: {}
       execution_container: test-container
   ```

   :::important[Bundle definition override]

   By default, the <NT>Step bundle</NT> will run in the referred container in any <NT>Workflow</NT> you add it to. However, if you set a container for the <NT>Step bundle</NT> within a <NT>Workflow</NT>, it overrides the container set in the <NT>Step bundle</NT> definition.

   :::
1. If your container is already running but you want to run a clean instance of the container, set the `recreate` property to `true` .

   By default, the <NT>Step</NT> or <NT>Step bundle</NT> will use an already running container if there is one. Use the property to change the default behavior.

   ```yaml
   workflows:
     ci:
       steps:
       - git-clone: {}
       - script:
           execution_container:
           node:21
       - bundle::test-bundle-id:
           execution_container:
             node-21:
               recreate: true
   ```

</TabItem>
</Tabs>
