<NT>Bitrise</NT> supports automatic user and group provisioning via [SCIM 2.0](https://scim.cloud/) with Microsoft <NT>Entra ID</NT>. Once configured, <NT>Entra ID</NT> becomes the authoritative source for <NT><GlossTerm baseform="workspace">workspace</GlossTerm></NT> membership: users and group assignments are pushed to <NT>Bitrise</NT> automatically, and removing a user in Entra deactivates their access on <NT>Bitrise</NT>.

## SCIM requirements

- SCIM provisioning requires an **<NT>Enterprise</NT> plan**.
- You must have [SAML SSO configured](/en/bitrise-platform/accounts/saml-sso-in-bitrise/setting-up-azure-ad-sso-for-bitrise) between <NT>Bitrise</NT> and <NT>Entra ID</NT> before enabling SCIM.
- You must have a [verified domain](/en/bitrise-platform/accounts/saml-sso-in-bitrise/configuring-saml-sso-on-bitrise#verifying-your-domain). Domain verification can take up to 72 hours. SCIM credentials cannot be generated until at least one domain is verified.
- You will need both the [Bitrise workspace owner](/en/bitrise-platform/workspaces/collaboration-and-permissions-in-workspaces/roles-and-permissions-in-workspaces) and an <NT>Entra ID</NT> administrator available during setup.

## Setting up SCIM 

The setup consists of three steps:

1. [Generating SCIM credentials on Bitrise.](#generating-scim-credentials-on-bitrise)
1. [Configuring SCIM provisioning in Entra ID.](#configuring-scim-provisioning-in-entra-id)
1. [Verifying the connection.](#verifying-the-connection)

### Generating SCIM credentials on <NT>Bitrise</NT>

1. <Partial_OpeningTheWorkspaceSettingsPage />
1. Select **<NT>Single sign-on</NT>** from the left menu.
1. Select the **SCIM** tab.
1. Click **Generate <NT>SCIM credentials</NT>**.
1. Copy and save both values shown in the dialog:
   - **<NT>SCIM base URL</NT>**
   - **<NT>SCIM authentication token</NT>**

   The token is not displayed again after you close the dialog. If you lose it, you can regenerate it, but doing so immediately invalidates the previous token.

### Configuring SCIM provisioning in <NT>Entra ID</NT>

<NT>Bitrise</NT> is not a gallery app in <NT>Entra ID</NT>. Use Microsoft's [tutorial for configuring SCIM with non-gallery applications](https://learn.microsoft.com/en-us/entra/identity/app-provisioning/use-scim-to-provision-users-and-groups) and apply the following <NT>Bitrise</NT>-specific values:

- **Tenant URL:** [your SCIM base URL](#generating-scim-credentials-on-bitrise).
- **<NT>Secret</NT> token:** [your SCIM authentication token](#generating-scim-credentials-on-bitrise).
- **Unique identifier field:** `userName` (mapped to the user's email address).

When configuring **Provisioning Mode**, select **<NT>Automatic</NT>**. Use **<NT>Sync</NT> only assigned users and groups** to control which users and groups are pushed to <NT>Bitrise</NT>.

:::important[Enable <NT>Push Groups</NT>]

<NT>Workspace</NT> membership on <NT>Bitrise</NT> is managed through groups. You must enable group provisioning (<NT>Push Groups</NT>) so that <NT>Entra ID</NT> can manage group membership in <NT>Bitrise</NT>. Without it, users will be provisioned but not added to any group.

:::

### Verifying the connection

Click **<NT>Test Connection</NT>** in the <NT>Entra ID</NT> provisioning configuration. This queries the <NT>Bitrise</NT> `/ServiceProviderConfig` endpoint and confirms that the credentials are valid. A successful test means <NT>Entra ID</NT> can reach <NT>Bitrise</NT> and authenticate correctly.

## Attribute mapping

<NT>Bitrise</NT> implements the SCIM 2.0 Core User Schema and Core Group Schema (RFC 7643/7644). <NT>Entra ID</NT>'s default attribute mapping works without custom transforms.

Supported user attributes:

- `userName`: unique identifier for the user; <NT>Entra ID</NT> maps this to the user's User Principal Name (UPN), which typically matches their email address
- `emails`: the user's email address(es)
- `name.givenName`: <NT>first name</NT>
- `name.familyName`: <NT>last name</NT>
- `active`: whether the user is active; set to `false` to deprovision
- `externalId`: the user's stable, opaque ID assigned by <NT>Entra ID</NT>; remains constant even if the user's UPN changes

Supported group attributes:

- `displayName`: the group's name in <NT>Entra ID</NT>
- `members`: the list of users in the group

For <NT>workspace</NT> role assignment via the `roles` attribute, see [Managing workspace roles via SCIM](/en/bitrise-platform/accounts/saml-sso-in-bitrise/configuring-scim#managing-workspace-roles-via-scim).

## Group behavior

**<NT>Name</NT> matching:** When <NT>Entra ID</NT> pushes a group, <NT>Bitrise</NT> links it to an existing group with the same name, or creates a new one. Matching is case-sensitive and exact. `ios_dev` and `iOS Dev` are treated as different groups. Make sure group names in <NT>Entra ID</NT> match your existing <NT>Bitrise</NT> group names exactly before enabling sync.

**<NT>Entra ID</NT> becomes authoritative for membership:** Once <NT>Entra ID</NT> is syncing a group, it controls that group's membership. Any members added manually in <NT>Bitrise</NT> who are not in the corresponding <NT>Entra ID</NT> group will be removed on the next sync.

**<NT>Protected</NT> groups:** Global Access groups and SAML default groups cannot be deleted via SCIM. Keep these groups out of <NT>Entra ID</NT>'s sync scope to avoid unintended membership changes.

## User provisioning behavior

**No email activation step:** Users provisioned via SCIM on a verified domain are created in a confirmed, active state immediately. They do not receive an email activation link from <NT>Bitrise</NT>. On first login they are redirected directly to the <NT>Entra ID</NT> SSO flow.

**<NT>Default</NT> <NT>workspace role</NT>:** All SCIM-provisioned users receive the <NT>Workspace</NT> Viewer role by default, which does not include <NT>Bitrise CI</NT> product access. To assign a different role at provisioning time, configure the `roles` attribute in <NT>Entra ID</NT>. See [Managing workspace roles via SCIM](/en/bitrise-platform/accounts/saml-sso-in-bitrise/configuring-scim#managing-workspace-roles-via-scim).

## Deprovisioning behavior

When you remove a user's assignment in <NT>Entra ID</NT>, or <NT>Entra ID</NT> sends a `PATCH active:false` or `DELETE` request:

- The user is removed from the <NT>workspace</NT> and from all their groups within it simultaneously.
- Their personal access tokens remain active but lose access to all resources in the <NT>workspace</NT>.
- Their workflows and secrets are preserved and not reassigned automatically.
- The seat is freed immediately.

**Re-provisioning a deprovisioned user:** A deprovisioned user's account is not deleted; they are removed from the <NT>workspace</NT>. Re-provisioning them via SCIM with the same email is supported, once they are assigned again in <NT>Entra ID</NT>, it can manage their group membership via SCIM as normal.

## Migrating existing <NT>workspace</NT> members

SCIM is push-based. <NT>Bitrise</NT> only acts on requests <NT>Entra ID</NT> sends. Existing <NT>workspace</NT> members who are not assigned in <NT>Entra ID</NT> are not touched and remain in the <NT>workspace</NT>.

To migrate an existing <NT>workspace</NT> with members and groups:

1. Verify all relevant email domains before enabling SCIM.
1. Align <NT>Entra ID</NT> group names to match your existing <NT>Bitrise</NT> group names exactly.
1. Enable SCIM with **<NT>Sync</NT> only assigned users and groups** in <NT>Entra ID</NT>.
1. Assign a small pilot group first to validate end-to-end, then expand incrementally.

There is no dry-run or preview mode. The incremental pilot approach is the practical equivalent.

## Users on unverified domains

Users whose email is on a domain that has not been verified in <NT>Bitrise</NT> cannot be managed via SCIM. Any attempt to provision or modify them will be rejected. Exclude those users from <NT>Entra ID</NT>'s assignment scope until their domain is verified.

To verify additional domains, go to <NT>Workspace Settings</NT> and add each domain under **<NT>Domain</NT> verification**. Each domain must be verified separately.
