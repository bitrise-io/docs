Roles determine the permissions of the <NT><GlossTerm baseform="workspace">workspace</GlossTerm></NT> members. There are two main types of roles that can be assigned to <NT>workspace</NT> members:

- **<NT>Workspace</NT>-level roles**: These roles determine what actions a user can perform within a <NT>workspace</NT>. <NT>Workspace</NT>-level roles can be assigned to individual members of a <NT>workspace</NT>.
- **Project-level roles**: These roles determine what actions a user can perform on a project. Project-level roles can be assigned to <NT>workspace</NT> groups. The roles are different based on the product: <NT>Bitrise CI</NT> and <NT>Release Management</NT> have their own set of roles.


## <NT>Workspace</NT>-level roles

There are four main roles in a <NT>workspace</NT>:

- **<NT>Owner</NT>**: The <NT>workspace</NT> owner has access to all CI projects and <NT>Release Management</NT> apps and have full control over them.
- **<NT>Manager</NT>**: The user can access and modify <NT>workspace settings</NT> such as connected service accounts, can manage members but can't access billing details and can't delete the <NT>workspace</NT>. <NT>Workspace</NT> Managers can create new <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>s. Within projects they have created, they can create CI configurations and/or <NT>Release Management</NT> <NT>connected apps</NT>. They cannot access or modify CI configurations or <NT>connected apps</NT> in <NT>Bitrise</NT> <NT>project</NT>s that they haven’t been given direct access to via user or group access settings.
- **<NT>Contributor</NT>**: The user can't access <NT>workspace settings</NT> and can't add new members or manage existing members. 
- **<NT>Viewer</NT>**: The user can't add new projects or <NT>Release Management</NT> apps to the <NT>workspace</NT>.

These roles mainly determine the user's permissions within the <NT>workspace</NT>. However, in the case of the <NT>workspace</NT> owners and the <NT>workspace</NT> managers, they provide additional permissions on the project level, as described above. But project access is mostly determined by project-level roles.


## Project-level roles

You can assign roles on different projects to members, <NT>workspace</NT> groups, and outside contributors. You can do so either when:

- Creating a group.
- Modifying a group's settings.
- Modifying a member's details.

Project-level roles are only relevant to a specific project. The exact roles and their permissions depend on the product: both <NT>Bitrise CI</NT> and <NT>Release Management</NT> have their own roles and permissions:

- [Bitrise CI roles and permissions](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).
- [Release Management roles and permissions](/en/release-management/configuring-connected-apps/release-management-roles-and-permissions).

Both individual members and <NT>workspace</NT> groups can be assigned roles on the <NT>workspace</NT>'s projects. You can set up:

- **<NT>Admin access</NT>**: The user or group can access to both <NT>Bitrise CI</NT> and <NT>Release Management</NT> on the <NT>selected projects</NT>, with the project admin role.
- **<NT>Bitrise CI</NT> access**: The user or group can access <NT>Bitrise CI</NT> configurations of the <NT>selected projects</NT>. They can be assigned different roles on different projects.
- **<NT>Release Management</NT> access**: The user or group can access <NT>Release Management</NT> apps of the <NT>selected projects</NT>.

  :::note[New projects and <NT>Release Management</NT> apps]

  If a new project is added to a <NT>workspace</NT>, users won't automatically become members of the project.

  Similarly, if a new <NT>Release Management</NT> app is added to an existing project, the user or group won't automatically have access to it just because they have access to other apps of the project.

  :::


## Managing roles and permissions in a <NT>workspace</NT>

You can update the roles and permissions assigned to any <NT>workspace</NT> member or <NT>workspace</NT> group at any time.

### Updating <NT>workspace</NT> roles for a member

1. <Partial_OpeningTheWorkspaceSettingsPage />
1. On the left, select **<NT>Collaboration</NT>**.
1. Select the **<NT>Members</NT>** tab and find the <NT>workspace</NT> member you need in the list.
1. Click the options menu (⋮) next to their name and click **<NT>View details</NT>**.
1. On the <NT>workspace</NT> role card, click **<NT>Change</NT>**.
1. Select the new role and click **<NT>Save changes</NT>**.

### Managing product and project access for a member

You can, at any time, modify product and project access for individual members of a <NT>workspace</NT>. You can modify access for all the projects a user has access to, or just to a specific project.

<Tabs>
<TabItem value="all-projects" label="All projects" default>

1. <Partial_OpeningTheWorkspaceSettingsPage />
1. On the left, select **<NT>Collaboration</NT>**.
1. Select the **<NT>Members</NT>** tab and find the <NT>workspace</NT> member you need in the list.
1. Click the options menu (⋮) next to their name and click **<NT>View details</NT>**.
1. Click **<NT>Manage access</NT>**.
1. Click **<NT>Change settings</NT>** at the correct product type:

   - **<NT>Admin access</NT>**: Grants access to all products, managing all aspects of <NT>selected projects</NT> with the project admin role.
   - **<NT>Bitrise CI</NT>**: The user has access to <NT>Bitrise CI</NT> only, and it can be assigned [Bitrise CI roles](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).
   - **<NT>Release Management</NT>**: The user has access to <NT>Release Management</NT> only, and it can be assigned [Release Management roles](/en/release-management/configuring-connected-apps/release-management-roles-and-permissions).
1. <NT>Set up</NT> project access and access roles. The roles will be different based on the product.
1. When done, click **<NT>Done</NT>** in the dialog, and then click **<NT>Save changes</NT>**.

</TabItem>
<TabItem value="specific-project" label="Specific project">

1. <Partial_OpeningTheWorkspaceSettingsPage />
1. On the left, select **<NT>Collaboration</NT>**.
1. Select the **<NT>Members</NT>** tab and find the <NT>workspace</NT> member you need in the list.
1. Click the options menu (⋮) next to their name and select **<NT>View details</NT>**.
1. Select the **Projects** tab and find the project you need in the list.
1. Click the options menu (⋮) next to its name and select **<NT>Change</NT> access**.
1. Click **<NT>Change settings</NT>** at the correct product type:

   - **<NT>Admin access</NT>**: Grants access to all products, managing all aspects of <NT>selected projects</NT> with the project admin role.
   - **<NT>Bitrise CI</NT>**: The user has access to <NT>Bitrise CI</NT> only, and it can be assigned [Bitrise CI roles](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).
   - **<NT>Release Management</NT>**: The user has access to <NT>Release Management</NT> only, and it can be assigned [Release Management roles](/en/release-management/configuring-connected-apps/release-management-roles-and-permissions).
1. <NT>Set up</NT> project access and access roles. The roles will be different based on the product.
1. When done, click **<NT>Done</NT>** in the dialog, and then click **<NT>Save changes</NT>**.

</TabItem>
</Tabs>
