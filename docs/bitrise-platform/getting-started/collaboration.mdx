Collaboration is the way different people can work on <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>s together. <NT>Bitrise</NT> offers multiple levels of collaboration options, with different product solutions having their own fine-grained collaboration features.

Collaboration covers:

- Membership in <NT>Bitrise</NT> <NT><GlossTerm baseform="workspace">workspace</GlossTerm></NT>s, including <NT>workspace</NT> groups.
- Roles and permissions in <NT>Bitrise</NT> <NT>workspaces</NT>: actions that users with different access levels can perform.
- <NT>Product access</NT>: you can restrict users to access only <NT>Bitrise CI</NT> or <NT>Release Management</NT>.
- Roles and permissions in projects, including CI configurations and <NT>Release Management</NT> apps.


## Collaboration in <NT>workspaces</NT>

The main organizing entity of <NT>Bitrise</NT> is the <NT>workspace</NT>. <NT>Workspaces</NT> own projects, you can connect third-party integrations on <NT>workspace</NT> level, and you can set up collaboration in a <NT>workspace</NT>.

### <NT>Workspace</NT> owners

Each <NT>workspace</NT> has at least one owner: by default, it's the account that created the <NT>workspace</NT>. Owners have full control over all aspects of the <NT>workspace</NT>, including collaboration features.

For more information, check out [Changing the owners of a Workspace](/en/bitrise-platform/workspaces/changing-the-owners-of-a-workspace).

### <NT>Workspace</NT> members

You can add individual members to <NT>workspaces</NT>. You can assign product access and project roles to members:

- <NT>Product access</NT> means selecting the products the member can work on. For example, you can configure members to only have access to <NT>Bitrise CI</NT>, not <NT>Release Management</NT>.
- <NT>Project roles</NT> mean the roles and permissions on the specific projects owned by the <NT>workspace</NT>. You can assign different roles and permissions for different projects. Within the same project, you can assign different roles for <NT>Bitrise CI</NT> and <NT>Release Management</NT>, even if the member has access to both products.

As such, even if your <NT>workspace</NT> has hundreds of projects, you can configure fine-grained access for team members so they only see the products and projects that they need.

For more information, check out [Workspace collaboration](/en/bitrise-platform/workspaces/collaboration-and-permissions-in-workspaces/workspace-collaboration).

### <NT>Workspace</NT> groups

<NT>Workspace</NT> groups are made up of <NT>workspace</NT> members. Groups make it easy to manage product and project access in bulk. You don't need to set up specific fine-grained access for every member separately; instead, you invite a member and then add them to a group with the required permissions.

Just like members, groups can be restricted to certain products and projects. Groups can be assigned roles and permissions on projects: this means that all members in the group will be assigned those roles and permissions.

Group permissions don't override <NT>workspace</NT> member permissions. For example, let's say that:

- A user is part of a group that has Developer access to a CI project.
- The same user has Admin access to the same CI project as a <NT>workspace</NT> member.

In this case, the user will have Admin access to the project.

For more information, check out [Workspace groups](/en/bitrise-platform/workspaces/collaboration-and-permissions-in-workspaces/workspace-groups).

### Outside contributors in <NT>workspaces</NT>

Outside contributors are users who aren't members of the <NT>workspace</NT> but they are added to one or more projects that the <NT>workspace</NT> owns. These users won't have access to the <NT>workspace</NT> itself, only the assigned projects. Their roles and permissions can be configured only on the **<NT>Project settings</NT>** page.

Outside contributors have to be enabled in the **<NT>Workspace settings</NT>** page.

### <NT>Workspace</NT> roles

When inviting someone to a <NT>workspace</NT>, you can also assign a <NT>workspace</NT> role to them. <NT>Workspace</NT> roles define the users' access to the <NT>workspace</NT> itself, what actions the users can perform. For example, a user with the **<NT>Manager</NT>** role can update <NT>workspace</NT> integrations, add or remove members but can't access billing details.

For more information, check out [Roles and permissions in workspaces](/en/bitrise-platform/workspaces/collaboration-and-permissions-in-workspaces/roles-and-permissions-in-workspaces).


## Collaboration in projects

Project-level collaboration means users working together on a <NT>Bitrise</NT> <NT>project</NT>. User can have different roles and permissions within the same project. If the project has both a CI configuration and a <NT>Release Management</NT> app, you can configure separate access for those two products.

Project collaboration is defined in two ways: by adding contributors and by assigning <NT>workspace</NT> groups to the project.

### <NT>Project contributors</NT>

A project contributor is a user who is individually assigned to a project, not as part of a group. A contributor can be a member of the <NT>workspace</NT> that owns the project, or they can be an outside contributor. Outside contributors can be added only if the feature is enabled for the <NT>workspace</NT> that owns the project.

Contributors are assigned their own roles and permissions to the project. You can add <NT>workspace</NT> members as contributors to a project either from the **<NT>Workspace settings</NT>** page or the **<NT>Project settings</NT>** page. <NT>Outside contributors</NT> can only be added on the **<NT>Project settings</NT>** page.

For details, check out [Managing user access to a project](/en/bitrise-platform/projects/managing-user-access-to-a-project).

### Groups

Groups are <NT>workspace</NT> groups: you can assign <NT>workspace</NT> groups to any project owned by the <NT>workspace</NT>. On the project level, this means that every member of the group will have the same roles and permissions that the group has on the project.

You can assign <NT>workspace</NT> groups to projects either from the **<NT>Workspace settings</NT>** page or **<NT>Project settings</NT>** page.

For details, check out [Managing user access to a project](/en/bitrise-platform/projects/managing-user-access-to-a-project).

### Roles and permissions in projects

Projects have no separate roles just for project access: roles and permissions are based on product access. <NT>Bitrise CI</NT> and <NT>Release Management</NT> both have their own set of roles and permissions. Each contributor and group can be assigned separate roles for these two products.

There is one exception: you can grant a user or group full **Admin** access to a project. **Admin** access means managing all aspects of a project, with access to all products. This is effectively the same as separately granting the **Admin** role on both <NT>Bitrise CI</NT> and <NT>Release Management</NT>.

Read more in [Product access](/en/bitrise-platform/getting-started/collaboration#product-access).


## Product access

Collaboration includes roles and permissions: what actions users can perform. On <NT>Bitrise</NT>, the roles and permissions differ by product: you can configure fine-grained access separately for <NT>Bitrise CI</NT> and <NT>Release Management</NT>.

Product access is mostly independent of <NT>workspace</NT> roles and permissions. A user who is an Admin on a CI project or a <NT>Release Management</NT> app doesn't necessarily have full access to the settings of the <NT>workspace</NT> that owns the project. The exception to that is <NT>Workspace</NT> owners: all owners have full access to every aspect of the projects owned by the <NT>workspace</NT>.

### Admin access

You can grant **<NT>Admin access</NT>** to users and groups to enable them to manage all aspects of one or more <NT>selected projects</NT>. This includes access to all products, such as <NT>Bitrise CI</NT> and <NT>Release Management</NT>.

You can configure admin access for users or groups both from the **<NT>Workspace settings</NT>** page and the **<NT>Project settings</NT>** page.

For details, check out [Updating roles and permissions](/en/bitrise-platform/workspaces/collaboration-and-permissions-in-workspaces/roles-and-permissions-in-workspaces#managing-roles-and-permissions-in-a-workspace).

### <NT>Bitrise CI</NT> roles and permissions

<NT>Bitrise CI</NT> has its own set of roles and permissions that define how users can interact with the CI/CD process. Each user or <NT>workspace</NT> group can be granted access to <NT>Bitrise CI</NT> and then assigned roles on CI projects.

The specific CI roles and permissions define access to things such as <NT>project settings</NT>, <NT><GlossTerm baseform="pipeline">Pipeline</GlossTerm></NT>s, <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>s, and other important parts of CI.

For details, check out [Roles and permissions for Bitrise CI](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).

### <NT>Release Management</NT> roles and permissions

<NT>Release Management</NT> has its own set of roles and permissions. These define how users can interact with releases, test distribution, and CodePush.

In addition, <NT>Release Management</NT> has a specific collaboration feature: tester groups. A tester group is a group of users who can automatically receive installable artifacts for testing. Testers have limited permissions and they can only be selected from users who are already members of the project that the <NT>Release Management</NT> app belongs to.

For details, check out [Release Management roles and permissions](/en/release-management/configuring-connected-apps/release-management-roles-and-permissions).
