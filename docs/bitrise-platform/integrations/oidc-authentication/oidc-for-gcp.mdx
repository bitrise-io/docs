Generate OIDC tokens during your <NT>Bitrise</NT> build to exchange them for scoped access tokens with Google Cloud Platform (<NT>GCP</NT>).

To use OIDC tokens for <NT>GCP</NT>, you'll need to:

- Create a workload identity pool.
- Connect a Google service account to the identity pool.

On <NT>Bitrise</NT>, you can get the tokens by using either the [**Get OIDC Identity Token**](https://github.com/bitrise-steplib/bitrise-step-get-identity-token) <NT><GlossTerm baseform="step">Step</GlossTerm></NT> or the [**Authenticate with Google Cloud Platform (GCP)**](https://github.com/bitrise-steplib/bitrise-step-authenticate-with-gcp) <NT>Step</NT>.


## Creating a workload identity pool

Create a workload identity pool in <NT>GCP</NT> to enable Workload Identity Federation.

:::tip[Workload Identity Federation]

Read more about how authentication with Workload Identity Federation works on <NT>GCP</NT> in the [official Google documentation.](https://cloud.google.com/sap/docs/abap-sdk/on-premises-or-any-cloud/latest/authentication-wif)

:::

1. In the Google Cloud console, go to **<NT>IAM & Admin</NT>** then **Workload Identity Federation**.
1. In the **<NT>Create</NT> an identity pool** section, enter a value for the following fields:

   - **<NT>Name</NT>**: The name for the pool. The name is used as the pool ID and you can't change the pool ID later.
   - **<NT>Description</NT>**: The purpose of the pool.
1. Under **<NT>Add</NT> a provider to pool**, add <NT>Bitrise</NT> as a provider:

   - In the **<NT>Select</NT> a provider** field, select **OpenID <NT>Connect</NT> (OIDC)**.
   - **Provider name**: Enter a name. For simplicity, we recommend using `Bitrise`.
   - **<NT>Issuer</NT> URL**: https://token.builds.bitrise.io.
   - **Audience**: <NT>Select</NT> **Allowed audience**. When using the token, <NT>GCP</NT> will check if the allowed audience matches the audience encoded in the `aud` claim. On <NT>Bitrise</NT>, you can set the audience in our <NT>Steps</NT> providing OIDC services: [Fetching and exchanging tokens with GCP](/en/bitrise-platform/integrations/oidc-authentication/oidc-for-gcp#fetching-and-exchanging-tokens-with-gcp).
1. Click **<NT>Continue</NT>**.
1. Under **<NT>Configure</NT> provider attributes**, configure the required attribute mapping values:

   - The `google.subject` key is a unique identifier for the user. You can use `assertion.sub` to identify the build that requests the token.
   - Use the `attribute.NAME` format to add up to 50 custom attributes, each with a value in the `assertion.VALUE` format. You can find the list of available attributes in the <NT>Bitrise</NT> token here: [Information in the OIDC token](/en/bitrise-platform/integrations/oidc-authentication/oidc-authentication-overview#information-in-the-oidc-token).

   :::important[Explicit mapping required]

   You must map all the attributes you want to use. Google only lets you use the attributes which are explicitly mapped. Read more about [attribute mappings in Google's documentation](https://cloud.google.com/iam/docs/workload-identity-federation?_gl=1*1sz7kri*_ga*MTM0MTUyNjA1OC4xNzYyNTMwMjA5*_ga_WH2QY8WWF5*czE3NjM0Nzc1MzYkbzIkZzEkdDE3NjM0Nzc4MDEkajUwJGwwJGgw#mapping).

   :::
1. Optionally, add attribute conditions. You will also be able to filter identity tokens based on attributes directly at the pool level when creating the service account.


## Connecting a Google service account to the workload identity pool

To start using OIDC authentication for <NT>GCP</NT> services, you need to connect a workload identity pool to a Google service account. This involves:

- Creating a service account.
- Granting access to the identity pool using a service account impersonation.

### Create a service account

1. In Google Cloud console, go to **<NT>IAM & Admin</NT>** and then **<NT>Service accounts</NT>**.
1. Click **<NT>Create</NT> <NT>service account</NT>**.
1. Add a display name, a unique service account ID, and a description.
1. Click **<NT>Create</NT> and continue**.
1. Under **Permissions**, add <NT>all roles</NT> that you need.

   :::note[IAM conditions]

   Optionally, you can add IAM conditions based on the previously mapped attribute values by clicking **<NT>Add</NT> IAM condition**.

   :::
1. Click **<NT>Done</NT>**.

### Granting access to the workload identity pool

1. In Google Cloud console, go to **<NT>IAM & Admin</NT>** and then **Workload Identity Federation**.
1. Select your identity pool.
1. Click **<NT>Grant access</NT>**.
1. Select the **<NT>Grant access</NT> using service account impersonation** option.
1. Select the <NT>service account</NT> you've created.
1. Select the principals: identities that can access the service account.

   The available options are limited to the keys from the provider attribute mapping of the identity pool. For example, if you use the attribute mapping `google.subject=assertion.sub`, set **Attribute name** to `subject` and **Attribute value** to `sub`. You can see the available values in the <NT>Bitrise</NT> token here:
1. Click **<NT>Save</NT>**.
1. In the dialog, select the provider you created for <NT>Bitrise</NT> OIDC tokens.
1. Add the OIDC ID token path: https://token.builds.bitrise.io and set the format to **text**.
1. Click **<NT>Download</NT> config** and save the file somewhere you can access it.


## Fetching and exchanging tokens with <NT>GCP</NT>

After you successfully configured a workload identity pool and connected it to a Google service account, your <NT>Bitrise</NT> builds can exchange tokens with <NT>GCP</NT>. You can:

- Use the **Authenticate with <NT>GCP</NT>** <NT>Step</NT> to handle the whole process.
- Fetch the token with **<NT>Get OIDC Identity Token</NT>** <NT>Step</NT>, then perform the credential exchange with a script.

:::note[Information in the OIDC token]

<NT>Bitrise</NT> includes the following claims in the OIDC token: [Information in the OIDC token](/en/bitrise-platform/integrations/oidc-authentication/oidc-authentication-overview#information-in-the-oidc-token).

:::

### Using the Authenticate with <NT>GCP</NT> <NT>Step</NT>

1. Open the <NT><GlossTerm baseform="workflow editor">Workflow Editor</GlossTerm></NT> on <NT>Bitrise</NT>.
1. Add the **Authenticate with <NT>GCP</NT>** <NT>Step</NT> to your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>.
1. Set the **Client config** and the **<NT>Token audience</NT>** inputs:

   - The <NT>token audience</NT> should match the allowed audience you set when configuring the workload identity pool on Google Cloud.
   - The **Client config** input should point to the file you downloaded at the end of the process.

### Using the Get OIDC Identity token <NT>Step</NT>

1. Open the <NT>Workflow Editor</NT> on <NT>Bitrise</NT>.
1. Add the **<NT>Get OIDC Identity Token</NT>** <NT>Step</NT> to your <NT>Workflow</NT>.
1. Use the BITRISE_IDENTITY_TOKEN Environment Variable: write it to a file and update the [client library configuration file](/en/bitrise-platform/integrations/oidc-authentication/oidc-for-gcp#connecting-a-google-service-account-to-the-workload-identity-pool) to reference the file containing the Env Var. You can set this up in a **Script** <NT>Step</NT>:

   ```bash
   tmpfile="$(mktemp)"       
   printf '%s' "$BITRISE_IDENTITY_TOKEN" > "$tmpfile"

   # Replace the token path in client library configuration

   gcloud auth login --cred-file=/path/to/updated-client-library-config.json
   gcloud auth list
   ```

   After, the gcloud CLI will automatically pick them up and use them. Any kind of action through the CLI will just simply work.
