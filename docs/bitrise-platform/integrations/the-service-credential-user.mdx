The service credential is a user whose connected Git provider account (<NT>GitHub</NT>, <NT>GitLab</NT> or <NT>Bitbucket</NT>) is used to access those Git services for certain actions that <NT>Bitrise</NT> can execute.

:::important[<NT>GitHub</NT> project]

The <NT>service credential user</NT> is NOT required if you use the [GitHub App integration](/en/bitrise-platform/repository-access/github-app-integration) to connect to your Git repositories.

:::

You need to set a <NT>service credential user</NT>, among other things, to:

- Send [build status reports](/en/bitrise-ci/configure-builds/configuring-build-settings/reporting-the-build-status-to-your-git-hosting-provider) back to your Git provider.
- Use the [Selective builds](/en/bitrise-ci/configure-builds/configuring-build-settings/selective-builds) feature that allows you to specify certain files or directories in your repository that need to be modified to <NT><GlossTerm baseform="trigger">trigger</GlossTerm></NT> an automatic <NT>Bitrise</NT> build.
- [Store the `bitrise.yml` file in your repository](/en/bitrise-ci/configure-builds/configuration-yaml/storing-an-apps-configuration-yaml#storing-the-bitriseyml-file-in-your-repository).

:::tip[Actions and requirements]

For the full list of actions and their access requirements, see [Service credential user actions and their required repository access](/en/bitrise-platform/integrations/the-service-credential-user#service-credential-user-actions-and-their-required-repository-access).

:::

By default, the user who added the project to <NT>Bitrise</NT> is set as the <NT>service credential user</NT>. Even if you change it, it should always be a user who has access to the project's repository.

It isn't mandatory to have a <NT>service credential user</NT>: you can use the **No <NT>service credential user</NT>** setting to not have one. You'll still be able to run builds and trigger builds automatically but <NT>Bitrise</NT> won't be able to send build status reports to your Git provider, you won't be able to use the [Selective builds](/en/bitrise-ci/configure-builds/configuring-build-settings/selective-builds) feature or to store the <NT><GlossTerm baseform="bitrise.yml">bitrise.yml</GlossTerm></NT> file in your repository.


## Changing the <NT>service credential user</NT>

The <NT>service credential user</NT> can be changed at any time but you can only set yourself as the <NT>service credential user</NT> on a <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>.

:::important[Role requirement]

To change the <NT>service credential user</NT>, you need to have **Admin**[role on the project's team](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci) on <NT>Bitrise</NT>.

:::

To change the <NT>service credential user</NT>:

1. <NT>Log in</NT> to <NT>Bitrise</NT> with the account that will be the new <NT>service credential user</NT>.
1. Make sure that account has **Admin** role on the [project's team](/en/bitrise-platform/projects/roles-and-permissions-for-bitrise-ci).
1. Go to your <NT><GlossTerm baseform="Workspace">workspace</GlossTerm></NT>'s main page.
1. Select the project where you want to change the <NT>service credential user</NT>.
1. Click on **<NT>Project Settings</NT>**.
1. On the left, select **<NT>Integrations</NT>** from the menu options.
1. Scroll down to the **<NT>Service credential user</NT>** part.
1. Click **<NT>Change to me</NT>**.

   ![service-credential-user.png](/img/_paligo/uuid-6519a945-1fd2-97fd-da4a-fdab48705503.png)

   :::note[Disabling OAuth project access restrictions for your <NT>GitHub</NT> organization]

   If you receive an error while trying to connect or change the <NT>service credential user</NT> for your project, make sure that the user has the project access to the <NT>GitHub</NT> repository and that **Third-party app access policy restrictions** are disabled.

   For more information on how to disable the Third-party app access policy on <NT>GitHub</NT>, check out [Disabling OAuth project access restrictions for your organization](https://docs.github.com/en/organizations/managing-oauth-access-to-your-organizations-data/about-oauth-app-access-restrictions).

   :::
1. To confirm, click **Set <NT>service credential user</NT>**.


## <NT>Service credential user</NT> actions and their required repository access

:::important[<NT>GitHub</NT> project]

The <NT>service credential user</NT> is NOT required if you use the [GitHub App integration](/en/bitrise-platform/repository-access/github-app-integration) to connect to your Git repositories.

:::

The <NT>service credential user</NT> allows <NT>Bitrise</NT> to execute a number of different actions that require access to your project's Git repository. Different actions require different levels of access: for example, [to use a `bitrise.yml` file from the repository](/en/bitrise-ci/configure-builds/configuration-yaml/storing-an-apps-configuration-yaml#storing-the-bitriseyml-file-in-your-repository) requires **read** access on <NT>GitHub</NT> while [registering a new SSH key](/en/bitrise-platform/repository-access/configuring-ssh-keys) requires **admin** access.

The following table contains all the actions that require the <NT>service credential user</NT>, as well as the access level of the <NT>service credential user</NT>'s account at the the three main Git providers

| Action | Required access level on <NT>GitHub</NT> | Required access level on <NT>Bitbucket</NT> | Required access level on <NT>GitLab</NT> |
| --- | --- | --- | --- |
| [Using a `bitrise.yml` from the repository](/en/bitrise-ci/configure-builds/configuration-yaml/storing-an-apps-configuration-yaml#storing-the-bitriseyml-file-in-your-repository) | read | read | reporter |
| List the branches of the repository | read | read | reporter |
| [Selective builds](/en/bitrise-ci/configure-builds/configuring-build-settings/selective-builds) | read | read | reporter |
| [Sending Git status reports](/en/bitrise-ci/configure-builds/configuring-build-settings/reporting-the-build-status-to-your-git-hosting-provider) | write | write | <NT>GitLab</NT>.com: developer  Self-hosted <NT>GitLab</NT>: maintainer |
| [Registering SSH keys to the repository](/en/bitrise-platform/repository-access/configuring-ssh-keys) | admin | admin | maintainer |
| [Register a webhook](/en/bitrise-platform/integrations/webhooks/adding-incoming-webhooks#registering-an-incoming-webhook-automatically) | admin | admin | maintainer |
| Checking the Git connection on bitrise.io | admin | admin | maintainer |
