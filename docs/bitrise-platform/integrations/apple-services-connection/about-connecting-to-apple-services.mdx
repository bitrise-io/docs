Both [Bitrise CI](/en/bitrise-ci) and [Release Management](/en/release-management) helps you automate Mobile DevOps processes that require <NT>Apple</NT>'s online services. To connect to these services, such as the <NT>Apple Developer Portal</NT> or <NT>App Store Connect</NT>, you need to provide authentication data to <NT>Bitrise</NT> and select the established authentication method for your project.

You can authenticate with <NT>Apple</NT>’s official API key or with <NT>Apple ID</NT> and password.


## <NT>Apple</NT> two-factor authentication requirements

<NT>Apple</NT>’s [two-factor authentication](https://developer.apple.com/support/authentication/) (2FA) provides an extra layer of security on your <NT>Apple</NT> account.

If you have been authenticating with the API key so far, you are not affected by the two-factor authentication requirement.

If, however, you have been authenticating with an <NT>Apple ID</NT> and a password, and the [new 2FA requirement](https://developer.apple.com/support/authentication/) affects you, then you’ll have to reconnect your <NT>Apple</NT> Developer Account on the **<NT>Apple Service connection</NT>** page of your <NT>Bitrise</NT> profile. You’ll also have to provide the two-factor authentication/two-step verification code and an <NT>app-specific password</NT> as well. Please find the official <NT>Apple</NT> documentation on [how to generate an app-specific password](https://support.apple.com/en-us/HT204397).


## <NT>Steps</NT> that require connecting to your <NT>Apple</NT> Developer account

The following <NT>Steps</NT> require connection to <NT>Apple</NT> services (such as <NT>App Store Connect</NT> or the <NT>Apple Developer Portal</NT>). If you’re using any of these <NT>Steps</NT>, make sure you establish connection with the right method.

| <NT>Steps</NT> | <NT>Connection type</NT> |
| --- | --- |
| <NT>Manage iOS Code Signing</NT> | API key authentication, <NT>Apple ID</NT> authentication, API key authentication through <NT>Step inputs</NT> |
| <NT>Xcode Archive & Export for iOS</NT> | API key authentication, <NT>Apple ID</NT> authentication, API key authentication through <NT>Step inputs</NT> |
| Export iOS and tvOS <NT>Xcode archive</NT> | API key authentication, <NT>Apple ID</NT> authentication, API key authentication through <NT>Step inputs</NT> |
| <NT>Xcode Build for testing for iOS</NT> | API key authentication, <NT>Apple ID</NT> authentication, API key authentication through <NT>Step inputs</NT> |
| fastlane | API key authentication, <NT>Apple ID</NT> authentication, API key or <NT>Apple ID</NT> authentication through <NT>Step inputs</NT> |
| <NT>Deploy to App Store Connect</NT> with Deliver (formerly iTunes Connect) | API key authentication, <NT>Apple ID</NT> authentication, API key or <NT>Apple ID</NT> authentication through <NT>Step inputs</NT> |
| <NT>Deploy to App Store Connect</NT> - Application Loader (formerly iTunes Connect) | API key authentication, <NT>Apple ID</NT> authentication, API key or <NT>Apple ID</NT> authentication through <NT>Step</NT> |

Depending on which authentication you can use in your project, you have the following options:

- <NT>API key</NT> authentication: If you can, we recommend you use this authentication method. It does not require two-factor authentication. All it takes is connecting to the <NT>Apple</NT> services by providing **<NT>Name</NT>**, **<NT>Issuer ID</NT>**, **<NT>Key ID</NT>** and upload a **<NT>Private</NT> <NT>Key</NT> (.p8)**, then selecting an account under the **<NT>Team</NT>** tab in your app’s settings. The data you give automatically populates the respective fields of the <NT>Steps</NT> that work with API key authentication.
- <NT>Apple ID authentication</NT>: If you cannot use the API key authentication, you can authenticate with your <NT>Apple ID</NT> and password. Provide your **<NT>Apple ID</NT>**, **password**, **2FA code** and **<NT>app-specific password</NT>** then <NT>select an account</NT> under the **<NT>Team</NT>** tab in your app’s settings. The data you give automatically populates the respective fields of the <NT>Steps</NT> that work with the <NT>Apple ID</NT> authentication.
- <NT>API key</NT> or <NT>Apple Id</NT> authentication through <NT>Step inputs</NT>: If you wish to deploy to multiple teams or deploy to a team where authentication is different from the connected one you’ve been using, then you can add your preferred authentication into the <NT>Step</NT>’s inputs. Some <NT>Steps</NT>, such as <NT>Xcode Archive & Export for iOS</NT>, only have an API key authentication override option, while some <NT>Steps</NT>, like fastlane, have options for both API key and <NT>Apple ID</NT> <NT>Step</NT> level authentication override.


## <NT>Apple</NT> service permissions

In order to successfully connect <NT>Bitrise</NT> to <NT>Apple</NT> services, you need to set up the right roles and accesses in your <NT>Apple</NT> account for your <NT>Apple ID</NT> and your API keys. You must set up the appropriate access rights to be able to:

- Use <NT>automatic code signing</NT>.
- Deploy your app to the App Store.

The exact roles and accesses you need depend on a number of factors: your authentication method, whether you use <NT>Xcode</NT> managed signing, and the details of your app deployment process, among other things.

In this guide, we'll list the roles based on the two main [authentication methods](/en/bitrise-platform/integrations/apple-services-connection/steps-requiring-apple-authentication): API key authentication and <NT>Apple ID</NT> authentication.

### Required access with API key authentication

To use <NT>Bitrise Steps</NT> with <NT>Apple</NT> API key authentication, you need to create an <NT>App Store Connect</NT> API key with the appropriate access level. The appropriate level depends on what you need to do.

If, for example, you use <NT>Xcode</NT> managed signing in your project and wish to export the generated IPA file with a Distribution certificate and an App Store provisioning profile, the <NT>App Store Connect</NT> API key must have **Admin** access.

The following table contains the required access for <NT>automatic code signing</NT>. In the table, we grouped <NT>code signing</NT> actions based on the type of the IPA file we're attempting to export. There are two main types:

- Development IPA: this is an IPA exported with the **development** method.
- Distribution IPA: this is an IPA exported with the **app-store**, **ad-hoc**, or **enterprise** <NT>distribution method</NT>.

| <NT>Code signing</NT> action | Required access with <NT>Xcode</NT> managed signing turned ON | Required access with <NT>Xcode</NT> managed signing turned OFF |
| --- | --- | --- |
| Exporting Development IPA. This can include:  - Creating development provisioning profiles. - Deleting development provisioning profiles. - Downloading provisioning profiles. - Registering and configuring App IDs. - Adding device UDIDs | Developer | Developer |
| Exporting App Store IPA. This can include:  - Creating development and distribution provisioning profiles. - Deleting development and distribution provisioning profiles. - Downloading provisioning profiles. - Registering and configuring App IDs. - Adding device UDIDs | Admin | Developer |

The following table contains the required roles for deploying your app to the App Store.

For App Store deployment, the required access depends on how you wish to upload the generated IPA file. You can either:

- Upload only the IPA without any additional steps.
- Upload the IPA with metadata and screenshots, and submit the app for review.

| App Store deployment actions | Required access for API key |
| --- | --- |
| Uploading a new IPA without any metadata | Developer |
| Uploading a new IPA and:  - Updating app metadata. - Uploading screenshots. - Submitting the app for App Store review. | App Manager |

### Required access with <NT>Apple ID</NT> authentication

To use <NT>Bitrise Steps</NT> with <NT>Apple ID</NT> authentication, you need to make sure that your <NT>Apple ID</NT> has the appropriate role in your <NT>Apple</NT> Developer team.

The following table contains the necessary roles for using <NT>automatic code signing</NT> on <NT>Bitrise</NT>. In the table, we grouped <NT>code signing</NT> actions based on the type of the IPA file we're attempting to export. There are two main types:

- Development IPA: this is an IPA exported with the **development** method.
- Distribution IPA: this is an IPA exported with the **app-store**, **ad-hoc**, or **enterprise** <NT>distribution method</NT>.

Read more about the different <NT>distribution method</NT>s: [Creating a signed IPA for Xcode projects](/en/bitrise-ci/code-signing/ios-code-signing/creating-a-signed-ipa-for-xcode-projects).

:::note[<NT>Xcode</NT> managed signing]

If you use <NT>Apple ID</NT> authentication on <NT>Bitrise</NT>, <NT>Xcode</NT> managed signing is automatically turned off in your project. Instead, <NT>Bitrise</NT> uses its own <NT>automatic code signing</NT> logic.

:::

| <NT>Code signing</NT> action | Required role |
| --- | --- |
| Exporting development IPA. This can include:  - Creating development provisioning profiles. - Deleting development provisioning profiles. - Downloading provisioning profiles. - Registering and configuring App IDs. - Adding device UDIDs | App Manager |
| Exporting an App Store IPA. This can include:  - Creating development and distribution provisioning profiles. - Deleting development and distribution provisioning profiles. - Downloading provisioning profiles. - Registering and configuring App IDs. - Adding device UDIDs | App Manager |

The following table contains the required roles for deploying your app to the App Store.

For App Store deployment, the required access depends on how you wish to upload the generated IPA file. You can either:

- Upload only the IPA without any additional steps.
- Upload the IPA with metadata and screenshots, and submit the app for review.

| App Store deployment actions | Required role for <NT>Apple ID</NT> |
| --- | --- |
| Uploading a new IPA without any metadata | Developer |
| Uploading a new IPA and:  - Updating app metadata. - Uploading screenshots. - Submitting the app for App Store review. | App Manager |
