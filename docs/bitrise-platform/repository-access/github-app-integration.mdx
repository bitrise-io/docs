:::important[Supported for <NT>GitHub Cloud</NT> users]

The <NT>Bitrise</NT> <NT>GitHub</NT> app is supported for all <NT>GitHub Cloud</NT> users. <NT>GitHub Enterprise Server</NT> users need to create and install a different <NT>GitHub</NT> app, as described in our [GH Enterprise integration guide](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise).

:::

The best way to connect your <NT>Bitrise</NT> account to your <NT>GitHub</NT> repositories is by integrating with [a GitHub app](https://docs.github.com/en/apps/overview). The <NT>Bitrise</NT> <NT>GitHub</NT> app has a number of features that make integration easier:

- The app eliminates the need for an SSH key, a Personal Access Token, and a <NT>service credential user</NT>. For access, it generates a temporary access token before every build, significantly increasing security.
- It provides granular access to your repositories: no need to grant access to an entire <NT>GitHub</NT> organization, instead select the specific repositories the app can access.
- With the app installed, you automatically receive <NT>Bitrise</NT> status updates directly on <NT>GitHub</NT> with the <NT>GitHub</NT> Checks app. No need for configuring status reports separately. Note that while only one <NT>GitHub</NT> account can be connected to a <NT>Bitrise</NT> <NT><GlossTerm baseform="workspace">Workspace</GlossTerm></NT>, <NT>GitHub</NT> Checks can still be configured for repositories owned by other <NT>GitHub</NT> accounts.

:::important[HTTPS URL required]

The <NT>GitHub</NT> App requires an HTTPS URL for your repository instead of an SSH one. Normally, you don't have to worry about this: setting up the <NT>GitHub</NT> App connection changes the URL of your project. If there's an issue, you can change the URL manually: [Changing the repository URL](/en/bitrise-platform/projects/configuring-the-repository-url-and-the-default-branch#changing-the-repository-url).

:::


## Event subscriptions

When you install the <NT>Bitrise</NT> <NT>GitHub</NT> App on a repository or organization, <NT>GitHub</NT> automatically registers event subscriptions for that installation. These are separate from the repo-level webhooks you can configure manually in <NT>GitHub</NT>'s repository settings.

The <NT>Bitrise</NT> <NT>GitHub</NT> App subscribes to the following events:

- `push`: triggers builds on branch pushes and tag pushes.
- `pull_request`: triggers builds when a pull request is opened, updated, or synchronized.
- `issue_comment`: enables comment-based build triggers on pull requests.

These subscriptions are managed by <NT>GitHub</NT> at the app installation level. They do not appear as webhooks under **<NT>Settings → Webhooks</NT>** in your <NT>GitHub</NT> repository.

:::warning[Remove manual webhooks after switching to the <NT>GitHub</NT> App]

If your repository also has manually configured webhooks pointing to `hooks.bitrise.io` — for example, from a previous OAuth-based <NT>Bitrise</NT> connection — each qualifying event will trigger two builds: one from the <NT>GitHub</NT> App subscription and one from the manual webhook.

After switching to the <NT>GitHub</NT> App, remove any manual <NT>Bitrise</NT> webhooks from your repository's **<NT>Settings → Webhooks</NT>** page on <NT>GitHub</NT> to avoid duplicate builds.

:::


## Installing the <NT>GitHub</NT> app integration

This guide is intended for <NT>GitHub Cloud</NT> users, including <NT>GitHub</NT> Enterprise Cloud users, who wish to install the <NT>Bitrise</NT> <NT>GitHub</NT> app and connect their <NT>Bitrise</NT> <NT>Workspace</NT> to a <NT>GitHub</NT> account or organization with the app.

For <NT>GitHub Enterprise Server</NT> users, we have a separate guide: [Integrating GitHub Enterprise with Bitrise](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise)

You can connect via the <NT>GitHub</NT> app integration in two ways:

- From the **<NT>Workspace settings</NT>** page.
- When adding a new <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>.

:::important[<NT>GitHub</NT> App installations per <NT>Workspace</NT>]

Note that:

- A maximum of ten <NT>GitHub</NT> app installations can be connected to a <NT>Workspace</NT>.
- A <NT>GitHub</NT> app installation cannot be connected to multiple <NT>Workspaces</NT>.
- The <NT>GitHub</NT> Checks functionality of the app is available for repositories owned by other <NT>GitHub</NT> accounts.

:::

### Connecting a <NT>GitHub</NT> app from the <NT>Workspace settings</NT> page

A <NT>Bitrise</NT> <NT>GitHub</NT> app installation is tied to a <NT>Workspace</NT>. You can always install and check its connection from the **<NT>Workspace settings</NT>** page.

:::note[SSO on <NT>GitHub</NT>]

If you install the <NT>GitHub</NT> App integration for a <NT>GitHub</NT> organization that requires SAML SSO, you need to have an active SAML SSO session on <NT>GitHub</NT> to be able to connect to the organization's repositories.

If your installed <NT>GitHub</NT> app can't access <NT>all repositories</NT> it should, you might to need to revoke the app and authorize it again on <NT>GitHub</NT> while you have an active SAML SSO session.

For more information and the detailed procedure, [check GitHub's documentation](https://docs.github.com/en/enterprise-cloud@latest/apps/using-github-apps/saml-and-github-apps).

:::

1. On the <NT>Bitrise</NT> main page, select your <NT>Workspace</NT> from the dropdown menu.
1. From the navigation menu on the left, select **<NT>Settings</NT>**.
1. On the **<NT>Workspace settings</NT>** page, select **<NT>Integrations</NT>**.
1. Select **<NT>GitHub</NT>** and click **<NT>Connect</NT>**.

   ![github-account-connection.png](/img/_paligo/uuid-1b49695a-ea36-fe03-f9a1-48a6bfc9a737.png)
1. If you haven't authorized the <NT>Bitrise</NT> <NT>GitHub</NT> app before, you will be prompted for authorization at this point.
1. You will be prompted to select the <NT>GitHub account</NT> or organization you want to connect. If you don't find the one you need in the list, you need to install the App to that account or organization first: click the link below the list of accounts.

   ![install-the-app.png](/img/_paligo/uuid-cae94617-17e3-33ce-8cf7-99205e85bd22.png)
1. Once you click the link, you will be prompted to select an account or organization. After selecting one, you will be transferred to the **<NT>Install & Authorize</NT>** page.
1. On the **<NT>Install & Authorize</NT>** page, select the access type:

   - **<NT>All repositories</NT>**: the <NT>Bitrise</NT> <NT>GitHub</NT> App will have access to <NT>all repositories</NT> belonging to the account or organization, including future ones.
   - **<NT>Only select repositories</NT>**: Select one or more repositories that <NT>Bitrise</NT> will be able to access. You can add more later but to do so, you will need to reconfigure your repository access on the <NT>GitHub</NT> <NT>App</NT>'s page.

   ![authorizing-app.png](/img/_paligo/uuid-a25560d9-ce2a-4181-2dc6-59cc4318c577.png)
1. When done, click **<NT>Install & Authorize</NT>**. You will be redirected to the **<NT>Workspace settings</NT>** page.

   :::important[Authorization]

   If you are <NT>not authorized</NT> on <NT>GitHub</NT> to install the app, you can still request the installation. Once a <NT>GitHub</NT> Admin approves the installation, you can come back and select the **<NT>App</NT> installation** from the list.

   :::
1. If you have existing apps with OAuth connection, we recommend [switching them over to the GitHub App connection](/en/bitrise-platform/repository-access/github-app-integration#switching-from-oauth-connection-to-the-github-app).

## Extending <NT>GitHub</NT> app permissions to the builds

:::tip[Using the API]

This section describes how to configure the feature on the <NT>Bitrise website</NT>. You can also configure it via the <NT>Bitrise</NT> API: [GitHub app configuration API](/en/bitrise-ci/api/github-app-configuration-api).

:::

The <NT>Bitrise</NT> <NT>GitHub</NT> app generates a short-term, temporary token for each build that is triggered via the app. This token has only one permission by default: `content:read`. This means the build can access the <NT>GitHub</NT> repository but can't do anything else.

You can extend these permissions so that you can perform other operations during a build. For example, this can enable users to push Git tags from their builds, create custom status reports, put a label on a pull request, or push a new version number.

:::important[Linked repositories]

When extending permissions, you extend permissions to all [additional linked repositories](/en/bitrise-platform/repository-access/github-app-integration#additional-linked-repositories-via-a-github-app) as well!

:::

The exact permissions you can extend depends on your <NT>GitHub</NT> account:

- On <NT>GitHub Cloud</NT>, the <NT>GitHub</NT> app has the following permissions:

  - Read access to issues and metadata
  - Read and write access to checks, code, commit statuses, and pull requests
- [GitHub Enterprise Server](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise) users define their <NT>GitHub</NT> app permissions when creating the app.

To extend the permissions:

1. <Partial_GettingToTheProjectSettingsPage />
1. Select **Repository** from the menu.
1. Find the **<NT>Extend GitHub App permissions to builds</NT>** option and toggle it on. This extends permissions to the default repository and all additional linked repositories, too.


## Additional linked repositories via a <NT>GitHub</NT> App

:::tip[Using the API]

This section describes how to configure the feature on the <NT>Bitrise website</NT>. You can also configure it via the <NT>Bitrise</NT> API: [GitHub app configuration API](/en/bitrise-ci/api/github-app-configuration-api).

:::

Each <NT>Bitrise</NT> <NT>project</NT> has a primary Git repository. This is where your project's code is stored and this repository is cloned when we start a build. By default, a <NT>Bitrise</NT> <NT>project</NT> can't access other repositories.

However, your project might require other repositories. For example, it might have private submodules that it must access during a build. If you use the <NT>GitHub</NT> App, you can do this via linking additional repositories. The feature is available on both <NT>GitHub Cloud</NT> and <NT>GitHub Enterprise Server</NT>.

:::note[Other connection methods]

If you are not using the <NT>Bitrise</NT> <NT>GitHub</NT> App, you can still configure access to additional repositories: [Apps with submodules or private repo dependencies](/en/bitrise-platform/repository-access/apps-with-submodules-or-private-repo-dependencies).

:::

A linked repository is a repository that a <NT>Bitrise</NT> <NT>project</NT> can access using a <NT>GitHub</NT> App installation but it's not the project's primary repository.

When running a build of a project connected via a <NT>GitHub</NT> App installation, <NT>Bitrise</NT> generates a temporary token that, by default, can only access the primary repository. Once an additional repository is linked, the tokens generated for the build can access the additional repository.

To be able to link a repository, all of the following conditions must be met:

- The <NT>GitHub</NT> App is installed to the account or organization that owns the repository on <NT>GitHub</NT>.
- The app installation is enabled for the repository.
- The user attempting to link the repository can access the repository on <NT>GitHub</NT>. If your <NT>GitHub</NT> user account doesn't have access to a repository, you can't link it on <NT>Bitrise</NT>.

  :::note[<NT>GitHub Enterprise Server</NT>]

  If you use a <NT>GitHub</NT> App to connect to a [GitHub Enterprise Server](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise) instance, the app doesn't check user access. You can link any repository that your <NT>GitHub</NT> App has access to.

  :::

### Linking additional repositories

To link additional repositories to a project with an established <NT>GitHub</NT> App connection:

1. <Partial_GettingToTheProjectSettingsPage />
1. On the left, select **Repository**.
1. Under **<NT>Authorization</NT>**, find the **<NT>Additional repositories</NT>** card and click **<NT>Change</NT>**.
1. In the **<NT>Enable</NT> <NT>additional repositories</NT>** dialog, select the repositories you want to link.

   ![SCR-20260331-pmhi.png](/img/_paligo/uuid-324a0ec5-3e09-e10f-9e9e-7ae1359ffdc2.png)

   You can link a maximum of 50 additional repositories.

   You will see a list of repositories. To see a repository on the list:

   - On <NT>GitHub Cloud</NT>, your <NT>GitHub</NT> user needs to have access to it. If you use [GitHub Enterprise Server](/en/bitrise-platform/repository-access/integrating-github-enterprise-with-bitrise), user permissions are not checked.
   - The <NT>GitHub</NT> App installation has access to it.

   :::important[Disabling repositories]

   If you already have repositories linked, please note that users who otherwise don't have access to those repositories can disable them but cannot add them back!

   :::
1. When done, click **<NT>Save changes</NT>**.

### Linking <NT>all repositories</NT> automatically

<NT>GitHub</NT> organization owners can switch the **<NT>Grant access to all current and future repositories</NT>** toggle so that <NT>Bitrise</NT> builds can have access to every <NT>GitHub</NT> repository your <NT>GitHub</NT> App is allowed to access, including any new repositories added in future.

To successfully use this functionality:

- For security reasons, you have to be logged in as a <NT>Bitrise</NT> user who is also the <NT>GitHub</NT> **owner** of your organization. If you aren't the owner, ask the owner to log in to <NT>Bitrise</NT> and toggle the feature on.
- Make sure you configured the <NT>GitHub</NT> App to have access to necessary repositories.

1. On <NT>Bitrise</NT>, go to the **<NT>Project Settings</NT>** page of your project.
1. On the left, select **Repository**.
1. Under **<NT>Authorization</NT>**, find **<NT>Additional repositories</NT>** and click **<NT>Change</NT>**.
1. Toggle the **<NT>Grant access to all current and future repositories</NT>** switch to the right.

   ![linkadditionalrepositories.png](/img/_paligo/uuid-bfc425a6-84a8-0672-31bb-d3d7b07b56ce.png)

   Your project's build will now have access to all current and future <NT>GitHub</NT> repositories the <NT>Bitrise</NT> <NT>GitHub</NT> App has access to.

:::note[<NT>GitHub</NT> email notification about allowing permission]

When <NT>Bitrise</NT> released this feature, <NT>Bitrise</NT> <NT>GitHub</NT> App users will have received an email from <NT>GitHub</NT> on a new permission which is needed to use the <NT>Grant access to all current and future repositories</NT> toggle. <NT>Bitrise</NT> <NT>GitHub</NT> App requests additional permission which is called: read organization membership. Only <NT>GitHub</NT> organization ownershave the necessary rights to follow the link in the email and to make this functionality available for their teams.

This is a standard and legitimate message from <NT>GitHub</NT> and allowing this permission will not have any detrimental effect on <NT>Bitrise</NT> builds. You can read more about the new permission <NT>Bitrise</NT> requires [here](https://docs.github.com/en/rest/orgs/members?apiVersion=2022-11-28#get-organization-membership-for-a-user).

:::


## Using a private <NT>Step</NT> library via the <NT>GitHub</NT> App

Each <NT>Bitrise</NT> <NT>project</NT> has a default <NT>Step</NT> library: a repository of <NT>Steps</NT>. If the exact source of a <NT>Step</NT> is not specified in the configuration, the <NT>Bitrise CLI</NT> pulls the <NT>Step</NT> data from the default library.

This library can be a private <NT>Step</NT> library. To access such a library via the <NT>GitHub</NT> App, you need to add the **[Authenticate host with netrc](https://bitrise.io/integrations/steps/authenticate-host-with-netrc)** <NT>Step</NT> as the very first <NT>Step</NT> which will run in your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>:

```yaml
format_version: "13"
default_step_lib_source: https://github.com/my-private-org/my-bitrise-steplib.git
project_type: ios
workflows:
  example:
    steps:
    - https://github.com/bitrise-io/bitrise-steplib.git::authenticate-host-with-netrc@0:
```

Note that the **[Authenticate host with netrc](https://bitrise.io/integrations/steps/authenticate-host-with-netrc)** <NT>Step</NT> has [a full source identifier](/en/bitrise-ci/references/steps-reference/step-reference-id-format) to ensure it is pulled from the official <NT>Bitrise Step</NT> library, not your private library.

Using the <NT>Step</NT> requires specifying three inputs: the host, the Git username, and the Git password. The <NT>Bitrise</NT> <NT>GitHub</NT> App integration uses token-based authentication: each build receives a one time token under the <NT>GIT_HTTP_PASSWORD</NT> Environment Variable. This Env Var can be used as the password. The username can't be empty because the <NT>Step</NT> will fail but it is not used so it doesn't matter what you put there.

```yaml
format_version: '13'
default_step_lib_source: https://github.com/tothszabi/steplib.git
project_type: ios
workflows:
  example:
    steps:
    - https://github.com/bitrise-io/bitrise-steplib.git::authenticate-host-with-netrc@0:
        inputs:
        - host: github.com
        - username: username
        - password: "$GIT_HTTP_PASSWORD"
    - private-script@1: {}
```


## Running <NT>git clone</NT> with linked repositories

If you attempt to run `git clone` or other git commands for a [linked repository](/en/bitrise-platform/repository-access/github-app-integration#additional-linked-repositories-via-a-github-app), your build might fail with an authentication error. This is because the authentication credentials are stored in a local `netrc` file. This file is automatically generated by the official **<NT>Git Clone</NT>** <NT>Step</NT>.

However, if you use your own script to perform `git clone`, or you use any other Git command requiring authentication before the **<NT>Git Clone</NT>** <NT>Step</NT>, you need to create the `netrc` file. You have two ways:

- You can do it manually.
- You can use the **[Authenticate host with netrc](https://bitrise.io/integrations/steps/authenticate-host-with-netrc)** <NT>Step</NT>. It should be the first <NT>Step</NT> of your <NT>Workflow</NT>.

:::important[HTTP URLs]

Make sure you always use HTTP URLs when cloning private repository dependencies.

:::

Using the <NT>Step</NT> requires specifying three inputs: the host, the Git username, and the Git password. The <NT>Bitrise</NT> <NT>GitHub</NT> App integration uses token-based authentication: each build receives a one time token under the <NT>GIT_HTTP_PASSWORD</NT> Environment Variable. This Env Var can be used as the password. The username can't be empty because the <NT>Step</NT> will fail but it is not used so it doesn't matter what you put there.

```yaml
workflows:
  example:
    steps:
    - authenticate-host-with-netrc@0:
        inputs:
        - host: github.com
        - username: "username"
        - password: "$GIT_HTTP_PASSWORD"
```

## Reverting back to OAuth connection

If you used an OAuth connection for repository access before switching to the <NT>GitHub</NT> app, you can revert back to it at any time.

:::note

The OAuth connection requires an SSH key or a personal access token at the Git provider, depending on the <NT>authorization method</NT>. When switching to the <NT>GitHub</NT> app, you don't have to remove these: if you didn't, you can use them when reverting to the OAuth connection.

:::

1. <Partial_GettingToTheProjectSettingsPage />
1. Under **<NT>Bitrise CI</NT>**, select **<NT>Integrations</NT>** and then the **<NT>Git provider</NT>** tab.
1. Find the **<NT>Connection type</NT>** card and click **<NT>Change</NT>**.
1. In the dialog, select **OAuth** and click **<NT>Change</NT>**.
