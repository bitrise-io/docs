For <NT><GlossTerm baseform="workspace">Workspace</GlossTerm></NT>s, <NT>Bitrise</NT> supports connecting to self-hosted <NT>GitLab</NT> instances. Connecting is simple, and once it’s done, you can add projects to <NT>Bitrise</NT> from privately hosted repositories. All functions that you got used to with publicly hosted repositories will be available!

<NT>Bitrise</NT> needs to be authorized as an application for the entire <NT>GitLab</NT> instance and then the **Application Id** and the **<NT><GlossTerm baseform="secret">Secret</GlossTerm></NT>** value belonging to the <NT>Bitrise</NT> OAuth application must be added to the <NT>Workspace</NT> on [bitrise.io](https://www.bitrise.io).


## Configuring the <NT>GitLab</NT> connection

To access privately hosted <NT>GitLab</NT> repositories on <NT>Bitrise</NT>, you need to create a new OAuth application on your <NT>GitLab</NT> account, and assign the appropriate credentials to access your server.

:::warning

When you create a new <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>, you must pick the dedicated **<NT>GitLab</NT> Self-hosted** option in the provider picker. If you paste your repository URL into the **<NT>Other</NT>** field instead, <NT>Bitrise</NT> will save the project as a generic Custom-provider repository, and the following features will be silently unavailable:

- The [Service Credential User](/en/bitrise-platform/integrations/the-service-credential-user) UI in <NT>App Settings</NT>.
- The [Store bitrise.yml in repository](/en/bitrise-ci/configure-builds/configuration-yaml/storing-an-apps-configuration-yaml#storing-the-bitriseyml-file-in-your-repository) option.
- <NT>Build status</NT> reporting back to <NT>GitLab</NT> merge requests.

This cannot be fixed by editing the repository URL afterwards. If you have already created the project this way, <NT>contact support</NT> to have the provider type corrected.

:::

:::warning[Publicly resolvable IP address]

Your self-hosted instance must have a publicly resolvable IP address, otherwise <NT>Bitrise</NT> won’t be able to connect to it.

:::

The connection requires two URLs: a <NT>Bitrise</NT> OAuth callback URL and a **<NT>GitLab instance</NT>** URL:

| URL | Where it goes | Example |
| --- | --- | --- |
| <NT>Bitrise</NT>'s OAuth callback URL | In <NT>GitLab</NT>, in the OAuth application's **Callback URL** field | `https://app.bitrise.io/users/auth/gitlab/callback` |
| Your <NT>GitLab instance URL</NT> | In <NT>Bitrise</NT>, in the <NT>Workspace</NT> Integrations **<NT>Self-hosted URL</NT>** field | `https://gitlab.example.com` |

1. Allowlist the <NT>Bitrise</NT> backend workers for your <NT>GitLab</NT> instance: [Configuring your network to access our build machines](/en/bitrise-platform/infrastructure/build-machines/configuring-your-network-to-access-our-build-machines).

   Be aware that the <NT>Bitrise</NT> backend workers that power, among other things, app.bitrise.io and the <NT>Bitrise</NT> <NT>build machines</NT> have a different IP address range. Allowlisting the <NT>build machines</NT> is not sufficient for this feature.

   If allowlisting the backend workers is not an option for you, [reach out to Bitrise](https://www.bitrise.io/contact) so we can discuss other options.
1. <NT>Log in</NT> to your <NT>GitLab</NT> instance with **root** privileges. If you connect to <NT>Bitrise</NT> **without root privileges**, your <NT>Workspace</NT>’s other team members will **not have access to the repository on the <NT>GitLab</NT> instance**.
1. Go to the admin area by clicking the little wrench icon on the top menu bar.
1. On the left menu bar, select **Applications**, then click **<NT>New</NT> Application**.

   ![New_Application.png](/img/_paligo/uuid-c139a2c3-3f9a-980e-41f0-a85bc3ea6bbb.png)
1. In the **<NT>Name</NT>** field, enter a name that clearly identifies the application. For the sake of simplicity, we recommend `Bitrise`.
1. In the **Callback URL** field, enter `https://app.bitrise.io/users/auth/gitlab/callback`.

   This is <NT>Bitrise</NT>'s URL. <NT>GitLab</NT> will redirect users here after they authenticate.
1. In the **<NT>Scope</NT>s** menu, check **api**.

   ![New_Application_settings.png](/img/_paligo/uuid-9e8a91b7-c03a-f513-62a7-0e91426f1676.png)
1. Set the **Confidential** option to **No**.

   If the newly created app is configured to be confidential, <NT>Bitrise</NT> won't be able to access it!
1. Click **<NT>Submit</NT>**. On the next page, you should find the **Application Id** and a **<NT>Secret</NT>**. You’ll need both to connect to your <NT>GitLab</NT> instance on [bitrise.io](https://www.bitrise.io).

   ![Connecting_self-hosted_GitLab_instances.png](/img/_paligo/uuid-94e2570d-ce4b-720a-db91-52b46de0a96a.png)
1. <NT>Log in</NT> to <NT>Bitrise</NT> and select your <NT>workspace</NT>.
1. On the left, select **<NT>Integrations</NT>**.
1. Find the **Self-hosted <NT>GitLab self-managed</NT>** section.

   ![self-hosted-gitlab.png](/img/_paligo/uuid-ee43070a-a23d-02a4-3860-9899749044e2.png)
1. Click **<NT>Add credentials</NT>** and add your self-hosted <NT>GitLab</NT> credentials:

   - **<NT>App ID</NT>**
   - **<NT>Secret</NT>**
   - **<NT>Self-hosted URL</NT>**

     The <NT>Self-hosted URL</NT> is your <NT>GitLab</NT> instance's URL (for example, https://gitlab.example.com). It is not the same as the <NT>Bitrise</NT> callback URL that you pasted in the <NT>GitLab</NT> instance configuration!

   :::note

   Optionally, you can add custom HTTP headers: [Connecting through a proxy](/en/bitrise-platform/repository-access/connecting-self-hosted-gitlab-instances#connecting-through-a-proxy).

   :::

You are done! Now you are able to access your privately hosted repositories with <NT>Bitrise</NT>.


## <NT>Service credential user</NT> role

<NT>Bitrise</NT> uses [a Service Credential User (SCU)](/en/bitrise-platform/integrations/the-service-credential-user) on your <NT>GitLab</NT> instance to register webhooks, post commit statuses on merge requests, and read repository contents. On self-hosted <NT>GitLab</NT>, the SCU needs **Maintainer** role on the repository at minimum:

| <NT>GitLab</NT> operation | Required role |
| --- | --- |
| Read `bitrise.yml`, list branches | Reporter |
| Register webhook | Maintainer |
| Post commit / merge request status | Maintainer |

Reporter or Developer is not sufficient. Status reporting will silently fail. See [The service credential user](/en/bitrise-platform/integrations/the-service-credential-user) for the full SCU configuration flow.


## Connecting through a proxy

You can add custom HTTP headers to your self-hosted <NT>GitLab</NT> instance to be able to connect through a proxy. The <NT>header value</NT>s are stored encrypted.

To add a custom HTTP header, go to the **<NT>Workspace settings</NT>** page and select **<NT>Integrations</NT>**. Find the **<NT>GitLab self-managed</NT>** card and you can edit your credentials there.

For example, Cloudflare Access customers can set their client ID and client secret: `CF-Access-Client-Id` and `CF-Access-Client-Secret`.


## Adding a new project from a self-hosted <NT>GitLab</NT> repository

1. Start the process of [adding a new project](/en/bitrise-build-cache/getting-started-with-the-build-cache/getting-started-with-the-build-cache#adding-a-new-connection-to-the-build-cache).
1. When prompted to select your Git provider, select the **<NT>Self-hosted GitLab</NT>** option.
1. Click **<NT>Connect</NT>**.

Once connected, proceed as usual: [Adding your first project](/en/bitrise-build-cache/getting-started-with-the-build-cache/getting-started-with-the-build-cache#adding-a-new-connection-to-the-build-cache).
