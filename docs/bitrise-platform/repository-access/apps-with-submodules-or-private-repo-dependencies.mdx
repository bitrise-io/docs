:::note[<NT>GitHub</NT> App integration]

If your project uses a <NT>GitHub</NT> App installation to connect to your repository, you can simply [link additional repositories](/en/bitrise-platform/repository-access/github-app-integration#additional-linked-repositories-via-a-github-app). You don't need any of the workarounds described in this document.

:::

If you have a project with one or more submodules or other private repository dependencies (for example, <NT>CocoaPods</NT> repositories), <NT>Bitrise</NT> needs access to <NT>all repositories</NT> or submodules for a successful build.

For private projects with OAuth connections, <NT>Bitrise</NT> uses SSH to access Git repositories: to grant access, you need to make sure all the repositories can be accessed with the public [SSH key generated](/en/bitrise-platform/repository-access/configuring-ssh-keys) for your <NT>Bitrise</NT> <NT><GlossTerm baseform="project">project</GlossTerm></NT>.

There are two ways to achieve this:

- [Register the same SSH key](/en/bitrise-platform/repository-access/configuring-ssh-keys) for every repository you have to access during the build.
- Register the SSH key with a bot user and add that user to <NT>all repositories</NT>.

Registering the same SSH key for every repository is the best, most secure way - but not all services support it. <NT>GitLab</NT> and <NT>Bitbucket</NT> does support it, <NT>GitHub</NT>, however, doesn’t. If your code is stored on <NT>GitHub</NT> and you use an OAuth connection, read on!

We’ll go through the other option, using a bot user or machine user - <NT>GitHub</NT> calls them machine users - in detail. In brief, the concept is simple: you register the <NT>Bitrise</NT> public SSH key to a user and add that user to <NT>all repositories</NT> that have to be accessed for your <NT>Bitrise</NT> build.

:::note[Machine user with read only access]

It is not required to use a special bot/machine user: you can add the SSH key to your own account on the git hosting service. The best practice, however, is to use a machine user, with read only access, for those repositories you want to access during the build.

:::

[GitHub recommends this method](https://docs.github.com/en/developers/overview/managing-deploy-keys#machine-users) for accessing multiple repositories. Let’s quickly go through an example.

**The MultiBit project**

We have a <NT>Bitrise</NT> <NT>project</NT> we’ll call MultiBit. It has a main repository and it needs to pull additional data from two other, private repositories during a <NT>Bitrise</NT> build. Our main user is called BitMan and his <NT>GitHub</NT> account is linked to his <NT>Bitrise</NT> account. To access the private repositories during the build, BitMan creates another user on <NT>GitHub</NT>, called BitBot. BitBot will be the machine user.

Now, BitMan accesses the **<NT>Project settings</NT>** page of his <NT>Bitrise</NT> <NT>project</NT>, and copies the public SSH key. BitMan then adds the SSH key to the BitBot user on <NT>GitHub</NT> and adds BitBot to the two private repositories as a collaborator. Now BitBot has the <NT>Bitrise</NT> public key and has access to the private repositories as necessary. It’s time to test if BitMan’s setup works.

BitMan goes to <NT>Bitrise</NT> and opens the <NT><GlossTerm baseform="Workflow Editor">Workflow Editor</GlossTerm></NT>. He has the **<NT>Activate</NT> <NT>SSH Key</NT>** <NT><GlossTerm baseform="step">Step</GlossTerm></NT> in his repository so the SSH key will work. BitMan adds a [**Script**](https://github.com/bitrise-io/steps-script) <NT>Step</NT> to clone the private repositories - as the **<NT>Git Clone</NT>** <NT>Step</NT> only works with the main repository! Once all that is done, the build should work as expected.


## Using a machine user to access private repositories

A [machine or bot user](https://docs.github.com/en/developers/overview/managing-deploy-keys#machine-users) is a <NT>GitHub</NT> user that is not used by humans, instead it is exclusively used for automation. This is the best way to access a private repository: you create a machine user, add a public SSH key to the user, and then provide the user read access to the repository.

### Adding the machine user to your repository

1. Create a new <NT>GitHub</NT> user account, one that will serve as the machine user.
1. Go to your repository on <NT>GitHub</NT> and select the **<NT>Settings</NT>** tab.
1. On the left side menu, select **Collaborators & teams**.

   ![Can_I_add_projects_with_submodules_or_with_private_repo_dependencies_.png](/img/_paligo/uuid-987a7495-6bb7-2c5e-0a19-6871b2ca8f1f.png)
1. Scroll down to the **Collaborators** window.
1. In the search input field, search for the username of your newly created account.
1. Click **<NT>Add</NT> Collaborator**.
1. Change the user permission to **Read**.

   By default, the invited collaborator’s permission is **Write**. You can keep it that way, of course, but a Read permission is enough for <NT>Bitrise</NT>.

### Adding the SSH key to the machine user

In order for <NT>Bitrise</NT> to be able to use the machine user to access your repository, you must add the same SSH key to the machine user and the project on <NT>Bitrise</NT>.

#### When adding a new project

1. Start the process of [adding your project on Bitrise](/en/bitrise-ci/getting-started/adding-a-new-project).
1. When prompted to setup repository access, you can choose from three options:

   **<NT>Auto-add a generated SSH key to your repository</NT>**: <NT>Bitrise</NT> automatically registers a public SSH key to your <NT>GitHub</NT> repository.

   **<NT>Copy a generated SSH key to your Git provider manually</NT>**: <NT>Bitrise</NT> generates an SSH keypair for you, and you have to manually register the public key to your Git repository.

   **<NT>Add</NT> your own <NT>SSH key</NT>**: You provide your own SSH keypair for authentication, and you have to manually register the public key to your Git repository.
1. Finish the process.
1. Add the same SSH key to your machine user that you added to the project.

#### When your project already exists

1. <Partial_GettingToTheProjectSettingsPage />
1. On the left, select **Repository**.
1. In the **<NT>Authorization</NT>** section, find the **<NT>Authorization method</NT>** card, and click **<NT>Copy public key</NT>**.

   ![SCR-20260401-pvvq.png](/img/_paligo/uuid-f021126a-47d0-41a9-56f7-79b9c8461a0e.png)
1. [Add it to your GitHub machine user](https://help.github.com/en/articles/adding-a-new-ssh-key-to-your-github-account).


## Git cloning submodules and repository dependencies

You have three options when it comes to accessing multiple repositories during a <NT>Bitrise</NT> build:

- Cloning all the repositories on the virtual machine, and accessing them as needed.
- Adding the additional repositories as submodules to your main repository.
- Using the [GitHub App integration](/en/bitrise-platform/repository-access/github-app-integration) and [linking additional repositories](/en/bitrise-platform/repository-access/github-app-integration#additional-linked-repositories-via-a-github-app).

In the latter case, you do not need to worry about cloning them: if you set up [SSH access](/en/bitrise-platform/repository-access/configuring-ssh-keys) correctly, the **<NT>Git Clone</NT>** <NT>Step</NT> will take care of everything.

If you don’t want to or can’t add your repository dependencies as submodules, read on: we’ll talk about how to clone them.

:::note[The <NT>Git Clone</NT> <NT>Step</NT>]

The **<NT>Git Clone</NT>** <NT>Step</NT> only works with the main repository. If you need to access multiple private repositories, do not add multiple **<NT>Git Clone</NT>** <NT>Steps</NT>. Use **Script** <NT>Steps</NT> to clone those repositories on the <NT>Bitrise</NT> virtual machine.

:::

To clone additional private repositories during the build:

1. Make sure you added a user with the <NT>Bitrise</NT> public SSH key to all the repositories.
1. Make sure you have the **<NT>Activate</NT> <NT>SSH Key</NT>** <NT>Step</NT> and the **<NT>Git Clone</NT>** <NT>Step</NT> at the start of your <NT><GlossTerm baseform="workflow">Workflow</GlossTerm></NT>.
1. Add one or more **Script** <NT>Steps</NT> to clone the additional private repositories to the build.
1. Run a build.

:::note[Cloning issues]

If you encounter issues with git cloning - for example, not all submodules are cloned - try the following command after cloning:

```bash
git submodule update –recursive –remote –merge –force
```

:::

### Public vs private projects

There is one important detail to keep in mind when you want to give access to all submodules or private repository dependencies for a <NT>Bitrise</NT> <NT>project</NT>: the project’s privacy settings determine what [git URL should you use](/en/bitrise-platform/projects/configuring-the-repository-url-and-the-default-branch#changing-the-repository-url).

- If you have a private project: use SSH URLs everywhere! Most services support SSH key based authentication only for SSH URLs (for example, `git@github.com:bitrise-io/bitrise.git`). Therefore every private repository you want to use have to be addressed with the SSH URL. If you have direct private git repo references in your <NT>CocoaPods</NT> <NT>Podfile</NT>, you’ll have to use the SSH URL there as well. The same applies for submodules and every other private git repository URL you want to use with [the SSH key you register](/en/bitrise-platform/repository-access/configuring-ssh-keys) on [bitrise.io](https://www.bitrise.io/).
- If you have a public project: use HTTPS URLs everywhere! SSH URLs require SSH keys even if the repository is public. For security reasons, public projects CANNOT have SSH keys. As HTTPS <NT>git clone</NT> URLs do not require any authentication in the case of public repositories, they should be used for public <NT>Bitrise</NT> <NT>project</NT>s.
